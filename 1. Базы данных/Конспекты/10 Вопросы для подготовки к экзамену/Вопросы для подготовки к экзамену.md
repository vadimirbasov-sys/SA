# Лекция: Фундаментальные концепции баз данных

## 1. Что такое база данных? Какие виды бывают?

### 1.1. База данных — это...

Представьте огромную библиотеку будущего, но не ту, где книги стоят на полках, а ту, где все знания человечества организованы по определённым правилам. База данных — это именно такая организованная коллекция структурированной информации, хранящаяся в электронном виде.

Если говорить техническим языком, база данных — это:
- **Упорядоченный набор данных**, организованный для эффективного хранения
- **Структурированная информация**, связанная определёнными отношениями
- **Электронный архив**, позволяющий быстро находить, добавлять, изменять и удалять данные
- **Цифровая система учета**, подобная картотеке, но с безграничными возможностями поиска

**Жизненная аналогия**: Медицинская карта пациента в больнице. В ней хранится вся история болезней, анализы, назначения, но организована она так, чтобы врач мог быстро найти нужную информацию.

### 1.2. Виды баз данных: Два великих семейства

Мир баз данных делится на две большие семьи, как два разных подхода к организации знаний:

**1. Реляционные (SQL) базы данных**
Представьте классическую библиотеку с каталогом: книги (таблицы) расставлены по полкам, у каждой книги есть строгая структура (главы, страницы), а в каталоге указаны связи между книгами.

**Характеристики**:
- Данные организованы в таблицы с строками и столбцами
- Чёткая структура (схема), определённая заранее
- Используют SQL для запросов
- Гарантируют целостность данных (ACID)
- Примеры: MySQL, PostgreSQL, Oracle

**2. Нереляционные (NoSQL) базы данных**
Представьте мозг человека: информация хранится не в таблицах, а в нейронных сетях, где каждая идея связана с другими ассоциациями, и структура постоянно меняется и адаптируется.

**Характеристики**:
- Гибкая или отсутствующая схема данных
- Разные модели хранения (документы, ключ-значение, графы, колонки)
- Часто отказываются от строгих ACID гарантий ради масштабируемости
- Примеры: MongoDB, Redis, Cassandra, Neo4j

**Сравнительная таблица**:

| Аспект | Реляционные | Нереляционные |
|--------|-------------|---------------|
| **Структура** | Таблицы, строки, столбцы | Документы, графы, ключ-значения |
| **Схема** | Жёсткая, фиксированная | Гибкая, динамическая |
| **Масштабирование** | Вертикальное | Горизонтальное |
| **Консистентность** | Строгая (ACID) | Часто ослабленная (BASE) |
| **Язык запросов** | SQL | Разные API |

## 2. СУБД и РСУБД: Дирижёры оркестра данных

### 2.1. Что такое СУБД?

СУБД (Система Управления Базами Данных) — это программное обеспечение, которое управляет базой данных. Если база данных — это библиотека, то СУБД — это библиотекарь, администратор, охранник и каталогизатор в одном лице.

**Функции СУБД**:
1. **Хранение данных** — Организует физическое хранение на диске
2. **Манипуляция данными** — Позволяет добавлять, изменять, удалять, искать
3. **Управление доступом** — Контролирует, кто что может делать
4. **Обеспечение целостности** — Следит, чтобы данные были корректными
5. **Резервное копирование** — Заботится о сохранности данных
6. **Оптимизация запросов** — Находит самый быстрый способ выполнить запрос

**Примеры СУБД**: Oracle Database, Microsoft SQL Server, MySQL, PostgreSQL, MongoDB

### 2.2. Что такое РСУБД?

РСУБД (Реляционная Система Управления Базами Данных) — это частный случай СУБД, который работает именно с реляционными базами данных. Это СУБД, которая "мыслит" таблицами, строками и связями.

**Ключевые особенности РСУБД**:
- Поддерживает реляционную модель Кодда
- Использует SQL как основной язык запросов
- Обеспечивает ACID свойства
- Поддерживает целостность ссылок (внешние ключи)
- Позволяет создавать сложные связи между таблицами

**Примеры РСУБД**: MySQL, PostgreSQL, Oracle Database, Microsoft SQL Server

**Различие**: Все РСУБД являются СУБД, но не все СУБД являются РСУБД. MongoDB — это СУБД, но не РСУБД, потому что она не реляционная.

## 3. SQL vs MySQL: Язык vs Продукт

### 3.1. Чем SQL отличается от MySQL?

Это классическая путаница, как путать "русский язык" (SQL) с "учебником русского языка" (MySQL).

**SQL (Structured Query Language)** — это язык
- **Что это**: Язык программирования для работы с реляционными базами данных
- **Аналогия**: Русский язык
- **Назначение**: Общение с базой данных (запросы, обновления, управление)
- **Стандарт**: Есть стандарт ANSI/ISO, но разные диалекты
- **Используется**: Во всех реляционных СУБД

**MySQL** — это продукт
- **Что это**: Конкретная реляционная СУБД (одна из многих)
- **Аналогия**: Конкретный учебник русского языка
- **Разработчик**: Oracle Corporation (ранее Sun, раньше MySQL AB)
- **Особенности**: Открытый исходный код, популярен для веб-приложений
- **Использует**: SQL для запросов (со своими расширениями)

**Простая аналогия**:
- **SQL** — это правила дорожного движения
- **MySQL** — это конкретная марка автомобиля (например, Toyota)
- **Другие СУБД** — другие марки автомобилей (PostgreSQL — Volvo, Oracle — Mercedes)

**Другие СУБД, использующие SQL**:
- PostgreSQL
- Microsoft SQL Server
- Oracle Database
- SQLite
- MariaDB (форк MySQL)

## 4. Нереляционные базы данных: Как они выглядят?

### 4.1. Многообразие форм

Нереляционные базы данных отказываются от таблиц и предлагают разные модели хранения, каждая для своих задач:

**1. Документные базы данных** (MongoDB, CouchDB)
Представьте папку с документами. Каждый документ — это самостоятельная единица (часто в JSON формате), которая может иметь любую структуру.

**Пример**: Профиль пользователя в социальной сети:
```
{
  "id": 123,
  "name": "Иван",
  "age": 30,
  "hobbies": ["программирование", "гитара"],
  "address": {
    "city": "Москва",
    "street": "Ленина"
  }
}
```

**2. Ключ-значение хранилища** (Redis, Memcached)
Представьте ячейки на почте. У каждой ячейки номер (ключ), внутри — что угодно (значение).

**Пример**: Кэш сессии пользователя:
```
Ключ: "session:abc123"
Значение: "{user_id: 456, last_login: '2024-01-15'}"
```

**3. Графовые базы данных** (Neo4j, Amazon Neptune)
Представьте социальную сеть. Люди — узлы, дружба — связи.

**Пример**: Социальные связи:
```
(Алиса) -[ДРУЖИТ С]-> (Боб)
(Боб) -[РАБОТАЕТ В]-> (Компания X)
```

**4. Колонковые базы данных** (Cassandra, HBase)
Представьте таблицу, хранящуюся не по строкам, а по столбцам. Все имена вместе, все возрасты вместе.

**Пример**: Хранение метрик:
```
Столбец "timestamp": [1, 2, 3, 4, 5]
Столбец "temperature": [20, 21, 22, 23, 24]
Столбец "humidity": [45, 46, 47, 48, 49]
```

## 5. DDL и DML: Строитель и житель

### 5.1. DDL — Data Definition Language

DDL — это язык определения данных. Это архитектор, который проектирует и строит здания (таблицы), в которых будут жить данные.

**Что делает DDL**:
- Создаёт структуры (таблицы, базы данных)
- Изменяет структуры (добавляет столбцы, меняет типы)
- Удаляет структуры

**Основные команды DDL**:
- **CREATE** — Создать новый объект (таблицу, базу данных)
- **ALTER** — Изменить существующий объект
- **DROP** — Удалить объект полностью
- **TRUNCATE** — Очистить таблицу (удалить все данные, оставить структуру)

**Ментальная модель**: DDL работает со скелетом базы данных. Когда вы используете DDL, вы меняете саму структуру, каркас, на котором всё держится.

### 5.2. DML — Data Manipulation Language

DML — это язык манипуляции данными. Это жильцы, которые живут в зданиях, построенных DDL.

**Что делает DML**:
- Добавляет данные
- Изменяет данные
- Удаляет данные
- Читает данные

**Основные команды DML**:
- **SELECT** — Прочитать данные (самая частая операция)
- **INSERT** — Добавить новые данные
- **UPDATE** — Изменить существующие данные
- **DELETE** — Удалить данные

**Ментальная модель**: DML работает с содержимым базы данных. Структура уже готова (построена DDL), DML наполняет её жизнью.

**Ключевое различие**:
- **DDL** меняет структуру: "Давайте добавим в дом новый этаж"
- **DML** меняет содержимое: "Давайте поставим в комнату новый диван"

## 6. Типы связей в реляционных БД: Три вида отношений

### 6.1. Три фундаментальных типа связей

В реляционных базах данных существует три основных типа связей между таблицами, которые отражают реальные отношения между сущностями:

**1. Один к одному (1:1)**
Одна запись в первой таблице связана с одной записью во второй таблице.

**Пример**: Человек и паспорт. У каждого человека один паспорт, каждый паспорт принадлежит одному человеку.

**2. Один ко многим (1:N) или Многие к одному (N:1)**
Одна запись в первой таблице связана с несколькими записями во второй таблице.

**Пример**: Автор и книги. Один автор может написать много книг, но каждая книга имеет одного автора.

**3. Многие ко многим (N:N)**
Несколько записей в первой таблице связаны с несколькими записями во второй таблице.

**Пример**: Студенты и курсы. Один студент может посещать несколько курсов, на один курс может записаться несколько студентов.

### 6.2. Декартово произведение: Все со всеми

Декартово произведение в контексте реляционных баз — это результат соединения каждой строки одной таблицы с каждой строкой другой таблицы.

**Что это такое**:
- Если в таблице A 3 строки, а в таблице B 4 строки
- Декартово произведение даст 3 × 4 = 12 строк
- Каждая строка A сочетается с каждой строкой B

**Пример из жизни**:
Представьте, что у вас есть:
- Список рубашек: Красная, Синяя, Зелёная (3 элемента)
- Список брюк: Чёрные, Серые (2 элемента)

Декартово произведение даст все возможные сочетания:
1. Красная рубашка + Чёрные брюки
2. Красная рубашка + Серые брюки
3. Синяя рубашка + Чёрные брюки
4. Синяя рубашка + Серые брюки
5. Зелёная рубашка + Чёрные брюки
6. Зелёная рубашка + Серые брюки

**В SQL**: Декартово произведение получается, когда делают JOIN без условия или используют CROSS JOIN.

## 7. Реализация связей в БД: Практическая магия

### 7.1. Как реализовать связь 1 к 1?

Связь "один к одному" реализуется через внешний ключ с уникальным ограничением.

**Способ 1: В одной таблице**
Иногда проще просто добавить столбцы в одну таблицу, если связь действительно жесткая.

**Способ 2: Две таблицы с внешним ключом**
```
Таблица "Люди": id, имя, ...
Таблица "Паспорта": id, номер, серия, человек_id (UNIQUE, FOREIGN KEY)
```

**Особенности**:
- Внешний ключ в таблице-потомке ссылается на первичный ключ таблицы-родителя
- Добавляется ограничение UNIQUE, чтобы гарантировать 1:1
- Можно хранить связь в любой из таблиц

### 7.2. Как реализовать связь многие ко многим?

Связь "многие ко многим" требует промежуточной таблицы (junction table, association table).

**Структура**:
```
Таблица "Студенты": id, имя
Таблица "Курсы": id, название
Таблица "Записи_на_курсы": id, студент_id (FOREIGN KEY), курс_id (FOREIGN KEY)
```

**Как это работает**:
- Промежуточная таблица содержит пары "студент-курс"
- Каждая запись связывает одного студента с одним курсом
- Один студент может иметь несколько записей (на разные курсы)
- Один курс может иметь несколько записей (разные студенты)

**Дополнительные возможности**:
- В промежуточной таблице можно хранить дополнительную информацию: дата записи, оценка, статус
- Можно добавлять ограничения (UNIQUE на пару студент-курс, чтобы избежать дублирования)

## 8. Запросы выборки: Язык общения с данными

### 8.1. Операторы выборки данных

При написании запроса выборки используются следующие основные операторы:

**Обязательные операторы**:
- **SELECT** — Что выбрать (столбцы)
- **FROM** — Откуда выбрать (таблицы)

**Опциональные операторы**:
- **WHERE** — Условие отбора строк
- **GROUP BY** — Группировка строк
- **HAVING** — Условие отбора групп
- **ORDER BY** — Сортировка результатов
- **LIMIT/OFFSET** или **TOP** — Ограничение количества строк
- **DISTINCT** — Уникальные значения
- **JOIN** — Соединение таблиц
- **UNION/UNION ALL** — Объединение результатов

### 8.2. Порядок выполнения операторов

Хотя операторы пишутся в определённом порядке, СУБД выполняет их в другом порядке для оптимизации. Логический порядок выполнения:

1. **FROM** — Определяем, с какими таблицами работаем
2. **WHERE** — Фильтруем строки по условию
3. **GROUP BY** — Группируем оставшиеся строки
4. **HAVING** — Фильтруем группы
5. **SELECT** — Выбираем, какие столбцы показать
6. **ORDER BY** — Сортируем результат
7. **LIMIT** — Ограничиваем количество строк

**Пример мышления СУБД**:
"Сначала найду все таблицы (FROM), потом оставлю только нужные строки (WHERE), сгруппирую их (GROUP BY), оставлю только нужные группы (HAVING), потом решу, что показать (SELECT), отсортирую (ORDER BY) и обрежу (LIMIT)."

## 9. JOIN: Искусство соединения таблиц

### 9.1. Типы JOIN и их назначение

JOIN — это операция соединения таблиц. Представьте, что у вас есть разрозненные данные в разных таблицах, и JOIN позволяет собрать их в единую картину.

**1. INNER JOIN (Внутреннее соединение)**
Возвращает только строки, которые имеют совпадения в обеих таблицах.

**Пример**: Список сотрудников, у которых есть отдел. Если у сотрудника нет отдела или отдел без сотрудников — они не попадут в результат.

**2. LEFT JOIN (Левое соединение)**
Возвращает все строки из левой таблицы, даже если нет совпадений в правой. Если совпадений нет — NULL в столбцах правой таблицы.

**Пример**: Все сотрудники, даже если у них нет отдела. Сотрудники без отдела будут с NULL в столбцах отдела.

**3. RIGHT JOIN (Правое соединение)**
Зеркальное отражение LEFT JOIN. Все строки из правой таблицы, даже если нет совпадений слева.

**4. FULL OUTER JOIN (Полное внешнее соединение)**
Все строки из обеих таблиц. Где есть совпадение — соединяются, где нет — NULL.

**Пример**: Все сотрудники и все отделы. Будут и сотрудники без отделов, и отделы без сотрудников.

**5. CROSS JOIN (Декартово соединение)**
Каждая строка первой таблицы соединяется с каждой строкой второй таблицы.

**Пример**: Все комбинации размеров и цветов товара.

### 9.2. UNION и UNION ALL: Вертикальное объединение

**UNION** — объединяет результаты двух или более запросов, удаляя дубликаты.

**UNION ALL** — объединяет результаты, сохраняя все строки, включая дубликаты.

**Различие**:
- UNION выполняет сортировку и удаление дубликатов → медленнее
- UNION ALL просто складывает результаты → быстрее

**Пример**:
- UNION: Объединить уникальных клиентов из двух филиалов
- UNION ALL: Объединить все транзакции из двух филиалов (дубликаты возможны)

### 9.3. JOIN vs UNION: Горизонтально vs Вертикально

**JOIN** соединяет таблицы по горизонтали:
- Добавляет столбцы
- Соединяет на основе условия
- Результат: более широкие строки

**UNION** соединяет результаты по вертикали:
- Добавляет строки
- Не требует условия соединения
- Результат: более длинный список
- Требует одинаковой структуры

**Пример**:
- JOIN: К данным о сотруднике добавить данные об отделе
- UNION: Объединить списки сотрудников из Москвы и Санкт-Петербурга

### 9.4. RIGHT JOIN vs LEFT JOIN: Две стороны одной медали

**LEFT JOIN** и **RIGHT JOIN** — это одно и то же, просто с разных сторон.

**LEFT JOIN**: Все из левой таблицы + совпадения справа
**RIGHT JOIN**: Все из правой таблицы + совпадения слева

**Практическое правило**: Используйте LEFT JOIN, это более привычно. RIGHT JOIN можно заменить LEFT JOIN, поменяв таблицы местами.

**Пример**:
- `A LEFT JOIN B` = `B RIGHT JOIN A`
- Выбор зависит от того, какую таблицу считаем основной

## 10. Работа с дубликатами и ограничениями

### 10.1. Удаление дубликатов из таблицы

**Способы удаления дубликатов**:

1. **DISTINCT в SELECT** — Показать только уникальные значения
2. **GROUP BY** — Сгруппировать по всем столбцам
3. **Оконные функции** (ROW_NUMBER) — Пронумеровать дубликаты и удалить
4. **Создание новой таблицы** с DISTINCT и удаление старой
5. **DELETE с самообъединением** — Удалить строки с одинаковыми значениями

**Важное замечание**: При удалении дубликатов важно определить, что считать дубликатом. Полные дубликаты (все столбцы одинаковые) или частичные (только некоторые столбцы)?

### 10.2. Ограничение числа выводимых строк

**Операторы для ограничения**:

1. **LIMIT** (MySQL, PostgreSQL, SQLite) — Ограничить количество строк
2. **TOP** (SQL Server, MS Access) — Показать первые N строк
3. **ROWNUM** (Oracle) — Ограничить по номеру строки
4. **FETCH FIRST** (стандарт SQL) — Современный стандартный способ

**Примеры использования**:
- Показать первые 10 записей
- Показать записи с 11 по 20 (пагинация)
- Показать топ-5 по рейтингу

### 10.3. Подсчет строк в таблице

**Оператор COUNT()** — подсчитывает количество строк.

**Варианты**:
- `COUNT(*)` — подсчитать все строки, включая NULL
- `COUNT(столбец)` — подсчитать строки, где столбец не NULL
- `COUNT(DISTINCT столбец)` — подсчитать уникальные значения

**Пример**:
- Сколько всего сотрудников? — `COUNT(*)`
- Сколько сотрудников с указанной зарплатой? — `COUNT(зарплата)`
- Сколько уникальных должностей? — `COUNT(DISTINCT должность)`

### 10.4. Вывод уникальных значений столбца

**Оператор DISTINCT** — возвращает уникальные значения.

**Особенности**:
- Может применяться к одному или нескольким столбцам
- Для нескольких столбцов — уникальность комбинаций
- Всегда идёт после SELECT

**Примеры**:
- Уникальные города: `SELECT DISTINCT город FROM клиенты`
- Уникальные пары город-страна: `SELECT DISTINCT город, страна FROM клиенты`

## 11. Удаление данных: Три уровня

### 11.1. DELETE vs TRUNCATE

**DELETE** — удаление строк с возможностью отката:
- Можно удалять выборочно (с WHERE)
- Записывает в журнал транзакций каждую удаляемую строку
- Можно откатить (ROLLBACK)
- Вызывает триггеры
- Не сбрасывает счётчики автоинкремента
- Медленно для больших таблиц

**TRUNCATE** — быстрое удаление всех строк:
- Удаляет ВСЕ строки (без WHERE)
- Освобождает место на диске
- Обычно нельзя откатить
- Не вызывает триггеры
- Сбрасывает счётчики автоинкремента
- Очень быстро

**Когда что использовать**:
- DELETE: Нужно удалить конкретные строки, нужен откат
- TRUNCATE: Нужно быстро очистить всю таблицу, данные не нужны

### 11.2. TRUNCATE vs DROP

**TRUNCATE** — очистка таблицы:
- Удаляет данные, но оставляет структуру
- Таблица остаётся, можно сразу использовать
- Быстро освобождает место

**DROP** — удаление таблицы:
- Удаляет и данные, и структуру
- Таблицы больше не существует
- Нужно пересоздавать таблицу для использования
- Удаляет индексы, триггеры, привилегии

**Аналогия**:
- TRUNCATE — выбросить все вещи из шкафа, но шкаф оставить
- DROP — выбросить шкаф вместе с вещами

## 12. CRUD операции: Фундамент работы с данными

### 12.1. Что такое CRUD операции?

CRUD — акроним четырех базовых операций работы с данными:
- **C**reate (Создать)
- **R**ead (Прочитать)
- **U**pdate (Обновить)
- **D**elete (Удалить)

Это фундаментальные операции, которые поддерживает практически любая система работы с данными.

**Аналогия с библиотекой**:
- Create: Добавить новую книгу в каталог
- Read: Найти книгу по названию
- Update: Исправить описание книги
- Delete: Списать утерянную книгу

### 12.2. SQL операторы для CRUD

**Create** → **INSERT** — добавление новых записей
```
INSERT INTO таблица (столбцы) VALUES (значения)
```

**Read** → **SELECT** — чтение данных
```
SELECT столбцы FROM таблица WHERE условия
```

**Update** → **UPDATE** — изменение существующих данных
```
UPDATE таблица SET столбец = значение WHERE условия
```

**Delete** → **DELETE** — удаление данных
```
DELETE FROM таблица WHERE условия
```

**Важно**: Для каждой операции можно добавлять условия (WHERE), сортировку (ORDER BY), ограничения (LIMIT) и другие модификаторы.

## 13. Auto Increment: Автоматическая нумерация

### 13.1. Что такое Auto Increment?

Auto Increment (автоинкремент) — это свойство столбца, которое автоматически генерирует уникальные числовые значения при вставке новых строк.

**Как работает**:
- При вставке новой строки не нужно указывать значение для этого столбца
- Система автоматически берёт последнее значение и увеличивает на 1
- Гарантирует уникальность в пределах таблицы

**Использование**:
- Чаще всего для первичных ключей
- Когда нужны уникальные идентификаторы
- Упрощает вставку данных (не нужно генерировать ID вручную)

**Особенности**:
- При DELETE автоинкремент не сбрасывается (пропуски остаются)
- TRUNCATE сбрасывает счётчик автоинкремента
- В разных СУБД реализация может отличаться

## 14. Типы данных в SQL: Разнообразие форматов

### 14.1. Основные категории типов данных

**Числовые типы**:
- **INT/INTEGER** — целые числа
- **DECIMAL/NUMERIC** — числа с фиксированной точностью
- **FLOAT/REAL** — числа с плавающей точкой

**Строковые типы**:
- **CHAR** — строка фиксированной длины
- **VARCHAR** — строка переменной длины
- **TEXT** — большой текст

**Дата и время**:
- **DATE** — дата (год, месяц, день)
- **TIME** — время (часы, минуты, секунды)
- **DATETIME/TIMESTAMP** — дата и время вместе

**Бинарные типы**:
- **BLOB** — дворичные большие объекты (изображения, файлы)

**Специальные типы**:
- **BOOLEAN** — логический тип (TRUE/FALSE)
- **JSON/XML** — структурированные данные
- **UUID** — универсальные уникальные идентификаторы

### 14.2. CHAR vs VARCHAR

**CHAR (Character)** — строка фиксированной длины:
- Всегда занимает указанное количество символов
- Если строка короче — дополняется пробелами
- Быстрее для поиска (фиксированная длина)
- Тратит место, если строки обычно короче

**VARCHAR (Variable Character)** — строка переменной длины:
- Занимает столько места, сколько нужно + служебная информация
- Не дополняется пробелами
- Экономит место на диске
- Немного медленнее для поиска

**Когда что использовать**:
- CHAR: Коды (ISBN, телефонные номера), флаги ('Y'/'N')
- VARCHAR: Имена, адреса, описания

**Пример**:
- `CHAR(10)` для "ABC" хранит "ABC       " (7 пробелов)
- `VARCHAR(10)` для "ABC" хранит "ABC" (3 символа)

## 15. Агрегирующие функции: Математика над данными

### 15.1. Для чего нужны агрегирующие функции?

Агрегирующие функции выполняют вычисления над набором значений и возвращают одно значение. Они "сворачивают" множество строк в одну или несколько сводных строк.

**Основные агрегирующие функции**:

1. **COUNT()** — подсчёт количества
2. **SUM()** — сумма значений
3. **AVG()** — среднее значение
4. **MIN()** — минимальное значение
5. **MAX()** — максимальное значение

**Дополнительные**:
- **GROUP_CONCAT() / STRING_AGG()** — объединение строк
- **STDDEV()** — стандартное отклонение
- **VARIANCE()** — дисперсия

**Примеры использования**:
- Сколько всего заказов? — `COUNT(*)`
- Общая сумма продаж — `SUM(сумма)`
- Средний чек — `AVG(сумма)`
- Самый дорогой товар — `MAX(цена)`
- Самая ранняя дата — `MIN(дата)`

### 15.2. WHERE vs HAVING: Два фильтра

**WHERE** — фильтрация строк:
- Применяется ДО группировки
- Работает с отдельными строками
- Не может использовать агрегирующие функции
- Быстрее, так как сокращает данные раньше

**HAVING** — фильтрация групп:
- Применяется ПОСЛЕ группировки
- Работает с группами строк
- Может использовать агрегирующие функции
- Медленнее, так как работает с уже сгруппированными данными

**Пример**:
- WHERE: "Показать сотрудников с зарплатой > 50000" (фильтр строк)
- HAVING: "Показать отделы, где средняя зарплата > 50000" (фильтр групп)

**Правило**: Сначала WHERE (чтобы меньше данных группировать), потом HAVING (чтобы отфильтровать группы).

## 16. Ключи: Первичные и внешние

### 16.1. Первичный ключ (Primary Key)

Первичный ключ — это столбец или комбинация столбцов, которые уникально идентифицируют каждую строку в таблице.

**Свойства первичного ключа**:
1. **Уникальность** — нет двух строк с одинаковым значением
2. **Не NULL** — всегда должно быть значение
3. **Один на таблицу** — может быть составным (несколько столбцов)
4. **Стабильность** — редко меняется

**Примеры**:
- ID сотрудника
- Номер паспорта
- ISBN книги
- Составной ключ: номер рейса + дата

### 16.2. Внешний ключ (Foreign Key)

Внешний ключ — это столбец в одной таблице, который ссылается на первичный ключ в другой таблице.

**Назначение внешнего ключа**:
1. **Обеспечение ссылочной целостности** — нельзя ссылаться на несуществующую запись
2. **Создание связей между таблицами**
3. **Каскадные операции** — при удалении/изменении родителя можно автоматически обновить потомков

**Пример**:
```
Таблица "Заказы": id, клиент_id (внешний ключ), ...
Таблица "Клиенты": id (первичный ключ), имя, ...
```

**Ограничения внешнего ключа**:
- Можно задать действие при удалении родителя: RESTRICT, CASCADE, SET NULL
- Можно задать действие при обновлении родителя
- Замедляет операции вставки/обновления (проверка целостности)

## 17. NULL и 0: Пустота vs Ноль

### 17.1. Значение NULL

NULL — это специальное значение, которое означает "отсутствие данных", "неизвестно", "не применимо".

**Важные особенности NULL**:
- NULL ≠ 0
- NULL ≠ пустой строке ''
- NULL ≠ FALSE
- NULL — это отсутствие значения

**Сравнения с NULL**:
- `NULL = NULL` — возвращает NULL (не TRUE!)
- `NULL != NULL` — возвращает NULL
- Для проверки на NULL используется `IS NULL` или `IS NOT NULL`

**Арифметика с NULL**:
- Любая операция с NULL возвращает NULL
- `10 + NULL = NULL`
- `NULL * 5 = NULL`

### 17.2. NULL vs 0: Практические примеры

**Пример 1: Температура**
- 0°C — это конкретная температура
- NULL — температура не измерялась

**Пример 2: Бонус сотруднику**
- 0 — бонус нулевой (был рассчитан)
- NULL — бонус ещё не рассчитан

**Пример 3: Возраст**
- 0 — возраст известен и равен 0 (новорожденный)
- NULL — возраст неизвестен

**Вывод**: 0 — это значение, NULL — отсутствие значения.

## 18. Индексы: Ускорители поиска

### 18.1. Что такое индексы?

Индекс в базе данных — это дополнительная структура, которая ускоряет поиск данных. Это как алфавитный указатель в книге.

**Как работают индексы**:
- Создают отсортированную копию данных (или ссылок на данные)
- Позволяют быстро найти нужные строки без полного сканирования таблицы
- Ускоряют WHERE, JOIN, ORDER BY операции

**Аналогия с библиотекой**:
- Без индекса: Читать все книги, чтобы найти нужную
- С индексом: Посмотреть в каталоге и сразу взять нужную книгу

### 18.2. Типы индексов

**По структуре**:
1. **B-дерево** — самый распространённый, для диапазонных запросов
2. **Хэш-индекс** — для точного поиска (=)
3. **Bitmap-индекс** — для столбцов с малым количеством значений
4. **Full-text индекс** — для текстового поиска

**По охвату**:
1. **Кластеризованный** — определяет физический порядок данных (только один на таблицу)
2. **Некластеризованный** — отдельная структура (может быть много)

**Составные индексы** — по нескольким столбцам

### 18.3. Плюсы и минусы индексов

**Плюсы**:
- Ускорение операций чтения
- Ускорение сортировки
- Обеспечение уникальности

**Минусы**:
- Замедление операций записи (INSERT, UPDATE, DELETE)
- Занимают место на диске
- Требуют обслуживания

**Правила создания индексов**:
1. Индексируйте часто используемые в WHERE столбцы
2. Индексируйте столбцы для JOIN
3. Индексируйте столбцы для ORDER BY
4. Избегайте индексов по часто изменяемым столбцам
5. Удаляйте неиспользуемые индексы

## 19. Транзакции и ACID: Гарантии целостности

### 19.1. Что такое транзакция?

Транзакция — это последовательность операций с базой данных, которая выполняется как единое целое.

**Свойства транзакции**:
- Либо все операции выполняются успешно
- Либо ни одна операция не выполняется
- Не может быть частичного выполнения

**Пример из жизни**: Перевод денег между счетами:
1. Снять деньги с первого счёта
2. Положить деньги на второй счёт
Если ошибка на шаге 2 — откатить шаг 1

### 19.2. Что скрывается за ACID?

ACID — набор свойств, гарантирующих надёжность транзакций:

**A — Atomicity (Атомарность)**
- Транзакция выполняется как единое целое
- "Все или ничего"
- Если ошибка — откат всех изменений

**C — Consistency (Согласованность)**
- Транзакция переводит базу из одного согласованного состояния в другое
- Не нарушаются ограничения целостности
- До и после транзакции данные корректны

**I — Isolation (Изоляция)**
- Параллельные транзакции не мешают друг другу
- Результат параллельных транзакций такой же, как если бы они выполнялись последовательно
- Уровни изоляции: Read Uncommitted, Read Committed, Repeatable Read, Serializable

**D — Durability (Долговечность)**
- После завершения транзакции изменения сохраняются навсегда
- Не теряются при сбоях системы
- Гарантируется записью в журнал транзакций

## 20. Нормализация: Искусство структурирования

### 20.1. Что такое нормализация?

Нормализация — это процесс организации данных в базе для:
1. Устранения избыточности данных
2. Устранения аномалий обновления, вставки, удаления
3. Упрощения структуры данных

**Аналогия**: Организация документов в архиве:
- Без нормализации: Все документы в одной папке, дублирование информации
- С нормализацией: Разделение по папкам, ссылки между ними

### 20.2. Уровни нормализации

**Первая нормальная форма (1NF)**:
1. В каждой ячейке таблицы — атомарное значение (не массив, не список)
2. Нет повторяющихся групп столбцов
3. Все строки уникальны
4. Порядок строк не имеет значения

**Пример нарушения**: Столбец "телефоны" со значением "123-456, 789-012"
**Исправление**: Отдельная таблица для телефонов

**Вторая нормальная форма (2NF)**:
1. Уже в 1NF
2. Все неключевые столбцы полностью зависят от всего первичного ключа
3. Нет частичных зависимостей (только для составных ключей)

**Пример нарушения**: В таблице "заказ-товар" хранится ещё и "название товара" (зависит только от товара, а не от пары заказ-товар)
**Исправление**: Вынести название товара в таблицу товаров

**Третья нормальная форма (3NF)**:
1. Уже в 2NF
2. Нет транзитивных зависимостей (неключевые столбцы не зависят от других неключевых столбцов)

**Пример нарушения**: В таблице "сотрудники" хранится "название отдела" и "город отдела" (город зависит от отдела, а не от сотрудника)
**Исправление**: Вынести отделы в отдельную таблицу

### 20.3. Когда БД считается нормализованной?

**Критерии нормализованной БД**:
- Устранена избыточность данных
- Минимизированы аномалии обновления
- Упрощено поддержание целостности данных
- Обычно достаточно 3NF

**Дальнейшие нормальные формы**:
- BCNF (Бойса-Кодда)
- 4NF
- 5NF
Но на практике редко используются дальше 3NF.

**Компромисс**: Чем выше нормализация, тем больше JOIN в запросах. Иногда используют денормализацию для производительности.

## 21. Процедуры и представления (VIEW)

### 21.1. Процедуры (Stored Procedures)

Процедура — это предварительно скомпилированный набор SQL-операторов, хранящийся в базе данных.

**Преимущества процедур**:
1. **Производительность** — компиляция один раз
2. **Безопасность** — контроль доступа к процедурам вместо таблиц
3. **Сокращение сетевого трафика** — вызов процедуры вместо отправки всего SQL
4. **Инкапсуляция логики** — бизнес-правила в одном месте

**Пример использования**: Сложная операция перевода денег между счетами с проверками и журналированием.

### 21.2. Представления (VIEW)

Представление — это виртуальная таблица, основанная на результате SQL-запроса.

**Преимущества представлений**:
1. **Упрощение сложных запросов** — сложный запрос скрыт за простым VIEW
2. **Безопасность** — можно дать доступ к VIEW, но не к исходным таблицам
3. **Абстракция** — скрытие изменений в структуре таблиц
4. **Согласованность** — единая логика в разных местах

**Типы представлений**:
- **Простые** — на основе одной таблицы, часто обновляемые
- **Сложные** — с JOIN, GROUP BY, обычно только для чтения
- **Материализованные** — физическое хранение данных, периодическое обновление

**Пример**: VIEW "сотрудники_с_отделами", который скрывает сложный JOIN между таблицами сотрудников и отделов.

## 22. Оконные функции: Аналитика без группировки

### 22.1. Для чего нужны оконные функции?

Оконные функции позволяют выполнять вычисления над набором строк, связанных с текущей строкой, без группировки результата.

**Отличие от агрегирующих функций**:
- Агрегирующие: Сворачивают множество строк в одну
- Оконные: Сохраняют все строки, добавляя вычисленные значения

**Пример задачи**: Для каждого сотрудника показать его зарплату и среднюю зарплату по отделу.

**Без оконных функций**: Сложно, нужны подзапросы или JOIN
**С оконными функциями**: Просто и эффективно

### 22.2. Типы оконных функций

**Агрегирующие** (но без группировки):
- SUM() OVER()
- AVG() OVER()
- COUNT() OVER()

**Ранжирующие**:
- ROW_NUMBER() — номер строки
- RANK() — ранг с пропусками
- DENSE_RANK() — ранг без пропусков
- NTILE() — разбиение на группы

**Сдвиговые**:
- LAG() — значение из предыдущей строки
- LEAD() — значение из следующей строки
- FIRST_VALUE() — первое значение в окне
- LAST_VALUE() — последнее значение в окне

**Статистические**:
- PERCENT_RANK() — относительный ранг
- CUME_DIST() — кумулятивное распределение

**Преимущества оконных функций**:
- Более читаемые запросы
- Лучшая производительность
- Решение сложных аналитических задач
- Сохранение детализации данных

## 23. Шардинг: Горизонтальное масштабирование

### 23.1. Что такое шардинг?

Шардинг (sharding) — это метод горизонтального масштабирования базы данных, при котором данные распределяются между несколькими серверами.

**Аналогия**: Большая библиотека, которая не помещается в одном здании:
- Без шардинга: Строим огромное здание (вертикальное масштабирование)
- Со шардингом: Строим несколько зданий, распределяем книги по алфавиту (горизонтальное)

**Как работает шардинг**:
1. Данные разбиваются на части (шарды)
2. Каждая часть хранится на отдельном сервере
3. Запросы направляются к нужному шарду
4. При необходимости — объединение результатов

### 23.2. Стратегии шардинга

**По диапазону (Range-based)**:
- Данные делятся по диапазонам значений ключа
- Пример: Пользователи A-M на сервер 1, N-Z на сервер 2
- Плюсы: Простота, эффективность для диапазонных запросов
- Минусы: Возможность неравномерного распределения

**По хэшу (Hash-based)**:
- Хэш-функция от ключа определяет шард
- Пример: hash(user_id) % number_of_shards
- Плюсы: Равномерное распределение
- Минусы: Сложность диапазонных запросов

**По справочнику (Directory-based)**:
- Таблица соответствия "ключ → шард"
- Пример: Отдельная таблица, где для каждого ключа указан шард
- Плюсы: Гибкость, можно перераспределять
- Минусы: Дополнительная точка отказа

### 23.3. Преимущества и сложности шардинга

**Преимущества**:
- Масштабируемость (добавляем серверы → увеличиваем capacity)
- Производительность (меньше данных на сервер → быстрее)
- Отказоустойчивость (падение одного шарда не убивает всю систему)

**Сложности**:
- **Сложные JOIN** между шардами
- **Транзакции**, затрагивающие несколько шардов
- **Перебалансировка** при добавлении/удалении шардов
- **Усложнение архитектуры** приложения
- **Сложности резервного копирования**

**Когда использовать шардинг**:
- Очень большие объёмы данных (терабайты+)
- Высокие требования к производительности записи
- Когда вертикальное масштабирование слишком дорого

**Когда не использовать**:
- Маленькие объёмы данных
- Частые сложные JOIN между разными частями данных
- Частые транзакции, затрагивающие несколько шардов
- Нет экспертизы по распределённым системам

## 24. Заключение: Единство многообразия

### 24.1. Синтез знаний

Базы данных — это огромный и разнообразный мир, где каждая технология решает свои задачи:
- **Реляционные БД** — для структурированных данных с гарантиями целостности
- **NoSQL БД** — для гибких, масштабируемых сценариев
- **SQL** — универсальный язык общения с реляционными данными
- **Индексы** — ускорители, за которые платим скоростью записи
- **Транзакции** — гарантии надёжности в многопользовательской среде
- **Нормализация** — искусство структурирования без избыточности
- **Шардинг** — путь к горизонтальному масштабированию

### 24.2. Эволюция мышления

Профессионал в области данных сегодня — это:
1. **Архитектор**, выбирающий правильный тип БД для каждой задачи
2. **Оптимизатор**, понимающий, как работают индексы и запросы
3. **Проектировщик**, создающий нормализованные, эффективные схемы
4. **Стратег**, планирующий масштабирование с помощью шардинга

### 24.3. Будущее баз данных

Тренды современности:
1. **Конвергенция** SQL и NoSQL (PostgreSQL с JSON, MongoDB с транзакциями)
2. **Специализация** — разные БД для разных задач в одной системе
3. **Облачные** managed-сервисы
4. **Горизонтальное масштабирование** как стандарт
5. **Обработка в реальном времени** (stream processing)

### 24.4. Искусство компромиссов

Работа с базами данных — это постоянный баланс:
- **Нормализация vs Производительность** — меньше JOIN vs целостность
- **Консистентность vs Доступность** — ACID vs BASE
- **Скорость чтения vs Скорость записи** — индексы помогают первому, мешают второму
- **Гибкость vs Структура** — NoSQL vs SQL
- **Простота vs Масштабируемость** — одна БД vs шардинг

Освоение этих концепций — это не конец обучения, а начало пути. Пути, на котором вы из пользователя баз данных становитесь архитектором информационных систем, способным выбирать правильные инструменты для правильных задач и создавать системы, которые масштабируются, работают быстро и сохраняют целостность данных.

В мире, где данные становятся важнейшим активом, эти знания — не просто технические навыки, а ключ к созданию систем, которые действительно работают и приносят пользу.
