# Лекция: Недостатки реляционных баз данных

## 1. Введение: Критический взгляд на РБД

**Контекст:** Несмотря на доминирование реляционных баз данных более 40 лет, у них существуют фундаментальные ограничения, которые становятся критичными в современных условиях больших данных, высоких нагрузок и требований к гибкости.

**Важный момент:** Рассмотрение недостатков не означает, что РБД плохи, а помогает понять, когда следует рассмотреть альтернативы или использовать компромиссные решения.

## 2. Архитектурные ограничения

### 2.1. Жесткость схемы данных

**Проблема:** Необходимость предварительного определения структуры данных.

```sql
-- Классический пример: требуется миграция для добавления поля
ALTER TABLE users ADD COLUMN social_media_links JSONB;
-- Все существующие записи получат NULL в этом поле
-- Может потребоваться обновление миллионов записей
```

**Последствия:**
- **Замедление разработки:** Каждое изменение схемы требует миграций
- **Простои в продакшене:** Миграции больших таблиц могут занимать часы
- **Риск ошибок:** Неверная миграция может привести к потере данных

### 2.2. Сложность горизонтального масштабирования

**Парадигма "шкафчика":**
```
Традиционные РБД спроектированы для вертикального масштабирования:
  │
  │   Добавляем больше ресурсов на один сервер
  ▼
[СУПЕР-СЕРВЕР]
  │
  │   Ограничения: стоимость, физические лимиты
  ▼
СТЕНА МАСШТАБИРУЕМОСТИ
```

**Теорема CAP и РБД:**
- РБД традиционно жертвуют **Availability** и **Partition Tolerance** в пользу **Consistency**
- В распределенных системах это становится проблемой

## 3. Производительностные недостатки

### 3.1. Проблема N+1 запроса

**Типичный сценарий в ORM:**
```python
# Получаем список заказов
orders = Order.objects.all()  # 1 запрос

for order in orders:  # N запросов
    # Для каждого заказа отдельно получаем клиента
    customer = order.customer  # SELECT * FROM customers WHERE id = ?
```

**Математика проблемы:**
```
1000 заказов → 1001 запрос к базе
Время выполнения: 1000 * latency + processing
```

**Решение с JOIN:**
```sql
SELECT o.*, c.* 
FROM orders o
JOIN customers c ON o.customer_id = c.id;
-- 1 запрос вместо 1001
```

### 3.2. Сложность работы с иерархическими данными

**Пример: древовидная структура категорий**
```
Электроника
├── Смартфоны
│   ├── Apple
│   ├── Samsung
│   └── Xiaomi
└── Ноутбуки
    ├── Игровые
    └── Ультрабуки
```

**Решения в РБД и их недостатки:**

| Метод | Запрос | Проблемы |
|-------|--------|----------|
| **Adjacency List** | Рекурсивные CTE | Сложные запросы, низкая производительность |
| **Nested Sets** | BETWEEN запросы | Сложные обновления, блокировки |
| **Materialized Path** | LIKE запросы | Нет индексов, полное сканирование |
| **Closure Table** | Много JOIN | Избыточность данных, сложность поддержки |

### 3.3. Проблема "горячих" строк

**Сценарий счетчика просмотров:**
```sql
-- Множество одновременных обновлений одной строки
UPDATE articles SET views = views + 1 WHERE id = 123;
-- Каждое обновление блокирует строку
-- При высокой нагрузке: deadlocks, ожидания
```

**Статистика блокировок в высоконагруженных системах:**
- До 30% времени транзакций может тратиться на ожидание блокировок
- В системах с 10K+ RPS deadlocks становятся регулярными

## 4. Операционные сложности

### 4.1. Стоимость владения

**Компоненты TCO (Total Cost of Ownership):**

| Компонент | Процент от TCO | Заметки |
|-----------|---------------|---------|
| Лицензии | 25-40% | Oracle: $47,500 за ядро, MS SQL: $7,128 за ядро |
| Администрирование | 30-45% | DBA, monitoring, backup |
| Аппаратное обеспечение | 15-25% | Сервера, SSD, память |
| Простои | 5-15% | Миграции, обновления, восстановление |

### 4.2. Сложность репликации и шардинга

**Типичные проблемы шардинга:**

1. **Горячие шарды:** Неравномерное распределение данных
2. **Межшардовые JOIN:** Невозможны или требуют сложных решений
3. **Консистентность:** Сложности с распределенными транзакциями
4. **Решардинг:** Миграция данных между шардами при изменении схемы

```sql
-- Что происходит при шардинге по customer_id?
SELECT SUM(total) FROM orders 
WHERE date BETWEEN '2024-01-01' AND '2024-12-31';
-- Запрос должен собрать данные со всех шардов
-- Нет гарантии согласованности snapshot
```

## 5. Функциональные ограничения

### 5.1. Ограничения модели "таблица-строка"

**Что плохо ложится на реляционную модель:**

1. **Графовые данные:**
```
Социальные сети: пользователи → друзья → друзья друзей
Запрос: "Найти общих друзей на расстоянии 3 рукопожатий"
-- В РБД требует множественных JOIN с непредсказуемой глубиной
```

2. **Временные ряды:**
```sql
-- Хранение миллиардов метрик
-- Проблемы: размер индексов, скорость вставки, компрессия
```

3. **Полнотекстовый поиск:**
```sql
-- LIKE '%поиск%' не использует индексы
-- Full-text search в РБД ограничен по сравнению со специализированными системами
```

### 5.2. Сложность работы с полуструктурированными данными

**Пример JSON в PostgreSQL:**
```sql
-- Хранение вариативных атрибутов товаров
INSERT INTO products (id, name, attributes) VALUES
(1, 'Смартфон', '{"color": "black", "storage": "128GB", "ram": "8GB"}'),
(2, 'Книга', '{"author": "Толстой", "pages": 500, "genre": "классика"}');

-- Проблемы:
-- 1. Нет строгой схемы → ошибки на уровне приложения
-- 2. Сложность валидации
-- 3. Ограниченная поддержка индексов
-- 4. Сложные запросы к вложенным структурам
```

## 6. Проблемы при работе в распределенных средах

### 6.1. Распределенные транзакции

**Two-Phase Commit (2PC) проблемы:**
```
Фаза 1: Подготовка → все участники блокируют ресурсы
Фаза 2: Фиксация или откат

Проблемы:
1. Блокировки на длительное время
2. Координатор - единая точка отказа
3. Сложность восстановления после сбоев
4. Высокая latency (RTT * 2 + processing)
```

### 6.2. Согласованность в конечном счете

**РБД требуют сильной согласованности, что ограничивает:**

1. **Геораспределение:** Задержки между датацентрами убивают производительность
2. **Офлайн-режим:** Невозможность работы без связи с основным датацентром
3. **Конфликтующие обновления:** Требуют сложных механизмов разрешения конфликтов

## 8. Реальные кейсы неудачного применения РБД

### 8.1. Социальная сеть (2010-е годы)

**Проблема:** Facebook с ростом до 500 млн пользователей
- MySQL не справлялся с нагрузкой на "лайки" и "комментарии"
- **Решение:** Создание специализированных систем (Tao, Cassandra)

**Числа:**
- 4.5 миллиарда "лайков" в день
- 10 миллионов новых фотографий в час
- Пиковая нагрузка: 2.1 миллиона запросов в секунду

### 8.2. IoT платформа

**Сценарий:** 100,000 устройств, каждое отправляет данные каждую секунду
```
100,000 * 3600 * 24 = 8.64 миллиарда записей в день
```

**Проблемы с РБД:**
1. Скорость вставки: максимум 50,000 записей/сек
2. Размер индексов: превышает размер данных в 3 раза
3. Очистка старых данных: блокировка таблицы на часах

## 9. Технический долг и антипаттерны

### 9.1. Распространенные антипаттерны

```sql
-- 1. EAV (Entity-Attribute-Value) - попытка обойти жесткость схемы
CREATE TABLE product_attributes (
    product_id INT,
    attribute_name VARCHAR(100),
    attribute_value TEXT
);
-- Проблемы: нет типизации, сложные запросы, низкая производительность

-- 2. Овернормализация
-- 8 нормальная форма "потому что так правильно"
-- Результат: 15 JOIN для простого запроса

-- 3. Фетишизация ACID
BEGIN TRANSACTION;
-- Транзакция на 5 минут для отправки email
-- Блокировка критических таблиц
COMMIT;
```

### 9.2. Проблема legacy-систем

**Типичный сценарий:**
- Система разработана 15 лет назад
- Оригинальная команда ушла
- Документации нет
- Изменения требуют месяцев анализа
- Миграция на новую версию СУБД невозможна

## 10. Будущее: куда движутся РБД

### 10.1. Эволюционные пути

1. **Гибридные подходы:**
   - PostgreSQL + TimescaleDB (для временных рядов)
   - MySQL + Vitess (для шардинга)
   - SQL Server + PolyBase (для работы с разными источниками)

2. **Новые парадигмы:**
   - NewSQL: CockroachDB, YugabyteDB
   - Serverless: Aurora Serverless, Cosmos DB с SQL API

### 10.2. Когда всё-таки выбирать РБД?

**Золотые правила выбора:**
1. **Данные структурированы** и редко меняют структуру
2. **Требуются сложные JOIN** и агрегации
3. **Критична согласованность** данных
4. **Команда имеет экспертизу** в SQL и оптимизации
5. **Нагрузка предсказуема** и растет линейно

## 11. Заключение: баланс и pragmatism

**Ключевые выводы:**

1. **РБД — не серебряная пуля:** У них есть фундаментальные ограничения
2. **Контекст решает:** Выбор БД должен основываться на требованиях, а не привычках
3. **Гибридные архитектуры:** Часто оптимальное решение — комбинация разных БД
4. **Стоимость миграции:** Переход с РБД на NoSQL может быть дороже, чем решение проблем в РБД

**Рекомендации:**
- Начинайте с РБД для нового проекта (быстрее разработка, предсказуемость)
- Мониторьте болевые точки (производительность, масштабирование)
- Рассматривайте специализированные решения для конкретных проблем
- Не бойтесь использовать несколько БД в одной системе
