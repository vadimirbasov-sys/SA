# Лекция: Реляционные базы данных. Основная информация

## 1. Введение

**Реляционная база данных (РБД)** — это тип базы данных, которая хранит и предоставляет доступ к взаимосвязанным данным. Основой реляционной модели является представление данных в виде **таблиц (отношений)**, состоящих из строк и столбцов.

**Ключевая идея:** Данные представляются в виде набора таблиц, между которыми устанавливаются связи (отношения), что и дало название модели.

## 2. Краткая история

*   **1970:** Эдгар Ф. Кодд (IBM) публикует статью «A Relational Model of Data for Large Shared Data Banks», в которой закладывает теоретические основы реляционной модели.
*   **1970-е — 1980-е:** Появление первых прототипов и коммерческих систем (System R, Ingres, Oracle).
*   **1986:** Стандартизация языка **SQL (Structured Query Language)** — основного языка для работы с реляционными БД.
*   **1990-е — настоящее время:** Доминирование реляционных СУБД на рынке, развитие стандартов SQL, появление мощных открытых решений (PostgreSQL, MySQL).

## 3. Основные понятия и структура

### 3.1. Таблица (Отношение, Relation)
Основная структурная единица. Представляет собой набор строк и столбцов, описывающих сущность предметной области.
*   **Примеры сущностей:** `Пользователи`, `Заказы`, `Товары`, `Категории`.

### 3.2. Строка (Запись, Кортеж, Tuple)
Одна запись в таблице, содержащая данные об одном экземпляре сущности.
*   **Пример:** Конкретный пользователь с именем "Иван Иванов".

### 3.3. Столбец (Поле, Атрибут, Attribute)
Определяет характеристику сущности. Имеет имя и тип данных.
*   **Типы данных:** `INTEGER` (целое число), `VARCHAR(n)` (строка), `DATE` (дата), `BOOLEAN` (логический) и др.
*   **Примеры столбцов для таблицы `Пользователи`:** `id`, `имя`, `email`, `дата_регистрации`.

### 3.4. Первичный ключ (Primary Key, PK)
Столбец (или набор столбцов), значение которого **уникально идентифицирует каждую строку** в таблице.
*   **Свойства:** Уникальность, неизменяемость (как правило), не может быть `NULL`.
*   **Пример:** `id` в таблице `Пользователи`.

### 3.5. Внешний ключ (Foreign Key, FK)
Столбец (или набор столбцов) в одной таблице, который ссылается на первичный ключ в другой таблице. Обеспечивает **связь между таблицами** и целостность данных.
*   **Пример:** В таблице `Заказы` есть столбец `user_id`, который является внешним ключом, ссылающимся на `id` в таблице `Пользователи`. Это показывает, какой пользователь сделал заказ.

### 3.6. Схема базы данных (Database Schema)
Структура всей базы данных: описание таблиц, их столбцов, типов данных, ключей и связей.

## 4. Связи между таблицами (Relationships)

Связь определяется с помощью внешних ключей. Основные типы связей:

| Тип связи | Описание | Пример |
| :--- | :--- | :--- |
| **Один-ко-многим (One-to-Many, 1:N)** | **Наиболее распространенная.** Одна запись в таблице А связана с несколькими записями в таблице Б. | **1** категория товаров -> **N** товаров в этой категории. |
| **Многие-ко-многим (Many-to-Many, M:N)** | Одна запись в А связана с несколькими в Б, и наоборот. Реализуется через **промежуточную таблицу** (junction table). | **M** студенты -> **N** курсы. Промежуточная таблица `Зачисления` содержит `student_id` и `course_id`. |
| **Один-к-одному (One-to-One, 1:1)** | Одна запись в А связана не более чем с одной записью в Б, и наоборот. Используется реже, часто для разделения таблиц по соображениям безопасности или производительности. | Основные данные `Пользователя` и его паспортные данные в отдельной защищенной таблице. |

## 5. Язык SQL (Structured Query Language)

Язык структурированных запросов — стандартный язык для управления реляционными базами данных. Включает несколько подмножеств:

*   **DDL (Data Definition Language) — Язык определения данных.**
    *   Создание и изменение структуры: `CREATE`, `ALTER`, `DROP`.
    ```sql
    CREATE TABLE Пользователи (
        id INTEGER PRIMARY KEY,
        имя VARCHAR(50) NOT NULL,
        email VARCHAR(100) UNIQUE
    );
    ```

*   **DML (Data Manipulation Language) — Язык манипуляции данными.**
    *   Работа с данными: `SELECT` (получение), `INSERT` (добавление), `UPDATE` (обновление), `DELETE` (удаление).
    ```sql
    SELECT имя, email FROM Пользователи WHERE id = 1;
    INSERT INTO Заказы (user_id, сумма) VALUES (1, 1500.00);
    ```

*   **DCL (Data Control Language) — Язык управления доступом.**
    *   Управление правами: `GRANT`, `REVOKE`.

*   **TCL (Transaction Control Language) — Язык управления транзакциями.**
    *   Управление транзакциями: `COMMIT`, `ROLLBACK`, `SAVEPOINT`.

## 6. ACID — свойства транзакций

Транзакция — логическая единица работы, состоящая из одного или нескольких операторов SQL. Гарантии ACID обеспечивают надежность и предсказуемость.

*   **A (Atomicity) — Атомарность:** Транзакция выполняется либо полностью, либо не выполняется вовсе.
*   **C (Consistency) — Согласованность:** Транзакция переводит базу данных из одного согласованного состояния в другое (не нарушаются ограничения целостности).
*   **I (Isolation) — Изолированность:** Параллельные транзакции не должны мешать друг другу.
*   **D (Durability) — Долговечность:** Если транзакция завершена (`COMMIT`), её результаты сохранены в БД навседа, даже в случае сбоя системы.

## 7. Нормализация

Процесс организации данных в базе для **устранения избыточности** и **уменьшения аномалий** при вставке, обновлении и удалении данных. Основные цели:
*   Исключение повторяющихся данных.
*   Обеспечение логической целостности данных.
*   Упрощение поддержки структуры БД.

Основные нормальные формы (1NF, 2NF, 3NF, BCNF) представляют собой последовательные шаги по улучшению структуры базы данных.

## 8. Преимущества и недостатки реляционных БД

### Преимущества:
*   **Стандартизация:** Широкое распространение SQL.
*   **Целостность данных:** Жесткая схема, ограничения, внешние ключи.
*   **Гибкость запросов:** Мощный язык SQL для сложных выборок и соединений.
*   **Надежность:** Свойства ACID, развитые механизмы транзакций и восстановления.
*   **Безопасность:** Развитые системы управления доступом.

### Недостатки:
*   **Сложность масштабирования:** Горизонтальное масштабирование (распределение на несколько серверов) затруднено по сравнению с NoSQL.
*   **Ограничения схемы:** Сложность изменения структуры данных на лету в быстро меняющихся проектах.
*   **Производительность на больших объемах:** Для некоторых специфических задач (например, хранение иерархических или графических данных) может быть менее эффективна, чем специализированные NoSQL-решения.
*   **Риск избыточного усложнения:** Неправильное проектирование (овернормализация) может привести к медленным запросам из-за большого числа соединений (`JOIN`).

## 9. Популярные реляционные СУБД

*   **Коммерческие:** Oracle Database, Microsoft SQL Server, IBM Db2.
*   **Открытые (Open Source):** PostgreSQL, MySQL / MariaDB, SQLite.

## 10. Заключение

Реляционные базы данных остаются краеугольным камнем для подавляющего большинства приложений, где важны надежность, согласованность и сложные связи между данными (финансовые системы, ERP, CRM, государственные реестры и многие другие). Понимание основ реляционной модели, SQL и принципов проектирования является фундаментальным навыком для разработчиков, аналитиков и администраторов данных.
