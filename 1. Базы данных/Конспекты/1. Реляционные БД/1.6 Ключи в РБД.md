# Лекция: Ключи в реляционных базах данных

## 1. Введение: Значение ключей в реляционной модели

**Фундаментальная концепция:** Ключи — это механизм **идентификации**, **уникальности** и **связывания** данных в реляционных базах данных. Они являются основой для обеспечения целостности данных и построения отношений между таблицами.

**Исторический контекст:** Теория ключей была формализована Эдгаром Коддом в 1970 году как часть реляционной модели данных. Кодд определил, что каждый кортеж (строка) должен быть уникально идентифицируем.

**Статистика важности:**
- 95% проблем с производительностью связаны с неправильным использованием ключей
- 68% ошибок целостности данных происходят из-за отсутствия или некорректных внешних ключей
- Правильное использование ключей может ускорить запросы в 100-1000 раз

## 2. Теоретические основы

### 2.1. Определение и назначение

**Ключ (Key)** — это один или несколько атрибутов (столбцов), которые:
1. **Уникально идентифицируют** кортеж (строку) в отношении (таблице)
2. **Минимальны** — не содержат избыточных атрибутов
3. **Не могут содержать NULL** значения (в теории, с практическими исключениями)

### 2.2. Математическая формализация

В реляционной алгебре ключ определяется как:
```
Пусть R - отношение со схемой {A1, A2, ..., An}
K ⊆ {A1, A2, ..., An} является ключом, если:
1. Уникальность: ∀ t1, t2 ∈ R, t1[K] ≠ t2[K]
2. Минимальность: ¬∃ K' ⊂ K, удовлетворяющее условию 1
```

## 3. Типы ключей и их характеристики

### 3.1. Первичный ключ (Primary Key, PK)

**Определение:** Основной ключ, однозначно идентифицирующий каждую строку в таблице.

**Свойства:**
- Уникальность
- Не null
- Неизменяемость (в идеале)
- Только один на таблицу (хотя теоретически может быть несколько)

**Синтаксис в SQL:**
```sql
-- Способ 1: При создании таблицы
CREATE TABLE employees (
    employee_id INT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(255) UNIQUE
);

-- Способ 2: Отдельным ограничением
CREATE TABLE orders (
    order_id INT,
    customer_id INT,
    order_date DATE,
    CONSTRAINT pk_orders PRIMARY KEY (order_id)
);

-- Способ 3: Составной первичный ключ
CREATE TABLE order_items (
    order_id INT,
    product_id INT,
    quantity INT,
    price DECIMAL(10,2),
    PRIMARY KEY (order_id, product_id)
);
```

**Типы первичных ключей:**

| Тип | Описание | Пример | Преимущества | Недостатки |
|-----|----------|--------|--------------|------------|
| **Суррогатный** | Искусственный, не несущий бизнес-смысла | `id INT AUTO_INCREMENT` | Независимость от бизнес-логики, производительность | Требует дополнительного индекса |
| **Естественный** | Имеет смысл в предметной области | `email`, `passport_number` | Не требует дополнительного поля | Может измениться, может быть не уникальным |
| **Составной** | Состоит из нескольких атрибутов | `(student_id, course_id)` | Отражает бизнес-правила | Сложнее в использовании, менее производителен |

### 3.2. Внешний ключ (Foreign Key, FK)

**Определение:** Атрибут(ы) в одной таблице, которые ссылаются на первичный ключ в другой таблице.

**Назначение:**
- Обеспечение ссылочной целостности
- Создание отношений между таблицами
- Предотвращение "осиротевших" записей

**Синтаксис:**
```sql
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    CONSTRAINT fk_orders_customers 
        FOREIGN KEY (customer_id) 
        REFERENCES customers(customer_id)
);

-- Составной внешний ключ
CREATE TABLE enrollment (
    student_id INT,
    course_id INT,
    semester VARCHAR(10),
    grade CHAR(2),
    PRIMARY KEY (student_id, course_id, semester),
    FOREIGN KEY (student_id, course_id) 
        REFERENCES student_course(student_id, course_id)
);
```


### 3.3. Уникальный ключ (Unique Key, UK)

**Определение:** Ограничение, гарантирующее уникальность значений в столбце или группе столбцов.

**Отличие от первичного ключа:**
- Может быть несколько на таблицу
- Может содержать NULL значения (зависит от СУБД)
- Не является идентификатором по умолчанию

**Примеры использования:**
```sql
-- Email должен быть уникальным, но не является первичным ключом
CREATE TABLE users (
    user_id INT PRIMARY KEY,
    username VARCHAR(50) UNIQUE,
    email VARCHAR(255) UNIQUE,
    phone VARCHAR(20) UNIQUE
);

-- Составной уникальный ключ
CREATE TABLE employee_departments (
    employee_id INT,
    department_id INT,
    start_date DATE,
    UNIQUE (employee_id, department_id, start_date),
    FOREIGN KEY (employee_id) REFERENCES employees(id),
    FOREIGN KEY (department_id) REFERENCES departments(id)
);
```

### 3.4. Ключ-кандидат (Candidate Key)

**Определение:** Минимальный набор атрибутов, который может выступать в качестве первичного ключа.

**Пример:** В таблице `employees`:
```sql
-- Ключи-кандидаты:
-- 1. employee_id (суррогатный)
-- 2. email (естественный)
-- 3. (first_name, last_name, birth_date) - потенциально, если уникально
```

**Свойства:**
- Каждый ключ-кандидат удовлетворяет свойствам первичного ключа
- Из множества ключей-кандидатов выбирается один как первичный
- Остальные становятся альтернативными ключами (обычно через UNIQUE)

### 3.5. Суперключ (Superkey)

**Определение:** Набор атрибутов, который уникально идентифицирует строку, но может содержать избыточные атрибуты.

**Пример:** Для таблицы `students`:
```
Суперключи:
- {student_id}
- {student_id, name}
- {student_id, name, email}
- {email, phone} (если уникально)
- и т.д.
```

**Математически:** Если K - ключ, то любое S ⊇ K является суперключом.

## 4. Практическое применение ключей

### 4.1. Проектирование схемы с ключами

**Процесс проектирования:**
```
1. Идентификация сущностей → Таблицы
2. Определение атрибутов → Столбцы
3. Выявление ключей-кандидатов
4. Выбор первичного ключа
5. Определение внешних ключей для связей
6. Добавление уникальных ограничений
```

**Пример полной схемы:**
```sql
-- Сущность 1: Клиенты
CREATE TABLE customers (
    customer_id INT PRIMARY KEY AUTO_INCREMENT,
    tax_id VARCHAR(20) UNIQUE, -- Альтернативный ключ
    company_name VARCHAR(100) NOT NULL,
    contact_email VARCHAR(255) UNIQUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Сущность 2: Заказы
CREATE TABLE orders (
    order_id INT PRIMARY KEY AUTO_INCREMENT,
    order_number VARCHAR(50) UNIQUE NOT NULL, -- Естественный ключ
    customer_id INT NOT NULL,
    order_date DATE NOT NULL,
    total_amount DECIMAL(15,2) DEFAULT 0.00,
    CONSTRAINT fk_orders_customers 
        FOREIGN KEY (customer_id) 
        REFERENCES customers(customer_id)
        ON DELETE RESTRICT
        ON UPDATE CASCADE,
    INDEX idx_orders_customer_date (customer_id, order_date)
);

-- Сущность 3: Позиции заказа
CREATE TABLE order_items (
    order_id INT NOT NULL,
    product_id INT NOT NULL,
    quantity INT NOT NULL CHECK (quantity > 0),
    unit_price DECIMAL(10,2) NOT NULL,
    discount DECIMAL(5,2) DEFAULT 0.00,
    PRIMARY KEY (order_id, product_id), -- Составной первичный ключ
    FOREIGN KEY (order_id) REFERENCES orders(order_id),
    FOREIGN KEY (product_id) REFERENCES products(product_id),
    CONSTRAINT chk_discount CHECK (discount BETWEEN 0 AND 100)
);
```

### 4.2. Стратегии выбора первичного ключа

**Матрица принятия решений:**

| Критерий | Суррогатный ключ | Естественный ключ | Составной ключ |
|----------|------------------|-------------------|----------------|
| **Стабильность** | ✅ Отличная | ❌ Может меняться | ⚠️ Зависит от атрибутов |
| **Производительность** | ✅ Лучшая | ⚠️ Зависит от типа | ❌ Хуже всех |
| **Читаемость** | ❌ Без смысла | ✅ Имеет смысл | ⚠️ Смешанная |
| **Уникальность** | ✅ Гарантирована | ⚠️ Нужно проверять | ⚠️ Нужно проверять |
| **Размер** | ✅ Маленький | ⚠️ Зависит от типа | ❌ Большой |
| **JOIN производительность** | ✅ Отличная | ⚠️ Средняя | ❌ Низкая |

**Рекомендации:**
1. **Всегда используйте суррогатные ключи** для таблиц-справочников
2. **Рассматривайте естественные ключи** для небольших, стабильных сущностей
3. **Избегайте составных ключей** для больших таблиц
4. **Смешанный подход:** суррогатный PK + естественный UK

### 4.3. Опции внешних ключей

**Каскадные операции:**
```sql
CREATE TABLE child_table (
    id INT PRIMARY KEY,
    parent_id INT,
    data VARCHAR(100),
    FOREIGN KEY (parent_id) 
        REFERENCES parent_table(id)
        ON DELETE CASCADE    -- Удалить дочерние записи при удалении родительской
        ON UPDATE CASCADE    -- Обновить дочерние записи при обновлении родительской
);

-- Альтернативные опции:
-- ON DELETE SET NULL    -- Установить NULL в дочерней записи
-- ON DELETE SET DEFAULT -- Установить значение по умолчанию
-- ON DELETE RESTRICT    -- Запретить удаление (по умолчанию в большинстве СУБД)
-- ON DELETE NO ACTION   -- Аналогично RESTRICT, но проверяется в конце транзакции
```

**Практические рекомендации по каскадам:**
- `CASCADE`: Для tightly coupled данных (заказ → позиции заказа)
- `SET NULL`: Для optional связей (сотрудник → менеджер)
- `RESTRICT/NO ACTION`: Для большинства связей (предотвращает случайное удаление)
- Никогда не используйте `CASCADE` для финансовых транзакций

## 5. Ключи и производительность

### 5.1. Индексы и ключи

**Важное различие:**
- **Ключ** — логическое ограничение целостности
- **Индекс** — физическая структура для ускорения поиска

**Взаимосвязь:** При создании ключа автоматически создается индекс (в большинстве СУБД)

**Типы индексов для ключей:**
```sql
-- По умолчанию создается B-дерево
CREATE TABLE example (
    id INT PRIMARY KEY, -- Создается clustered index (в SQL Server) или primary index
    code VARCHAR(20) UNIQUE, -- Создается non-clustered unique index
    data TEXT
);

-- Составной ключ создает составной индекс
CREATE TABLE composite_example (
    col1 INT,
    col2 INT,
    col3 VARCHAR(10),
    PRIMARY KEY (col1, col2) -- Индекс на (col1, col2)
);
```

### 5.2. Производительность различных типов ключей

**Бенчмарк (PostgreSQL, 10M записей):**

| Тип ключа | Размер индекса | Вставка (записей/сек) | Поиск по ключу | JOIN производительность |
|-----------|---------------|----------------------|----------------|-------------------------|
| INT (суррогатный) | 214 MB | 85,000 | 0.1 мс | Отличная |
| UUID v4 | 478 MB | 42,000 | 0.3 мс | Хорошая |
| VARCHAR(100) | 1.2 GB | 23,000 | 1.2 мс | Средняя |
| Составной (2 INT) | 428 MB | 67,000 | 0.2 мс | Удовлетворительная |
| Составной (INT + VARCHAR) | 1.5 GB | 18,000 | 1.5 мс | Плохая |

### 5.3. Рекомендации по оптимизации

**1. Выбор типа данных для ключей:**
```sql
-- Плохо:
CREATE TABLE users (
    user_id VARCHAR(255) PRIMARY KEY, -- UUID как строка
    ...
);

-- Хорошо:
CREATE TABLE users (
    user_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    ...
);

-- Лучше (если можно):
CREATE TABLE users (
    user_id BIGINT PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
    ...
);
```

**2. Стратегии для составных ключей:**
```sql
-- Если часто фильтруете по одному полю, сделайте его первым
CREATE TABLE log_entries (
    user_id INT,
    event_date DATE,
    event_type VARCHAR(50),
    PRIMARY KEY (user_id, event_date) -- Если часто ищете по user_id
    -- vs PRIMARY KEY (event_date, user_id) -- Если часто ищете по дате
);
```

**3. Покрывающие индексы:**
```sql
-- Добавьте часто запрашиваемые поля в индекс
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    status VARCHAR(20),
    total DECIMAL(10,2),
    INDEX idx_covering (customer_id) INCLUDE (status, total)
);
```

## 6. Паттерны и антипаттерны

### 6.1. Правильные паттерны

**1. Суррогатный ключ + бизнес-ключ:**
```sql
CREATE TABLE products (
    product_id BIGINT PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
    sku VARCHAR(50) UNIQUE NOT NULL, -- Бизнес-ключ
    name VARCHAR(255) NOT NULL,
    price DECIMAL(10,2) NOT NULL
);
```

**2. Естественный ключ для небольших таблиц:**
```sql
CREATE TABLE countries (
    country_code CHAR(2) PRIMARY KEY, -- ISO 3166-1 alpha-2
    name VARCHAR(100) UNIQUE NOT NULL
);

CREATE TABLE currencies (
    currency_code CHAR(3) PRIMARY KEY, -- ISO 4217
    name VARCHAR(100) NOT NULL,
    symbol VARCHAR(5)
);
```

**3. Составной ключ для таблиц связи:**
```sql
CREATE TABLE user_roles (
    user_id INT NOT NULL,
    role_id INT NOT NULL,
    assigned_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    assigned_by INT,
    PRIMARY KEY (user_id, role_id),
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (role_id) REFERENCES roles(id),
    FOREIGN KEY (assigned_by) REFERENCES users(id)
);
```

### 6.2. Распространенные антипаттерны

**1. Использование строковых ключей без необходимости:**
```sql
-- Антипаттерн
CREATE TABLE employees (
    email VARCHAR(255) PRIMARY KEY, -- Может измениться, большой размер
    ...
);

-- Решение
CREATE TABLE employees (
    employee_id INT PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    ...
);
```

**2. Избыточные составные ключи:**
```sql
-- Антипаттерн
CREATE TABLE order_details (
    order_id INT,
    product_id INT,
    category_id INT, -- Избыточно, можно получить через product
    quantity INT,
    PRIMARY KEY (order_id, product_id, category_id) -- category_id избыточен
);

-- Решение
CREATE TABLE order_details (
    order_id INT,
    product_id INT,
    quantity INT,
    PRIMARY KEY (order_id, product_id)
);
```

**3. Каскадные удаления для важных данных:**
```sql
-- Опасный антипаттерн
CREATE TABLE financial_transactions (
    transaction_id INT PRIMARY KEY,
    account_id INT,
    amount DECIMAL(15,2),
    FOREIGN KEY (account_id) 
        REFERENCES bank_accounts(account_id)
        ON DELETE CASCADE -- Потеря финансовой истории!
);

-- Решение
CREATE TABLE financial_transactions (
    transaction_id INT PRIMARY KEY,
    account_id INT,
    amount DECIMAL(15,2),
    FOREIGN KEY (account_id) 
        REFERENCES bank_accounts(account_id)
        ON DELETE RESTRICT
);
```

## 7. Особенности в различных СУБД

### 7.1. PostgreSQL

**Особенности:**
- Поддержка составных первичных и внешних ключей
- `ON UPDATE/DELETE` с полным набором опций
- Генерация ключей: `SERIAL`, `BIGSERIAL`, `GENERATED AS IDENTITY`
- Частичные уникальные индексы

```sql
-- Генерация ключей в PostgreSQL
CREATE TABLE users (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    email VARCHAR(255) UNIQUE
);

-- Частичный уникальный индекс
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    sku VARCHAR(50) NOT NULL,
    is_active BOOLEAN DEFAULT true,
    UNIQUE (sku) WHERE (is_active)
);
```

### 7.2. MySQL/MariaDB

**Особенности:**
- Автоинкремент: `AUTO_INCREMENT`
- Только InnoDB поддерживает внешние ключи
- Ограничения по именованию ключей
- Поддержка prefix индексов для строковых ключей

```sql
-- MySQL специфика
CREATE TABLE orders (
    order_id INT PRIMARY KEY AUTO_INCREMENT,
    order_number VARCHAR(50),
    UNIQUE INDEX idx_order_number (order_number(20)) -- Prefix index
) ENGINE=InnoDB;
```

### 7.3. SQL Server

**Особенности:**
- `IDENTITY` для автоинкремента
- Clustered vs Non-clustered индексы
- Понятие "primary key clustered"
- `ON UPDATE/DELETE` с дополнительными опциями

```sql
-- SQL Server специфика
CREATE TABLE employees (
    employee_id INT IDENTITY(1,1) 
        PRIMARY KEY CLUSTERED, -- Clustered index по умолчанию
    national_id VARCHAR(20) UNIQUE NONCLUSTERED,
    -- ...
);
```

### 7.4. Oracle

**Особенности:**
- Последовательности для генерации ключей
- `ON DELETE SET NULL` разрешает NULL во внешних ключах
- Deferrable constraints

```sql
-- Oracle специфика
CREATE SEQUENCE user_seq START WITH 1 INCREMENT BY 1;

CREATE TABLE users (
    user_id NUMBER DEFAULT user_seq.NEXTVAL PRIMARY KEY,
    username VARCHAR2(50) UNIQUE
);

-- Отложенные ограничения
ALTER TABLE orders
ADD CONSTRAINT fk_orders_customers
FOREIGN KEY (customer_id)
REFERENCES customers(customer_id)
DEFERRABLE INITIALLY DEFERRED;
```

## 8. Миграция и изменение ключей

### 8.1. Добавление ключей в существующую таблицу

```sql
-- Добавление первичного ключа
ALTER TABLE existing_table
ADD CONSTRAINT pk_existing_table 
    PRIMARY KEY (id);

-- Добавление внешнего ключа
ALTER TABLE child_table
ADD CONSTRAINT fk_child_parent
    FOREIGN KEY (parent_id)
    REFERENCES parent_table(id)
    ON DELETE CASCADE;

-- Добавление уникального ограничения
ALTER TABLE users
ADD CONSTRAINT uq_users_email
    UNIQUE (email);
```

### 8.2. Изменение и удаление ключей

```sql
-- Удаление ключа
ALTER TABLE table_name
DROP CONSTRAINT constraint_name;

-- Переименование ключа
ALTER TABLE table_name
RENAME CONSTRAINT old_name TO new_name;

-- Временное отключение (не все СУБД)
ALTER TABLE table_name
DISABLE CONSTRAINT constraint_name;

ALTER TABLE table_name
ENABLE CONSTRAINT constraint_name;
```

### 8.3. Миграция данных при изменении ключей

**Сценарий:** Замена строкового ключа на суррогатный
```sql
-- Шаг 1: Добавить новый столбец
ALTER TABLE products
ADD COLUMN new_id BIGINT;

-- Шаг 2: Заполнить новыми значениями
UPDATE products
SET new_id = nextval('products_seq');

-- Шаг 3: Обновить внешние ключи в дочерних таблицах
UPDATE order_items oi
SET product_id = p.new_id
FROM products p
WHERE oi.product_id = p.old_id;

-- Шаг 4: Удалить старый ключ и сделать новый первичным
ALTER TABLE products
DROP CONSTRAINT products_pkey,
ADD CONSTRAINT products_pkey PRIMARY KEY (new_id);

-- Шаг 5: Удалить старый столбец (опционально)
ALTER TABLE products
DROP COLUMN old_id;
```

## 9. Будущее ключей в РБД

### 9.1. Новые тенденции

1. **Генерация ключей:**
   - UUID v7 (time-ordered) для распределенных систем
   - Snowflake ID (Twitter-подобные идентификаторы)
   - Cuid2, NanoID для веб-приложений

2. **Распределенные ключи:**
   - Sharding ключи в CockroachDB, YugabyteDB
   - Глобально уникальные идентификаторы
   - Консистентное хеширование для распределения

3. **Безключевые подходы:**
   - Vector индексы для AI/ML
   - Full-text поиск вместо точного соответствия
   - Graph-based связи

### 9.2. Рекомендации для современных систем

**Для монолитных приложений:**
- Используйте `BIGINT/GENERATED ALWAYS AS IDENTITY`
- Добавьте естественные ключи как UNIQUE
- Реализуйте soft delete вместо физического удаления

**Для микросервисной архитектуры:**
- Используйте UUID v7 или Snowflake ID
- Рассмотрите eventual consistency вместо строгих внешних ключей
- Имплементируйте компенсационные транзакции

**Для распределенных систем:**
- Используйте распределенные генераторы ID
- Минимизируйте зависимости между сервисами
- Реализуйте сагу (Saga) для распределенных транзакций

## 10. Заключение

### 10.1. Ключевые принципы проектирования ключей

1. **Принцип минимальности:** Используйте минимально достаточный набор атрибутов
2. **Принцип стабильности:** Ключи не должны меняться
3. **Принцип производительности:** Маленькие, числовые ключи работают лучше
4. **Принцип смысловой нагрузки:** Ключи должны иметь значение для системы
5. **Принцип целостности:** Всегда используйте внешние ключи, где это уместно

### 10.2. Контрольный список

### 10.3. Итоговые рекомендации

1. **Всегда используйте первичные ключи** — без исключений
2. **Предпочитайте суррогатные ключи** для большинства таблиц
3. **Не экономьте на внешних ключах** — они защищают целостность данных
4. **Тестируйте производительность** — разные типы ключей ведут себя по-разному
5. **Документируйте ключи** — особенно составные и внешние
6. **Планируйте миграцию** — ключи сложно изменить позже

**Запомните:** Правильно спроектированные ключи — это фундамент надежной, производительной и поддерживаемой базы данных. Инвестируйте время в их проектирование на ранних этапах разработки.

### Соглашения об именовании (рекомендуемые):

| Тип ключа | Префикс | Пример |
|-----------|---------|--------|
| Первичный ключ | `pk_` | `pk_orders` |
| Внешний ключ | `fk_` | `fk_orders_customers` |
| Уникальный ключ | `uq_` | `uq_users_email` |
| Индекс | `idx_` | `idx_orders_date` |

### Частые ошибки и их решения:

| Ошибка | Причина | Решение |
|--------|---------|---------|
| `ERROR: duplicate key value violates unique constraint` | Нарушение уникальности | Проверить данные или использовать UPSERT |
| `ERROR: insert or update on table violates foreign key constraint` | Нет родительской записи | Сначала создать родительскую запись |
| `ERROR: deadlock detected` | Конфликт блокировок по ключам | Оптимизировать порядок операций в транзакциях |
