Отлично, вы перешли от теории протоколов и моделей к **практике** — как всё это живёт в реальных проектах. **Инфраструктура** — это совокупность всех аппаратных и программных компонентов, которые обеспечивают работу ваших сервисов, API и приложений.

Если API и микросервисы — это «душа» проекта, то инфраструктура — это «тело»: серверы, сети, базы данных, очереди, контейнеры, системы мониторинга и т.д.

---

## Что входит в инфраструктуру проекта?

Укрупнённо инфраструктуру можно разделить на несколько слоёв:

| Слой | Что включает | Примеры технологий |
|------|--------------|---------------------|
| **Вычислительные ресурсы** | Где исполняется код | Физические серверы, виртуальные машины (VM), контейнеры (Docker), бессерверные (AWS Lambda) |
| **Сеть** | Как компоненты общаются | Локальные сети, VLAN, DNS, балансировщики нагрузки (NGINX, HAProxy), CDN, VPN |
| **Хранение данных** | Где лежат данные | Базы данных (PostgreSQL, MySQL, MongoDB), объектное хранилище (S3), кэши (Redis, Memcached) |
| **Очереди и стриминг** | Асинхронное взаимодействие | RabbitMQ, Apache Kafka, Amazon SQS, Redis Pub/Sub |
| **Оркестрация** | Управление контейнерами и сервисами | Kubernetes (K8s), Docker Swarm, Nomad |
| **CI/CD** | Автоматизация сборки и развертывания | GitLab CI, GitHub Actions, Jenkins, ArgoCD |
| **Мониторинг и логи** | Наблюдение за состоянием | Prometheus + Grafana, ELK/EFK (Elasticsearch, Logstash/Fluentd, Kibana), Datadog, Sentry |
| **Безопасность** | Защита данных и доступа | IAM, Vault (секреты), WAF, сертификаты TLS, сетевые политики |
| **Инфраструктура как код (IaC)** | Управление через конфиги | Terraform, Ansible, Pulumi, CloudFormation |

## Типы инфраструктуры

### 1. On‑premise (на своей «железке»)
- Серверы стоят в вашем дата‑центре или офисе.
- Полный контроль, но высокие затраты на оборудование, электричество, охлаждение, администрирование.
- Используется в банках, госсекторе, где данные нельзя выносить в облако.

### 2. Облачная (public cloud)
- Аренда ресурсов у провайдера (AWS, Google Cloud, Microsoft Azure, Yandex Cloud, VK Cloud).
- Платите только за потребление, легко масштабировать.
- Большинство современных проектов (стартапы, mid‑size, enterprise) используют облака.

### 3. Гибридная
- Часть ресурсов on‑premise, часть в облаке.
- Например, базы данных с персональными данными оставляют у себя, а статику и вычислительные задачи выносят в облако.

### 4. Бессерверная (Serverless)
- Вы не управляете серверами вообще. Пишете функции (например, AWS Lambda), а облако само запускает их под запрос.
- Идеально для API с неравномерной нагрузкой, обработки событий.

## Как выглядит инфраструктура для микросервисного проекта (пример)

Представим интернет-магазин. Его инфраструктура может быть такой:

```
[ Пользователь ] → [ CDN (CloudFlare) ] → [ Балансировщик (NGINX) ] → [ API Gateway (Kong / Traefik) ]
                                                                              │
                                    ┌─────────────────────────────────────────┼─────────────────────────────────────────┐
                                    ↓                                         ↓                                         ↓
                          [ Микросервис каталога ]                  [ Микросервис корзины ]                 [ Микросервис платежей ]
                                    │                                         │                                         │
                                    ↓                                         ↓                                         ↓
                          [ PostgreSQL ]                               [ Redis (кэш) ]                       [ Kafka (очередь) ]
                                                                                                                │
                                                                                                                ↓
                                                                                                      [ Микросервис уведомлений ]
                                                                                                      (email, SMS)
```

Поверх всего этого:
- **Kubernetes** управляет микросервисами (автоматически перезапускает, масштабирует).
- **Prometheus** собирает метрики (CPU, память, число запросов).
- **Grafana** показывает дашборды.
- **ELK** собирает логи (чтобы искать ошибки).
- **GitLab CI** при каждом коммите собирает Docker-образы и катит их в кластер.

## Связь с предыдущими темами (протоколы, OSI, TCP/IP)

Инфраструктура «заземляет» теоретические концепции:

| Концепция | Как реализована в инфраструктуре |
|-----------|----------------------------------|
| **HTTP/HTTPS** | Балансировщики и API Gateway работают с этими протоколами, часто заканчивают TLS. |
| **TCP/UDP** | Серверы открывают порты (например, 80, 443, 3306 для MySQL). Сетевые политики (firewall) пропускают или блокируют трафик. |
| **IP-маршрутизация** | VPC (виртуальные частные облака), подсети, маршрутизаторы в облаке или on‑premise. |
| **DNS** | Пользователь вводит `shop.com` → DNS преобразует в IP балансировщика. |
| **TLS/SSL** | Сертификаты хранятся в менеджерах секретов (Vault, AWS Certificate Manager). Балансировщик или Ingress‑контроллер терминует TLS. |
| **Микросервисы** | Каждый микросервис — это набор контейнеров (подов в K8s) с собственным IP и портом. |
| **Очереди (AMQP, Kafka)** | Отдельные кластеры брокеров, обеспечивающие асинхронное взаимодействие. |

| Стадия | Инфраструктура |
|--------|----------------|

По мере роста вы добавите контейнеры, оркестрацию, систему логирования и т.д.

- **Инфраструктура** — это всё, что нужно для работы вашего кода «в поле», от серверов до систем мониторинга.

Понимание того, что происходит "под капотом" браузера, — ключ к написанию эффективного и производительного кода. То, что пользователь видит на экране, — результат сложного, но очень структурированного процесса.

### ⚙️ "Под капотом": От кода до пикселей

Путь от написанного вами кода до готовой страницы называется **критическим путём рендеринга (Critical Rendering Path)**. Это последовательность шагов, которые браузер выполняет, чтобы отобразить страницу.

1.  **Построение DOM (Document Object Model)**: Браузер читает HTML-код и преобразует его в **DOM** — древовидную модель, представляющую структуру и содержимое страницы.
2.  **Построение CSSOM (CSS Object Model)**: Параллельно браузер загружает и анализирует CSS, создавая **CSSOM** — модель, описывающую стили для каждого элемента.
3.  **Создание Render Tree**: DOM и CSSOM объединяются в **Render Tree** («дерево рендеринга»). В него попадают только видимые элементы с их финальными стилями. Элементы со свойством `display: none` в него не включаются.
4.  **Layout (Верстка)**: На основе Render Tree браузер вычисляет геометрию: точные координаты (`x`, `y`) и размеры (`width`, `height`) каждого элемента на странице. Этот процесс также называют **reflow**.
5.  **Paint (Отрисовка)**: Браузер разбивает страницу на слои (например, слой для основного контента, слой для всплывающих окон) и заполняет пиксели для каждого из них — заливает цветом фона, рисует линии и текст. Этот процесс также называют **repaint**.
6.  **Composite (Композиция)**: Финальный этап, на котором слои, полученные на предыдущем шаге, накладываются друг на друга в правильном порядке и выводятся на экран.

> **Важно**: Понимание этого процесса помогает писать быстрый код. Например, частое чтение/запись свойств, влияющих на геометрию (`offsetWidth`, `getComputedStyle`), может вызывать принудительный **synchronous layout**, замедляющий работу.

### 🧱 Технологии и Инструменты

Современный фронтенд — это экосистема, которая постоянно развивается. Вот её ключевые элементы:

**Языки (Основы)**
*   **HTML**: Определяет структуру и семантику (смысловое значение) контента. Браузер преобразует его в DOM.
*   **CSS**: Отвечает за визуальное оформление и адаптивность. Трансформируется в CSSOM.
*   **JavaScript**: Обеспечивает интерактивность, динамически изменяя DOM и CSSOM в ответ на действия пользователя.

**Фреймворки (Библиотеки)**
*   **React**: Самый популярный фреймворк с огромным комьюнити, экосистемой и поддержкой от Meta. Использует виртуальный DOM для эффективного обновления интерфейса.
*   **Vue**: Ценится за низкий порог входа, понятную документацию и гибкость. Популярен в中小-проектах.
*   **Angular**: Мощный фреймворк от Google, предоставляющий "из коробки" всё необходимое для крупных корпоративных приложений.
*   **Svelte**: Выполняет компиляцию на этапе сборки, в результате в браузер попадает чистый, оптимизированный JS-код, что делает приложения очень быстрыми.

**Мета-фреймворки (Next.js, Nuxt, SvelteKit)**
*   Добавляют поверх базовых фреймворков расширенные функции: серверный рендеринг (SSR), генерацию статических сайтов (SSG), маршрутизацию и оптимизацию изображений.

**Сборщики (Bundlers)**
*   **Vite**: Быстрый сборщик, использующий нативные ES-модули во время разработки.
*   **Webpack**: Мощный и гибкий сборщик, ставший стандартом, особенно в экосистеме React.
*   **esbuild**: Чрезвычайно быстрый сборщик, написанный на Go, часто используется внутри Vite и других инструментов.

**Стилизация**
*   **CSS-in-JS (styled-components, Emotion)**: Позволяет писать CSS-код прямо внутри JS/TS-компонентов, инкапсулируя стили.
*   **Utility-first (Tailwind CSS)**: Предоставляет готовые CSS-классы (например, `flex`, `pt-4`, `text-center`) для быстрой стилизации прямо в разметке.
*   **CSS-препроцессоры (Sass, Less)**: Расширяют возможности обычного CSS, добавляя переменные, вложенности, миксины и другие конструкции.

### 🧪 Тестирование: Как проверять

Тестирование во фронтенде строится по принципу "пирамиды", где основание — быстрые и дешевые тесты, а вершина — медленные и дорогие, но дающие максимальную уверенность.

**Виды тестов**
*   **Unit-тесты**: Проверяют работу маленьких изолированных кусочков кода: функций, утилит, простых компонентов.
*   **Интеграционные тесты**: Проверяют, как несколько частей системы работают вместе. Например, взаимодействие компонента с хранилищем данных (state manager) и API-клиентом.
*   **E2E-тесты (End-to-End)**: Симулируют действия реального пользователя в настоящем браузере, проверяя критические бизнес-сценарии (регистрация, оформление заказа) от начала до конца.
*   **Визуальное (Snapshot) тестирование**: Делает "снимок" UI компонента и сравнивает его с эталоном при каждом изменении кода.

**Инструменты**
*   **Jest**: Универсальный фреймворк для unit- и интеграционного тестирования, популярный в React-сообществе.
*   **Vitest**: Быстрая альтернатива Jest, созданная для проектов на Vite.
*   **React Testing Library (RTL)**: Стандарт для тестирования React-компонентов, поощряющий проверку поведения с точки зрения пользователя.
*   **Cypress**: Инструмент для E2E-тестирования с удобным графическим интерфейсом и отличным отладчиком.
*   **Playwright**: Современный фреймворк для E2E-тестирования, поддерживающий все основные браузеры (Chrome, Firefox, Safari).
*   **Storybook**: Инструмент для разработки и тестирования UI-компонентов в изоляции от основного приложения.

### 🚀 Тренды фронтенда в 2025 году

Сейчас индустрия переживает несколько важных изменений:

*   **Server-First подход**: Тяжелые вычисления и даже часть рендеринга уходят на сервер (например, через React Server Components), чтобы уменьшить объем JavaScript-кода, отправляемого клиенту, и ускорить загрузку страниц.
*   **Зрелость и стабильность**: Основные фреймворки (React, Vue, Angular) достигли высокой степени зрелости. Конкуренция сместилась в область мета-фреймворков и инструментов сборки.
*   **Интеграция AI**: Искусственный интеллект активно используется для автодополнения кода, генерации тестов, рефакторинга и анализа кода.
*   **Производительность и доступность (A11y)**: Эти аспекты перестали быть просто "хорошей практикой" и превратились в обязательные требования, влияющие на бизнес-метрики и юридическую чистоту продукта.

## ⚙️ Что такое Middle Layer (Миддл) «под капотом»

Миддл — это **один или несколько сервисов**, которые:
- Принимают запросы от клиентов (браузер, мобильное приложение, другой сервис).
- Выполняют бизнес-логику (проверки, вычисления, преобразования).
- Общаются с базами данных, кэшами, очередями, внешними API.
- Возвращают ответ клиенту.

По сути, это **тот самый бэкенд**, который вы пишете на Node.js, Python, Go, Java, C# и т.д. Но в крупных системах миддл состоит из нескольких слоёв:

1. **Входной слой (API Gateway / Reverse Proxy)** — приём запросов, TLS-терминация, rate limiting, маршрутизация.
2. **Слой бизнес-логики (Application Servers)** — ваши сервисы (микросервисы, монолитные приложения). Здесь выполняются основные действия: регистрация пользователя, расчёт корзины, отправка уведомлений.
3. **Слой интеграции (Message Brokers, Orchestrators)** — асинхронное взаимодействие между сервисами, очереди задач.

**Ключевые характеристики:**
- Работает **поверх HTTP/HTTPS, gRPC, WebSocket** (в зависимости от API).
- Обычно **stateless** (не хранит состояние между запросами), чтобы легко масштабироваться горизонтально.
- Использует **пулы соединений** к БД и другим ресурсам для производительности.

## 🧰 Технологии, используемые в Middle Layer

### 1. Языки и фреймворки (серверная часть)

| Язык | Популярные фреймворки | Где чаще используется |
|------|----------------------|----------------------|
| **JavaScript / TypeScript** | Express, NestJS, Fastify, Koa | Высокая скорость разработки, единый язык с фронтом |
| **Python** | Django, Flask, FastAPI | Data science, ML-интеграция, быстрые прототипы |
| **Go** | Gin, Echo, Fiber | Высокая производительность, микросервисы, сетевые утилиты |
| **Java** | Spring Boot, Quarkus, Micronaut | Крупные корпоративные системы, надёжность |
| **C# (.NET)** | ASP.NET Core | Enterprise, Windows-экосистема |
| **Ruby** | Ruby on Rails | Стартапы, быстрая разработка MVP |
| **Rust** | Actix, Rocket | Ультра-производительные сервисы, системное программирование |

### 2. Middleware (промежуточное ПО в конвейере)

Это функции, которые выполняются **до** или **после** основной логики обработчика. Примеры из Express.js:

```javascript
// Логирование
app.use((req, res, next) => {
  console.log(`${req.method} ${req.url}`);
  next(); // передаёт управление дальше
});

// Проверка аутентификации
app.use('/api/admin', (req, res, next) => {
  if (!req.user.isAdmin) return res.status(403).send('Forbidden');
  next();
});

// Сжатие ответов
app.use(compression());
```

Типичные middleware:
- **CORS** (разрешение кросс-доменных запросов)
- **Body parser** (разбор JSON, form-data)
- **Rate limiting** (ограничение частоты запросов)
- **Helmet** (защита HTTP-заголовков)
- **Session / cookie middleware**
- **Логирование, метрики** (Prometheus)

### 3. API Gateway и балансировщики

| Технология | Назначение |
|------------|------------|
| **Nginx** | Обратный прокси, балансировка, отдача статики, terminate TLS |
| **HAProxy** | Высокопроизводительная балансировка TCP/HTTP |
| **Kong** | API Gateway с плагинами (авторизация, трансформация) |
| **Traefik** | Современный gateway для контейнеров (Kubernetes, Docker Swarm) |
| **Envoy** | Прокси от Lyft, основа для сервис-мешей (Istio) |
| **AWS API Gateway** | Управляемый сервис от Amazon |

### 4. Асинхронная связь (очереди, брокеры)

| Технология | Применение |
|------------|------------|
| **RabbitMQ** (AMQP) | Классические очереди задач, обмен сообщениями |
| **Apache Kafka** | Высокопроизводительные потоки событий, логи |
| **Redis Pub/Sub** | Лёгкие очереди, реальное время |
| **AWS SQS / SNS** | Управляемые очереди в облаке |

### 5. Взаимодействие с данными

| Тип | Технологии |
|-----|------------|
| **Реляционные БД** | PostgreSQL, MySQL, SQL Server (через ORM: Prisma, TypeORM, SQLAlchemy, Hibernate) |
| **NoSQL** | MongoDB, Cassandra, Couchbase |
| **Кэши** | Redis, Memcached (для сессий, временных данных) |
| **Поиск** | Elasticsearch, Algolia, Meilisearch |

### 6. Контейнеризация и оркестрация (хотя это больше инфраструктура, но миддл обычно живёт в них)

- **Docker** — упаковка сервисов.
- **Kubernetes** — управление микросервисами, автоматическое масштабирование.
- **Docker Compose** — для локальной разработки.

## 🧪 Тестирование Middle Layer

Тестирование серверной логики отличается от тестирования фронтенда. Здесь не нужно эмулировать клик мыши, но нужно проверять API-эндпоинты, бизнес-правила, интеграции с БД и внешними сервисами.

### 1. Unit-тесты (проверка отдельных функций, сервисов)

- **Инструменты:** Jest (JS/TS), PyTest (Python), JUnit (Java), go test (Go)
- **Что тестировать:** вспомогательные утилиты, валидацию, вычисления, преобразование данных.
- **Пример (JS/Node.js):**
```javascript
test('calculateTotal with discount', () => {
  const cart = [{ price: 100 }, { price: 200 }];
  expect(calculateTotal(cart, 0.1)).toBe(270);
});
```

### 2. Интеграционные тесты (проверка связки компонентов)

- **Что тестировать:** вызов репозитория (с реальной тестовой БД), взаимодействие с очередью, кэшем.
- **Инструменты:** Testcontainers (поднимает Docker-контейнер с БД), Memory-базы (SQLite in-memory), моки для внешних API (nock, WireMock).
- **Пример:** тест, который создаёт пользователя в БД через репозиторий, затем проверяет, что он сохранился.

### 3. API-тесты (end-to-end для HTTP/gRPC)

- **Что тестировать:** публичные эндпоинты, коды ответов, валидацию входных данных, заголовки.
- **Инструменты:** Supertest (для Express), REST Client (в VS Code), Postman/Newman, Insomnia.
- **Пример (Supertest):**
```javascript
request(app)
  .post('/api/users')
  .send({ email: 'test@example.com' })
  .expect(201)
  .expect(res => res.body.id).toBeDefined();
```

### 4. Контрактные тесты (для микросервисов)

- **Что тестировать:** что сервис-потребитель и сервис-провайдер понимают друг друга (формат запросов/ответов).
- **Инструменты:** Pact (Pact.io), Spring Cloud Contract.
- **Применение:** когда у вас есть, например, сервис заказов, который вызывает сервис платежей.

### 5. Нагрузочное тестирование

- **Что тестировать:** сколько запросов выдерживает миддл, время ответа, использование CPU/памяти.
- **Инструменты:** k6 (современный, на JS), Apache JMeter, Yandex.Tank, wrk, Vegeta.

### 6. Тестирование безопасности

- **Что тестировать:** проверка авторизации, SQL-инъекции, XSS (хотя это больше фронт), rate limiting, утечка данных.
- **Инструменты:** ZAP (OWASP), Burp Suite, автоматические сканеры (Snyk, npm audit).

### 7. Тестирование middleware (промежуточных функций)

- **Что тестировать:** логирование, аутентификацию, CORS, rate limiter — изолированно от основного приложения.
- **Подход:** создавать фейковый `req` и `res` объекты, вызывать middleware и проверять вызовы `next()` или отправку ответа.

Допустим, интернет-магазин:

```
[Frontend (React)] 
       │
       │ HTTPS
       ▼
[API Gateway (Kong)] — проверяет API-ключ, rate limit
       │
       │ маршрутизация
       ├──────► [Сервис каталога (Node.js + Express)] ──► (PostgreSQL)
       │
       ├──────► [Сервис корзины (Go + Gin)] ──► (Redis)
       │
       └──────► [Сервис заказов (Python + FastAPI)] ──► (Kafka) ──► [Сервис уведомлений (Node.js)]
```

**Что здесь миддл:** все сервисы, API Gateway, брокер сообщений. Фронтенд и БД не входят в миддл (БД — это уровень хранения).

| **Frontend** | Миддл — это то, с чем фронтенд общается через API. |

## Резюме

- **Middle Layer** (миддл) — это серверная часть приложения, реализующая бизнес-логику и связывающая клиента с базами данных и другими сервисами.
- **Технологии:** языки (JS, Python, Go, Java), фреймворки (Express, Django, Spring), API Gateway (Nginx, Kong), очереди (RabbitMQ, Kafka), базы данных, кэши.
- **Тестирование:** unit, интеграционные, API-тесты, контрактные, нагрузочные, безопасности.
- **Ключевое отличие от фронтенда:** миддл работает на сервере, не имеет UI, управляет данными и доступом, требует тестирования под высокой нагрузкой.

Если хотите, можем углубиться в конкретный язык/фреймворк (например, NestJS или FastAPI), или в детали тестирования микросервисов, или в архитектуру API Gateway.

-------------------------------------------------------------------------------------------------------------------------------------------------------------

-------------------------------------------------------------------------------------------------------------------------------------------------------------

Отлично, вы переходите к **Backend** — логическому завершению предыдущих тем (миддл, инфраструктура, протоколы). Backend — это «вся серверная часть» приложения, которая не видна пользователю, но обеспечивает его данными, бизнес-логикой и безопасностью.

Если **Middle Layer** — это фокус на слое между фронтом и базами данных (часто синоним серверной логики), то **Backend** — более широкое понятие. Оно включает:

- **Сервер приложений** (ваша бизнес-логика, API, миддл).
- **Базы данных и кэши** (хранение данных).
- **Очереди и брокеры** (асинхронная обработка).
- **Инфраструктуру** (серверы, контейнеры, сети) — но инфраструктуру обычно выделяют отдельно.
- **Интеграции** (внешние API, платежные шлюзы, CDN).

Часто в вакансиях «Backend Developer» пишут на Java/Python/Go/Node.js, работают с базами данных, проектируют API, настраивают серверы.

Я дам ответ в привычной структуре: «под капотом», технологии, тестирование, связь с предыдущими темами.

## ⚙️ Что такое Backend «под капотом»

Бэкенд — это **набор программ и сервисов**, которые работают на сервере и обрабатывают запросы от клиентов (браузер, мобильное приложение, другое приложение). Пользователь его не видит, но без него интерфейс был бы просто картинкой без данных.

Типичный путь запроса в бэкенд:

1. Клиент отправляет HTTP-запрос на `https://api.example.com/users/123`.
2. **DNS** преобразует домен в IP сервера.
3. **Балансировщик** (или API Gateway) принимает запрос и направляет его на один из серверов приложений.
4. **Веб-сервер** (Nginx, Apache) может обработать статику или передать запрос дальше.
5. **Application Server** (ваш код на Node.js, Django, Spring) парсит запрос, выполняет бизнес-логику (проверка прав, валидация, вызов других сервисов).
6. При необходимости **ORM/драйвер** обращается к базе данных (PostgreSQL, Redis).
7. Ответ (JSON, HTML, XML) возвращается обратно по цепочке клиенту.

**Ключевые характеристики бэкенда:**
- Работает 24/7, обрабатывает множество одновременных запросов.
- Должен быть надёжным (не падать), безопасным (защита данных), масштабируемым (выдерживать рост нагрузки).
- Часто **stateless** (не хранит состояние между запросами) — состояние выносится в базы данных или кэши.
- Общается с внешним миром через **API** (REST, GraphQL, gRPC, WebSocket).

### 1. Языки и фреймворки (основные)

| Тип | Примеры | Применение |
|-----|---------|------------|
| **Реляционные (SQL)** | PostgreSQL, MySQL, SQL Server, SQLite | Транзакции, целостность, сложные запросы. Основной выбор для большинства приложений. |
| **Документоориентированные (NoSQL)** | MongoDB, Couchbase | Гибкая схема, масштабирование, большие объёмы. |

| **Графовые** | Neo4j, ArangoDB | Связи между сущностями (соцсети, рекомендации). |
| **Поисковые движки** | Elasticsearch, Typesense, Meilisearch | Полнотекстовый поиск, логи. |

### 3. Кэширование

- **Redis** — самый популярный in-memory кэш. Используют для: сессий, частых запросов (чтобы не ходить в БД), rate limiting, очередей (Pub/Sub, Streams).
- **Memcached** — простой, быстрый, но меньше функций, чем Redis.

### 4. Асинхронная обработка и очереди

| Технология | Применение |
|------------|------------|
| **RabbitMQ** (AMQP) | Надёжные очереди, маршрутизация, подтверждения доставки. |
| **Apache Kafka** | Потоки событий, логи, аналитика в реальном времени. |
| **Celery** (Python) | Асинхронные задачи с брокером (RabbitMQ/Redis). |
| **Bull** (Node.js) | Очереди на основе Redis. |
| **AWS SQS/SNS** | Управляемые очереди в облаке. |

### 5. API и коммуникации

- **REST** (HTTP/1.1, HTTP/2) — самый распространённый стиль для публичных API.
- **GraphQL** — клиент сам выбирает нужные поля, один endpoint.
- **gRPC** (HTTP/2, Protobuf) — высокая производительность, строгий контракт, стриминг.
- **WebSocket** — двусторонний обмен в реальном времени (чаты, игры).

### 6. Аутентификация и безопасность

- **JWT** (JSON Web Tokens) — stateless аутентификация.
- **OAuth 2.0 / OpenID Connect** — делегированный доступ (вход через Google, GitHub).
- **Session-based** (cookie + сессия в Redis).
- **Basic Auth**, API-ключи.
- **BCrypt, Argon2** — хеширование паролей.

### 7. Инфраструктурные инструменты (часть бэкенд-разработки)

- **Веб-серверы / Reverse proxy**: Nginx, Apache, Caddy.
- **Контейнеризация**: Docker, Podman.
- **Оркестрация**: Kubernetes (K8s), Docker Swarm.
- **CI/CD**: GitLab CI, GitHub Actions, Jenkins.
- **Мониторинг**: Prometheus + Grafana, Sentry (ошибки), ELK (логи).

## 🧪 Тестирование бэкенда

Тестирование бэкенда критически важно, так как ошибки могут привести к потере данных, уязвимостям или отказу сервиса. Подходы во многом пересекаются с тем, что я описал в разделе Middle Layer, но здесь добавлю специфику для БД и интеграций.

### 1. Unit-тесты (изолированные, быстрые)

- **Что:** тестирование отдельных функций, классов, методов.
- **Инструменты:** JUnit (Java), PyTest (Python), Jest (JS), go test (Go), PHPUnit (PHP).
- **Особенность:** мокайте (заглушайте) внешние зависимости — базы данных, API, файловую систему.
- **Пример (Python):**
```python
def test_calculate_discount():
    assert calculate_discount(100, 10) == 90
```

### 2. Интеграционные тесты (с реальными зависимостями)

- **Что:** проверка взаимодействия с БД, очередью, кэшем, внешним сервисом.
- **Подходы:**
  - Поднимать лёгкие тестовые контейнеры (Testcontainers).
  - Использовать in-memory базы (H2 для Java, SQLite для Python) — но они не всегда идентичны production.
  - Создавать отдельную тестовую БД и очищать её после каждого теста.
- **Инструменты:** Testcontainers, Docker Compose для тестов, библиотеки для работы с транзакциями.

### 3. API-тесты (черный ящик)

- **Что:** отправка реальных HTTP-запросов к работающему приложению и проверка ответов.
- **Инструменты:** Postman/Newman, Supertest (JS), REST Assured (Java), pytest + requests (Python).
- **Что проверять:** коды ответа (200, 400, 401, 404), структуру JSON, заголовки, время ответа.
- **Пример (Supertest):**
```javascript
request(app)
  .post('/api/auth/login')
  .send({ email: 'test@example.com', password: '123' })
  .expect(200)
  .expect(res => res.body.token).toBeDefined();
```

### 4. Контрактные тесты (для микросервисов)

- **Что:** проверка, что сервис-поставщик (например, сервис платежей) соответствует ожиданиям сервиса-потребителя (сервис заказов).
- **Инструменты:** Pact, Spring Cloud Contract.

### 5. Нагрузочное и производительное тестирование

- **Что:** выяснить, сколько запросов в секунду выдерживает бэкенд, при какой нагрузке начинает падать или сильно тормозить.
- **Инструменты:** k6, JMeter, Yandex.Tank, Vegeta, wrk.
- **Метрики:** RPS (requests per second), latency (p95, p99), CPU/memory usage.

### 6. Тестирование безопасности

- **Что:** проверка уязвимостей: инъекции (SQL, NoSQL), сломанная аутентификация, чувствительные данные в логах, превышение лимитов.
- **Инструменты:** OWASP ZAP, Burp Suite, автоматические анализаторы (Snyk, npm audit, Safety для Python).

### 7. Тестирование БД (миграции, запросы)

- **Что:** проверка, что миграции (изменения схемы БД) работают корректно, сложные SQL-запросы возвращают ожидаемые данные.
- **Инструменты:** библиотеки для тестирования репозиториев (например, `@testcontainers/postgres`), ручные тесты через psql.

### 8. Сквозные (E2E) тесты (часто команда бэкенда не пишет, но они существуют)

- **Что:** полный сценарий пользователя через интерфейс (фронтенд + бэкенд + БД). Это уже выходит за рамки бэкенда.

## 🧭 Backend vs Middle Layer vs Frontend

| Понятие | Что включает | Пример |
|---------|-------------|--------|
| **Frontend** | Клиентская часть (браузер, мобильное приложение) | React, Vue, Swift, Kotlin |
| **Backend** | Серверная часть целиком: бизнес-логика + API + базы данных + очереди + интеграции | Java + Spring + PostgreSQL + Redis + RabbitMQ |
| **Middle Layer** | Часть бэкенда, отвечающая за приём запросов, бизнес-логику, вызовы к БД и другим сервисам | Node.js + Express, Go + Gin |

Таким образом, **Middle Layer — это подмножество Backend**. Термин «миддл» часто используют в контексте архитектуры «фронт → миддл → БД/интеграции», чтобы подчеркнуть, что бэкенд может быть многослойным (например, есть слой API Gateway и слой сервисов).

---

## 🔗 Связь с предыдущими темами

| Ранее изученное | Как относится к бэкенду |
|----------------|------------------------|
| **API, сервисы, микросервисы** | Бэкенд их реализует. REST API, gRPC — интерфейсы бэкенда. |
| **Протоколы (HTTP, TCP, TLS)** | Бэкенд работает поверх этих протоколов. |
| **Модели OSI/TCP/IP** | Бэкенд находится на прикладном уровне, использует транспортный и нижележащие. |
| **Инфраструктура** | Бэкенд запускается на серверах, контейнерах, использует сети, балансировщики. |
| **Frontend** | Бэкенд — это то, с чем общается фронтенд через API. |
| **Middle Layer** | Это часть бэкенда, часто синоним «сервер приложений». |

## 📦 Пример простого бэкенда на Node.js (Express) с PostgreSQL

```javascript
// app.js
const express = require('express');
const { Pool } = require('pg');

const app = express();
const pool = new Pool({ connectionString: process.env.DATABASE_URL });

app.use(express.json());

// GET /users/:id
app.get('/users/:id', async (req, res) => {
  const { id } = req.params;
  try {
    const result = await pool.query('SELECT id, name, email FROM users WHERE id = $1', [id]);
    if (result.rows.length === 0) return res.status(404).json({ error: 'User not found' });
    res.json(result.rows[0]);
  } catch (err) {
    res.status(500).json({ error: 'Database error' });
  }
});

// POST /users
app.post('/users', async (req, res) => {
  const { name, email } = req.body;
  if (!name || !email) return res.status(400).json({ error: 'Missing fields' });
  try {
    const result = await pool.query(
      'INSERT INTO users (name, email) VALUES ($1, $2) RETURNING id',
      [name, email]
    );
    res.status(201).json({ id: result.rows[0].id });
  } catch (err) {
    res.status(500).json({ error: 'Database error' });
  }
});

app.listen(3000, () => console.log('Backend running on port 3000'));
```

Это минимальный, но реальный бэкенд. Он имеет API, валидацию, общается с БД, обрабатывает ошибки.

## Резюме

- **Backend** — вся серверная часть приложения: бизнес-логика, API, базы данных, очереди, кэши, интеграции. Это то, что работает на сервере и не видно пользователю.
- **Технологий много:** языки (JavaScript, Python, Go, Java, C#, Ruby, PHP, Rust), базы данных (PostgreSQL, Redis, MongoDB), очереди (RabbitMQ, Kafka), фреймворки (Express, Django, Spring).
- **Тестирование** включает unit, интеграционные, API, контрактные, нагрузочные, безопасности.
- **Backend ≠ Middle Layer**, но middle layer — важная часть бэкенда, отвечающая за обработку запросов.

Если хотите, можем углубиться в конкретную технологию (например, как устроены транзакции в PostgreSQL, как работает асинхронность в Python FastAPI, или как писать тесты для бэкенда с использованием Testcontainers).
