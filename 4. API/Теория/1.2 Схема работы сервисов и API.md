Отлично, давайте нарисуем **схему работы** сервисов через API. Сначала простой вариант, потом более реальный — с микросервисами.

## 1. Базовая схема «Клиент ↔ Сервис через API»

```
[ Клиент (фронтенд, приложение, другой сервис) ]
          │
          │  HTTP-запрос (GET /user/123)
          │  ↓
      ┌───┴───┐
      │  API  │  <- "Привратник" сервиса
      └───┬───┘
          │  вызывает внутреннюю логику
          ↓
   [ Сервис (программа) ]
          │
          │  возвращает результат (JSON, XML)
          │  ↑
      ┌───┴───┐
      │  API  │
      └───┬───┘
          │  HTTP-ответ (200 OK, данные)
          ↓
[ Клиент получает ответ ]
```

**Как это работает:**
- **API** — это не отдельная программа, а слой (шлюз) самого сервиса. Он принимает запросы, проверяет их, направляет внутрь.
- Сервис выполняет работу (достаёт из БД, считает) и отдаёт результат через тот же API.
- Клиенту всё равно, как сервис устроен внутри — он знает только API (URL, методы, параметры).

## 2. Схема с несколькими микросервисами и API Gateway

В реальных системах много независимых сервисов. Чтобы клиент не запутался, ставят **API Gateway** — единую точку входа.

```
[ Мобильное приложение / Браузер ]
              │
              │ (один URL, например api.shop.com)
              ↓
        ┌─────────────┐
        │ API Gateway │  <- тоже имеет API (публичный)
        └──────┬──────┘
               │
      ┌────────┼─────────────┐
      │        │             │
      ↓        ↓             ↓
┌─────────┐ ┌─────────┐ ┌─────────┐
│Сервис   │ │Сервис   │ │Сервис   │
│пользова-│ │корзины  │ │платежей │
│телей    │ │         │ │         │
└────┬────┘ └────┬────┘ └────┬────┘
     │           │           │
     │    ┌──────┼──────┐    │
     │    │      │      │    │
     ↓    ↓      ↓      ↓    ↓
  [База] [База] [Очередь] [База]
```

**Что происходит при запросе «оформить заказ»:**
1. Клиент → API Gateway: `POST /order`.
2. Gateway спрашивает **Сервис пользователей** (через его API): есть ли такой клиент?
3. Gateway → **Сервис корзины** (через его API): возьми товары из корзины №123.
4. Gateway → **Сервис платежей** (через его API): спиши деньги.
5. Все ответы собираются, Gateway отдаёт клиенту итог.

## 3. Взаимодействие сервисов между собой (без клиента)

Сервисы тоже общаются друг с другом через API. Например, микросервис «Отправка email» предоставляет API:

```
[Сервис регистрации]  ---(HTTP POST /send-email)--->  [Сервис email]
         │                                                    │
         │                                                    │
         └──────────(JSON {"status": "ok"})<──────────────────┘
```

Или асинхронно через очередь (брокер сообщений):

```
[Сервис заказов] → (кладёт сообщение) → [Очередь RabbitMQ] → [Сервис доставки]
                                          (забирает сообщение)
```

## 4. Сводная схема с уровнями

Если представить стек:

```
┌──────────────────────────────────────────┐
│            Клиенты (UI, CLI, скрипты)     │
└─────────────────────┬────────────────────┘
                      │  HTTP/HTTPS, gRPC, GraphQL
┌─────────────────────▼────────────────────┐
│         API Gateway (публичное API)       │
└─────────────────────┬────────────────────┘
                      │  внутренние вызовы через API
┌─────────────────────▼────────────────────┐
│  Микросервис A  ◄──►  Микросервис B       │
│    (есть свой API)    (есть свой API)     │
└─────────────────────┬────────────────────┘
                      │  API к инфраструктуре
┌─────────────────────▼────────────────────┐
│      Базы данных, кэши, очереди           │
└──────────────────────────────────────────┘
```

## Ключевые принципы работы (по пунктам)

1. **Каждый сервис (и микросервис) предоставляет API** — набор чётко описанных операций.
2. **API работает по контракту**: клиент отправляет данные в определённом формате, сервис отвечает строго по спецификации.
3. **Сервисы не вторгаются в чужие внутренности** — только через API. Это даёт слабую связанность.
4. **Веб-сервис** — частный случай: его API работает через HTTP (REST, SOAP, GraphQL).
5. **Схема работы** всегда одна: «Запрос → API сервиса → обработка → ответ через API».

## Для запоминания в виде цепочки:

> **Клиент** → (вызывает) → **API** → (активирует) → **Сервис** → (может вызвать) → **API другого сервиса** → и т.д.

Если нарисовать это стрелками:  
`Клиент --> API_1 --> Сервис_1 --> API_2 --> Сервис_2 --> ...`

Итог: **API — это дверь, сервис — комната. Схема работы — как люди ходят из комнаты в комнату через двери.**
