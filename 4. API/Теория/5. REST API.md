**REST API** — это архитектурный стиль для построения API, который использует HTTP-протокол и его методы для управления ресурсами. REST (Representational State Transfer) был предложен Роем Филдингом в 2000 году и стал основой для большинства современных веб-API.

В этом ответе — полное объяснение: принципы, правила проектирования, примеры, сравнение с другими подходами, лучшие практики.

---

## 🧱 Основные принципы REST (6 ограничений)

Чтобы API называлось RESTful, оно должно следовать шести архитектурным ограничениям:

| Принцип | Суть | Пример |
|---------|------|--------|
| **Client-Server** | Разделение клиента и сервера. Они эволюционируют независимо. | Фронтенд на React не знает, как устроен бэкенд на Django. |
| **Stateless** | Сервер не хранит состояние клиента между запросами. Каждый запрос содержит всю необходимую информацию. | В запросе передаётся токен аутентификации, а не полагается на сессию на сервере. |
| **Cache** | Ответы сервера должны явно указывать, можно ли их кешировать (заголовки `Cache-Control`). | `GET /products` можно закешировать на 1 минуту. |
| **Uniform Interface** | Единообразие интерфейса: идентификация ресурсов (URI), представление ресурсов (JSON/XML), самодостаточные сообщения (HATEOAS), гипермедиа. | Все ресурсы имеют понятные URI, одинаковые методы, статусы. |
| **Layered System** | Клиент не знает, напрямую ли он общается с сервером или через прокси, балансировщик, кеш. | Прозрачное добавление CDN или API Gateway. |
| **Code on Demand (опционально)** | Сервер может передавать клиенту исполняемый код (например, JavaScript). | Редко используется в API. |

**Ключевой вывод:** REST — это не протокол, а набор правил. На практике REST API обычно строится поверх HTTP.

## 📐 RESTful-проектирование ресурсов

В REST всё — **ресурс**. Ресурс — это любой объект, данные или сервис, на который можно сослаться. Каждый ресурс идентифицируется **URI** (унифицированным идентификатором ресурса).

### Правила именования URI

| Правило | Хорошо | Плохо |
|---------|--------|-------|
| Используйте существительные, не глаголы | `/users`, `/orders` | `/getUser`, `/createOrder` |
| Используйте множественное число для коллекций | `/users` (список), `/users/123` (один) | `/user/123` |
| Вложенность для отношений | `/users/123/orders` | `/orders?userId=123` (тоже допустимо, но менее выразительно) |
| Избегайте глубокой вложенности (>3) | `/users/123/orders/456/items` — уже перебор | Лучше `/items/789` |
| Используйте kebab-case или snake_case (по договорённости) | `/user-profiles`, `/user_profiles` | `/userProfiles` (camelCase не стандарт для URL) |

### HTTP методы и их смысл

| Метод | Операция | Идемпотентность | Безопасность | Пример |
|-------|----------|----------------|--------------|--------|
| `GET` | Получение ресурса | ✅ | ✅ | `GET /users/123` |
| `POST` | Создание нового ресурса или выполнение действия | ❌ | ❌ | `POST /users` |
| `PUT` | Полная замена ресурса (или создание, если известен ID) | ✅ | ❌ | `PUT /users/123` |
| `PATCH` | Частичное обновление | ❌ | ❌ | `PATCH /users/123` |
| `DELETE` | Удаление | ✅ | ❌ | `DELETE /users/123` |

> **Идемпотентность**: повторный вызов с теми же параметрами не меняет состояние (результат тот же).  
> **Безопасность**: метод не изменяет состояние сервера.

### HTTP статус-коды (самые частые)

| Код | Значение | Когда использовать |
|-----|----------|---------------------|
| 200 OK | Успех | GET, PUT, PATCH, DELETE (с телом ответа) |
| 201 Created | Ресурс создан | POST (в ответе может быть Location: /users/123) |
| 204 No Content | Успех, тело ответа пусто | DELETE, PUT/PATCH без возврата данных |
| 400 Bad Request | Неверный запрос (валидация) | Отсутствует обязательное поле, неверный формат |
| 401 Unauthorized | Нет аутентификации | Отсутствует или неверный токен |
| 403 Forbidden | Нет прав | Аутентификация есть, но недостаточно прав |
| 404 Not Found | Ресурс не найден | GET /users/999 |
| 422 Unprocessable Entity | Семантическая ошибка | email уже занят |
| 429 Too Many Requests | Превышен лимит | Rate limiting |
| 500 Internal Server Error | Ошибка сервера | Необработанное исключение |

## 📦 Пример REST API для интернет-магазина

### Ресурсы: `products`, `cart`, `orders`, `users`

| Эндпоинт | Метод | Описание |
|----------|-------|----------|
| `/products` | GET | Список товаров (с фильтрацией, пагинацией) |
| `/products/{id}` | GET | Детали товара |
| `/cart` | GET | Содержимое корзины (требует auth) |
| `/cart/items` | POST | Добавить товар в корзину |
| `/cart/items/{id}` | DELETE | Удалить товар из корзины |
| `/orders` | POST | Оформить заказ |
| `/orders/{id}` | GET | Детали заказа |
| `/users` | POST | Регистрация |
| `/users/{id}` | GET | Профиль пользователя |

**Пример запроса на создание пользователя:**
```http
POST /users HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "name": "Alice",
  "email": "alice@example.com",
  "password": "secret"
}
```

**Ответ:**
```http
HTTP/1.1 201 Created
Location: /users/123
Content-Type: application/json

{
  "id": 123,
  "name": "Alice",
  "email": "alice@example.com",
  "created_at": "2025-01-15T10:30:00Z"
}
```

## 🔧 Параметры запроса: пагинация, фильтрация, сортировка

| Тип | Пример | Пояснение |
|-----|--------|-----------|
| Пагинация | `?limit=20&offset=40` | Классическая (offset) |
| Пагинация курсором | `?limit=20&cursor=abc123` | Более производительная для больших данных |

**Пример:**  
`GET /products?category=laptops&min_price=500&max_price=1500&sort=-rating&limit=10&offset=20`

## 🧪 HATEOAS (Hypermedia as the Engine of Application State)

Это самый спорный и редко реализуемый принцип REST. Он означает, что сервер должен возвращать клиенту ссылки на возможные действия, чтобы клиент мог навигировать по API без жёстко зашитых URL.

**Пример ответа с HATEOAS:**
```json
{
  "id": 123,
  "name": "Alice",
  "links": [
    { "rel": "self", "href": "/users/123" },
    { "rel": "orders", "href": "/users/123/orders" },
    { "rel": "cart", "href": "/users/123/cart" }
  ]
}
```

На практике большинство «REST API» не реализуют HATEOAS полностью, и их корректнее называть **HTTP API**, но в индустрии их всё равно называют RESTful.

## 🆚 REST против других стилей

| Характеристика | REST | GraphQL | gRPC |
|----------------|------|---------|------|
| Протокол | HTTP/1.1, HTTP/2 | HTTP/1.1, HTTP/2 | HTTP/2 |
| Формат данных | JSON, XML, HTML, другие | JSON (запрос), JSON (ответ) | Protobuf (бинарный) |
| Один эндпоинт? | Нет (много URL) | Да (обычно /graphql) | Да (сервис с методами) |
| Выбор полей | Нет (возвращается всё) | Да (клиент выбирает) | Нет (фиксированная схема) |
| Кеширование | Просто (HTTP cache) | Сложно (нужны инструменты) | Сложно |
| Генерация кода | Нет (OpenAPI — опционально) | Да (интроспекция) | Да (из .proto) |
| Сложность | Низкая | Средняя | Высокая (из-за Protobuf) |

**Когда выбирать REST:**
- Простые CRUD-приложения.
- Публичные API с широким кругом клиентов.
- Нужно лёгкое кеширование через CDN.
- Команда не хочет усложнять.

## 🛡️ Безопасность REST API

| Мера | Реализация |
|------|------------|
| **HTTPS (TLS)** | Обязательно, иначе всё в открытом виде |

| **Защита от инъекций** | Не доверять входным данным, параметризовать запросы к БД |
| **CORS** | Настроить разрешённые домены для браузерных клиентов |

## 📄 Документирование REST API

Стандарт де-факто — **OpenAPI (Swagger)**. Описание в формате YAML/JSON позволяет:
- Генерировать интерактивную документацию (Swagger UI, ReDoc).
- Генерировать клиентские SDK на многих языках.
- Валидировать запросы/ответы.
- Автоматически тестировать контракт.

**Пример минимальной OpenAPI спецификации:**
```yaml
openapi: 3.0.0
info:
  title: User API
  version: 1.0.0
paths:
  /users:
    get:
      summary: List users
      responses:
        '200':
          description: OK
```

Многие фреймворки генерируют OpenAPI автоматически из кода (FastAPI, SpringDoc, Swashbuckle для .NET).

## 🚀 Лучшие практики REST API

1. **Используйте существительные, не глаголы** (кроме редких действий, не вписывающихся в CRUD).
2. **Версионируйте API** — лучше в URL: `/v1/users`, `/v2/users`.
3. **Возвращайте полезные сообщения об ошибках** (код, понятное сообщение, ссылка на документацию).
4. **Используйте стандартные HTTP статусы**, не придумывайте свои.
5. **Поддерживайте пагинацию, фильтрацию, сортировку** через query параметры.
6. **Не возвращайте stack trace клиенту** — только в логи.
7. **Обеспечьте идемпотентность** для PUT, DELETE (и для POST через `Idempotency-Key`).
8. **Документируйте всё** — OpenAPI или как минимум человекочитаемая документация.
9. **Применяйте rate limiting** и возвращайте заголовки `X-RateLimit-Remaining`, `Retry-After`.
10. **Используйте Content-Type: application/json** (по умолчанию), но поддерживайте и другие форматы при необходимости.

Вы просите сфокусироваться именно на **шести принципах (ограничениях) REST**, которые Рой Филдинг описал в своей диссертации в 2000 году. Именно соблюдение этих правил отличает настоящий RESTful API от обычного HTTP API.

Ниже — каждый принцип с объяснением, зачем он нужен и как его нарушают.

---

## 1. Client-Server (Разделение клиента и сервера)

**Суть:**  
Клиент и сервер — независимые сущности. Клиент не знает, как устроен сервер (БД, бизнес-логика), а сервер не знает об интерфейсе клиента (UI, состояние сессии). Они взаимодействуют только через унифицированный интерфейс (HTTP).

**Зачем:**  
- Позволяет эволюционировать клиенту и серверу независимо.  
- Клиент можно переписать с React на Vue, не меняя сервер.  
- Сервер можно масштабировать горизонтально, не меняя клиент.

**Нарушение:**  
Сервер генерирует HTML для конкретного браузера, вшивает логику представления.  
Клиент знает структуру таблиц БД сервера.

---

## 2. Stateless (Отсутствие состояния на сервере)

**Суть:**  
Сервер **не хранит состояние клиента** между запросами. Каждый запрос от клиента должен содержать **всю необходимую информацию** для его обработки (аутентификация, контекст, параметры). Состояние (корзина, сессия) хранится на клиенте или в отдельном хранилище (БД, кэш).

**Зачем:**  
- Простое масштабирование: любой сервер может обработать любой запрос.  
- Устойчивость: сервер может упасть, но следующий запрос пойдёт к другому серверу.  
- Прозрачность: не нужно синхронизировать состояние между серверами.

**Нарушение:**  
- Использование `HttpSession` на сервере (хранит корзину в памяти).  
- Запрос `DELETE /cart` не передаёт `user_id`, а полагается на сохранённую сессию.

**Правильная реализация:**  
Каждый запрос содержит токен авторизации (JWT), а корзина хранится в Redis или передаётся в теле запроса.

---

## 3. Cache (Кеширование)

**Суть:**  
Ответы сервера должны явно объявлять, можно ли их кешировать (заголовки `Cache-Control`, `Expires`, `ETag`). Клиент или промежуточный кеш (CDN, прокси) может использовать эти ответы повторно без обращения к серверу.

**Зачем:**  
- Улучшает производительность и снижает нагрузку на сервер.  
- Уменьшает задержки для клиента.

**Нарушение:**  
- Сервер всегда возвращает `Cache-Control: no-cache` или не возвращает заголовков.  
- Динамические данные (например, `/users/123`) кешируются одинаково для всех пользователей — утечка данных.

**Правильная реализация:**  
`Cache-Control: max-age=60` для общедоступных ресурсов, `ETag` для условных запросов.  
Для приватных данных `Cache-Control: private`.

---

## 4. Uniform Interface (Единообразие интерфейса)

**Суть:**  
Это самый важный и самый сложный принцип. Он требует **единообразия** во взаимодействии между компонентами. Включает 4 подтребования:

1. **Идентификация ресурсов** — каждый ресурс идентифицируется URI (например, `/users/123`).
2. **Представление ресурсов** — ресурс отделён от его представления (JSON, XML, HTML). Клиент через `Accept` может запросить нужный формат.
3. **Самодостаточные сообщения** — сообщение содержит всю информацию, как его обработать (метод, заголовки, тело).
4. **HATEOAS** (Hypermedia as the Engine of Application State) — сервер возвращает ссылки на возможные действия, чтобы клиент не «зашивал» URL.

**Зачем:**  
- Упрощает архитектуру и делает её предсказуемой.  
- Позволяет серверу изменять URL, не ломая клиента (благодаря HATEOAS).

**Нарушение:**  
- Использование глаголов в URL: `/getUser`, `/createOrder`.  
- Возврат XML по умолчанию, а клиент вынужден парсить без указания формата.  
- Отсутствие HATEOAS — клиент конкатенирует строки: `BASE_URL + "/users/" + userId`. Это **не RESTful**, но допустимо на практике.

**Пример HATEOAS (реализуется редко):**
```json
{
  "id": 123,
  "name": "Alice",
  "links": [
    { "rel": "self", "href": "/users/123" },
    { "rel": "orders", "href": "/users/123/orders" }
  ]
}
```

> **Важно:** Большинство популярных «REST API» не реализуют HATEOAS, поэтому их корректнее называть **HTTP API**. Но в индустрии их всё равно называют RESTful.

---

## 5. Layered System (Слои)

**Суть:**  
Клиент не должен знать, напрямую он общается с сервером или через промежуточные слои (балансировщики, прокси, шлюзы, кеши). Каждый слой видит только следующий.

**Зачем:**  
- Позволяет добавлять уровни безопасности, кеширования, балансировки без изменения клиента.  
- Упрощает эволюцию архитектуры.

**Нарушение:**  
Клиент жёстко привязан к IP-адресу сервера и не может работать через CDN.  
API Gateway требует специальных заголовков, о которых клиент должен знать.

**Правильная реализация:**  
Клиент обращается к `https://api.example.com`, а за этим доменом может стоять балансировщик → кеш → несколько серверов. Клиент ничего не знает.

---

## 6. Code on Demand (опциональный)

**Суть:**  
Сервер может временно расширить функциональность клиента, передав ему исполняемый код (например, JavaScript, Java-апплеты). Это единственный **необязательный** принцип.

**Зачем:**  
- Позволяет уменьшить количество запросов, перенеся логику на клиента.  
- Позволяет динамически менять поведение клиента.

**Нарушение (не считается нарушением, т.к. опционально):**  
Сервер никогда не передаёт код — это нормально.

**Пример (редкий):**  
API возвращает скрипт, который клиент выполняет для рендеринга формы.

## Сравнительная таблица принципов

| Принцип | Обязателен? | Суть | Частота нарушения |
|---------|-------------|------|-------------------|
| Client-Server | ✅ | Разделение клиента и сервера | Редко |
| Stateless | ✅ | Сервер не хранит сессию | Часто (сессии на сервере) |
| Cache | ✅ | Явное указание кешируемости | Часто (нет заголовков) |
| Uniform Interface | ✅ | URI, представления, HATEOAS | Очень часто (нет HATEOAS) |
| Layered System | ✅ | Прозрачность слоёв | Редко |
| Code on Demand | ❌ | Передача кода клиенту | Почти всегда отсутствует |

## Как проверить, что ваш API соблюдает принципы?

1. **Client-Server** — можете ли вы заменить фронтенд (например, с React на мобильное приложение) без изменений в бэкенде?  
2. **Stateless** — если вы убьёте один сервер, запросы продолжат обрабатываться другими?  
3. **Cache** — есть ли заголовки `Cache-Control` для публичных ресурсов?  
4. **Uniform Interface** — используете ли вы глаголы в URL? возвращаете ли ссылки HATEOAS?  
5. **Layered System** — можете ли вы добавить CDN или балансировщик, не меняя клиента?  

> **Вывод:** На практике многие API называют «REST», но соблюдают только 3–4 принципа (чаще всего не реализуют HATEOAS и Stateless). Это не страшно, если вы понимаете компромиссы. Но если вам нужен строгий REST (например, для публичного API с долгой поддержкой), старайтесь следовать всем шести.

HTTP-запрос — это сообщение, которое клиент (браузер, мобильное приложение, другой сервер) отправляет серверу, чтобы попросить выполнить какое-то действие (получить страницу, отправить форму, удалить данные). Каждый HTTP-запрос имеет строгую структуру, определённую спецификацией.

Ниже — полное описание структуры HTTP-запроса с примерами.

---

## 📦 Структура HTTP-запроса

HTTP-запрос состоит из **трёх частей** (последняя может отсутствовать):

```
[Стартовая строка (Request Line)]
[Заголовки (Headers)]
[Пустая строка]
[Тело (Body)] — опционально
```

### 1. Стартовая строка (Request Line)

Содержит три элемента, разделённые пробелом:

```
<МЕТОД> <URI> <ВЕРСИЯ_HTTP>
```

- **МЕТОД** — глагол, указывающий действие (GET, POST, PUT, DELETE, PATCH и др.).
- **URI** — путь к ресурсу (например, `/users/123`). Может включать query-параметры: `/users?page=2&limit=10`.
- **ВЕРСИЯ_HTTP** — `HTTP/1.1`, `HTTP/2`, `HTTP/3`.

**Пример:**  
`GET /products?category=books HTTP/1.1`

---

### 2. Заголовки (Headers)

Набор строк вида `Ключ: Значение`. Они передают метаинформацию о запросе: формат данных, авторизацию, ожидаемый язык, настройки кеширования и т.д.

Заголовки заканчиваются **пустой строкой** (CRLF), которая отделяет их от тела.

**Примеры распространённых заголовков:**

| Заголовок | Пример значения | Назначение |
|-----------|----------------|-------------|
| `Host` | `api.example.com` | Обязателен в HTTP/1.1; указывает имя сервера |
| `User-Agent` | `Mozilla/5.0 ...` | Идентифицирует клиентское приложение |
| `Accept` | `application/json` | Какие форматы ответа понимает клиент |
| `Content-Type` | `application/json` | Формат тела запроса (для POST/PUT) |
| `Content-Length` | `348` | Длина тела в байтах |
| `Authorization` | `Bearer eyJhbGci...` | Токен для аутентификации |
| `Cookie` | `sessionId=abc123` | Передача кук |
| `Cache-Control` | `no-cache` | Управление кешированием |

**Пример набора заголовков:**
```
Host: api.example.com
User-Agent: curl/7.68.0
Accept: application/json
Content-Type: application/json
Content-Length: 45
Authorization: Bearer xyz123
```

---

### 3. Пустая строка

Обязательный разделитель между заголовками и телом. Это просто перевод строки (`\r\n`).

---

### 4. Тело запроса (Body)

Содержит данные, отправляемые серверу. Присутствует только у методов, которые могут передавать данные (обычно POST, PUT, PATCH). Для GET и DELETE тело обычно отсутствует (но теоретически может быть, хотя не рекомендуется).

Формат тела указывается в заголовке `Content-Type`.

**Популярные форматы тела:**

| Content-Type | Пример данных |
|--------------|---------------|
| `application/json` | `{"name":"Alice","email":"alice@example.com"}` |
| `application/x-www-form-urlencoded` | `name=Alice&email=alice%40example.com` |
| `multipart/form-data` | (для загрузки файлов) |
| `text/plain` | `Простой текст` |
| `application/xml` | `<user><name>Alice</name></user>` |

---

## 📝 Полный пример HTTP/1.1 запроса

### Пример: отправка формы регистрации (POST)

```
POST /v1/users HTTP/1.1
Host: api.example.com
User-Agent: Mozilla/5.0
Content-Type: application/json
Content-Length: 58
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

{"name":"Alice","email":"alice@example.com","password":"secret"}
```

**Разбор:**
- **Стартовая строка:** `POST /v1/users HTTP/1.1`
- **Заголовки:** Host, User-Agent, Content-Type, Content-Length, Authorization
- **Пустая строка** (после заголовков)
- **Тело:** JSON с данными пользователя

## 🧪 Пример GET-запроса (без тела)

```
GET /products?category=books&limit=10 HTTP/1.1
Host: api.example.com
Accept: application/json
```

Здесь нет пустой строки и тела, потому что после заголовков сразу идёт конец сообщения (но формально пустая строка всё равно есть, просто после неё ничего нет).

## 🧠 Нюансы для разных версий HTTP

- **HTTP/1.0** — каждый запрос открывает новое TCP-соединение.
- **HTTP/1.1** — поддерживается Keep-Alive (переиспользование соединения). Обязателен заголовок `Host`.
- **HTTP/2** — бинарный протокол, но семантика (методы, статусы, заголовки) остаётся той же. Заголовки сжимаются (HPACK).
- **HTTP/3** — поверх UDP (QUIC), структура запроса концептуально та же.

## 🔧 Как это связано с REST API и принципами REST

- **Методы** REST (GET, POST, PUT, DELETE) — это те самые методы в стартовой строке.
- **URI** идентифицирует ресурс (принцип Uniform Interface).
- **Stateless** означает, что каждый запрос должен содержать все необходимые заголовки (например, `Authorization`), а сервер не хранит контекст между запросами.
- **Заголовки `Cache-Control`** реализуют принцип кеширования.

**HTTP-методы** (глаголы) определяют тип операции, которую клиент хочет выполнить над ресурсом. Два важных свойства методов — **безопасность** и **идемпотентность**.

- **Безопасный (safe)** — метод не изменяет состояние сервера. Его можно вызывать без риска побочных эффектов. Только чтение.
- **Идемпотентный (idempotent)** — повторный вызов метода с теми же параметрами не изменяет состояние сервера (результат такой же, как после первого вызова). Важно для надёжности при сбоях сети.

|-------|------------|---------------|-----------|----------|--------|
| **GET** | ✅ да | ✅ да | ❌ нет | Получение ресурса или коллекции | `GET /users/123` |
| **HEAD** | ✅ да | ✅ да | ❌ нет | Получение только заголовков, как GET без тела | `HEAD /users/123` |

## 🔍 Детальный разбор идемпотентности

### Что такое идемпотентность простыми словами?

Вызов метода один раз и десять раз подряд с одинаковыми данными должен приводить к **одинаковому состоянию сервера** (как будто его вызвали один раз). При этом **ответ** может отличаться (например, первый раз 201 Created, последующие 200 OK или 204 No Content), но состояние не меняется.

### Почему идемпотентность важна?

В сетях бывают сбои: клиент отправил запрос, сервер обработал, но ответ потерялся. Клиент не знает, дошёл ли запрос. Если метод идемпотентен, клиент может безопасно повторить запрос, не боясь создать дубликат (например, дважды списать деньги).

---

## 🧪 Примеры идемпотентности для каждого метода

### GET, HEAD, OPTIONS, TRACE
Естественно идемпотентны, так как ничего не меняют.

### PUT
**Идемпотентен.** Замена ресурса целиком. Повторный PUT с тем же телом заменит ресурс на то же самое состояние.

- Первый `PUT /users/123 {"name":"Alice"}` создаст или обновит пользователя.
- Второй `PUT /users/123 {"name":"Alice"}` просто установит то же имя (состояние не изменилось).

### DELETE
**Идемпотентен.** Удаление ресурса. После первого удаления ресурса нет. Повторный DELETE уже ничего не удаляет, состояние сервера то же (ресурс отсутствует). HTTP-код ответа может быть 200, 204 или 404, но это не нарушает идемпотентность (состояние одинаковое).

### POST
**Не идемпотентен.** Каждый POST обычно создаёт новый ресурс с новым ID. Два одинаковых POST-запроса создадут два разных ресурса (например, два заказа, два пользователя).  
**Исключение:** Если POST используется для выполнения идемпотентного действия (например, активация аккаунта), то его можно сделать идемпотентным через заголовок `Idempotency-Key` (клиент генерирует уникальный ключ, сервер запоминает его и повторные вызовы с тем же ключом не создают новый ресурс). Это распространённая практика в платёжных API (Stripe, PayPal).

### PATCH
**Не гарантированно идемпотентен.** Всё зависит от реализации. Например, `PATCH /users/123 {"op":"increment","field":"visits"}` — каждый вызов увеличит счётчик, не идемпотентно. Если же PATCH устанавливает абсолютное значение (`{"name":"Alice"}`), то его можно сделать идемпотентным. Спецификация HTTP не требует идемпотентности от PATCH, но разработчик может её обеспечить.

---

## 🛡️ Безопасные методы (safe)

Методы, которые не изменяют состояние сервера: **GET, HEAD, OPTIONS, TRACE**. Их могут кешировать, предзагружать (prefetch), использовать поисковые роботы без опасения. Однако безопасность не означает, что ответ не может быть разным (например, счётчик просмотров может увеличиваться, но это не изменение состояния по запросу клиента, а внутренняя логика). По спецификации, безопасные методы не должны иметь побочных эффектов, но на практике они могут логировать или менять статистику — это допустимо.

## 🧪 Практический пример: повторная отправка POST

**Проблема:** Клиент отправляет `POST /orders` для оформления заказа. Ответ потерялся. Клиент повторяет запрос, и создаётся дубликат заказа.

**Решение:** Использовать **Idempotency-Key**.
1. Клиент генерирует уникальный ключ (например, UUID) и отправляет его в заголовке `Idempotency-Key: 123e4567-e89b-12d3-a456-426614174000`.
2. Сервер проверяет, не обрабатывал ли он этот ключ. Если нет — выполняет операцию и сохраняет результат (например, ID заказа) в хранилище ключей.
3. Если клиент повторяет запрос с тем же ключом, сервер сразу возвращает сохранённый результат, не создавая новый заказ.

Таким образом, даже неидемпотентный `POST` становится идемпотентным на уровне приложения.

**Коды ответа HTTP** — это трёхзначные числа, которые сервер возвращает клиенту, чтобы сообщить результат обработки запроса. Они являются ключевой частью протокола HTTP и позволяют клиенту понять, успешно ли выполнился запрос, нужно ли повторить его, изменить поведение или обработать ошибку.

Все коды делятся на пять классов (по первой цифре):

| Диапазон | Класс | Смысл |
|----------|-------|-------|
| 1xx | **Informational** | Информационные, запрос принят, обработка продолжается |
| 2xx | **Success** | Успех, запрос понят и принят |
| 3xx | **Redirection** | Перенаправление, клиенту нужно выполнить другое действие |
| 4xx | **Client Error** | Ошибка на стороне клиента (неверный запрос, отсутствие прав и т.п.) |
| 5xx | **Server Error** | Ошибка на стороне сервера (сервер не смог выполнить корректный запрос) |

Ниже — разбор **самых важных кодов** для API-разработки.

---

## 1xx – Информационные (редко используются в REST API)

| Код | Название | Когда использовать |
|-----|----------|---------------------|
| 100 | Continue | Сервер получил заголовки, клиент может продолжать отправлять тело (для больших запросов). |
| 101 | Switching Protocols | Переключение протоколов (например, с HTTP на WebSocket). |

В обычных REST API эти коды встречаются редко, чаще их обрабатывают низкоуровневые HTTP-библиотеки.

---

## 2xx – Успех (самые важные для API)

| Код | Название | Когда использовать | Пример |
|-----|----------|---------------------|--------|
| **200** | OK | Стандартный успех для GET, PUT, PATCH, DELETE (с телом ответа). | `GET /users/123` → возвращает пользователя. |
| **201** | Created | Ресурс создан (обычно для POST). В заголовке `Location` – URL нового ресурса. | `POST /users` → создан пользователь, `Location: /users/123`. |
| **202** | Accepted | Запрос принят в обработку, но результат ещё не готов (асинхронный режим). | `POST /reports` → отчёт генерируется, статус можно проверить позже. |
| **204** | No Content | Успех, но тело ответа пусто (обычно для DELETE или POST, которые не возвращают данных). | `DELETE /users/123` → удалено, возвращает пустое тело. |

**Важно:** 200 и 204 часто используются для идемпотентных методов (PUT, DELETE). Для создания (POST) предпочтительнее 201.

---

## 3xx – Перенаправления (важны для веба, реже для API)

| Код | Название | Когда использовать |
|-----|----------|---------------------|
| 301 | Moved Permanently | Ресурс permanently перемещён на новый URL (клиент должен обновить ссылки). |
| 302 | Found | Временное перенаправление (используется редко в API). |
| 304 | Not Modified | Клиент может использовать кешированную версию (условный GET с `If-None-Match` или `If-Modified-Since`). |

В REST API 304 часто возвращается для оптимизации трафика при кешировании. 301/302 больше для браузеров.

---

## 4xx – Ошибки клиента (клиент что-то сделал не так)

Это важнейший класс для проектирования API – он говорит клиенту, что запрос неверен, и что нужно исправить.

| Код | Название | Когда использовать | Тело ошибки |
|-----|----------|---------------------|--------------|
| **400** | Bad Request | Синтаксическая ошибка в запросе: невалидный JSON, отсутствие обязательного поля, неверный тип данных. | Желательно указать, какое поле и почему. |
| **401** | Unauthorized | Отсутствует или недействительная аутентификация (например, не передан токен, токен просрочен). | `{ "error": "Missing token" }` |
| **403** | Forbidden | Аутентификация есть, но недостаточно прав (пользователь не админ, не владелец ресурса). | `{ "error": "Access denied" }` |
| **404** | Not Found | Ресурс по указанному URI не существует. | `{ "error": "User not found" }` |
| **405** | Method Not Allowed | HTTP-метод не поддерживается для этого ресурса (например, DELETE для /products). | Возвращать заголовок `Allow: GET, POST`. |
| **409** | Conflict | Конфликт состояния (например, попытка создать пользователя с уже существующим email). | `{ "error": "Email already taken" }` |
| **422** | Unprocessable Entity | Семантическая ошибка – синтаксис верен, но бизнес-логика отвергает (например, товара нет на складе). | `{ "error": "Insufficient stock" }` |
| **429** | Too Many Requests | Превышен лимит запросов (rate limiting). | Заголовок `Retry-After: 60`. |

**Разница 400 vs 422:**  
- 400 – неверный синтаксис (например, `{"name":}`) или неверный тип (`{"age":"not_a_number"}`).  
- 422 – синтаксис правильный, но данные не проходят бизнес-валидацию (например, `{"email":"not_an_email"}` или попытка купить больше, чем есть).

**401 vs 403:**  
- 401 – «ты кто?» (не предоставлены учётные данные).  
- 403 – «ты кто, но тебе сюда нельзя» (учётные данные есть, но прав нет).

---

## 5xx – Ошибки сервера (клиент ни в чём не виноват)

Это проблемы на стороне сервера. Клиент может повторить запрос позже (кроме 501, 505). Важно, чтобы 5xx не возникали при корректных запросах – это показатель нестабильности.

| Код | Название | Когда использовать |
|-----|----------|---------------------|
| **500** | Internal Server Error | Общая ошибка сервера (необработанное исключение, сбой БД, таймаут). Клиент может повторить. |
| **501** | Not Implemented | Метод не поддерживается сервером (например, сервер не умеет PATCH). |
| **502** | Bad Gateway | Прокси/шлюз получил некорректный ответ от вышестоящего сервера. |
| **503** | Service Unavailable | Сервер временно перегружен или на обслуживании (клиент может повторить с заголовком `Retry-After`). |
| **504** | Gateway Timeout | Шлюз не дождался ответа от вышестоящего сервера. |

В хорошо спроектированном API 500 должен быть редким событием. Все предсказуемые ошибки лучше обрабатывать как 4xx.

| Действие / Ситуация | Метод | Успешный код | Код ошибки (пример) |
|---------------------|-------|--------------|----------------------|
| Получить список | GET | 200 | 400 (плохие параметры) |

| Асинхронная операция | POST | 202 | 400, 409 |
| Превышение лимита | любой | – | 429 |
| Неверная аутентификация | любой | – | 401 |
| Недостаточно прав | любой | – | 403 |

## 🔗 Связь с предыдущими темами

- **REST API** – коды ответов являются частью унифицированного интерфейса. REST рекомендует использовать стандартные коды.
- **Идемпотентность** – для идемпотентных методов (PUT, DELETE) при повторном вызове сервер может возвращать тот же код (200/204) даже после удаления – это нормально.
- **Метрики и логи** – коды 4xx и 5xx – ключевые метрики для мониторинга здоровья API.
- **Проектирование API** – выбор правильного кода ошибки улучшает клиентский опыт.

---

## 💎 Итог

- Коды 2xx = всё хорошо.
- 4xx = клиент ошибся (исправляй запрос).
- 5xx = сервер ошибся (повтори позже, сообщи админу).
- 3xx – редко в API, чаще для браузеров.
- 1xx – почти не используются в REST.

**Главное правило:** используйте семантически правильные коды, не придумывайте свои. Клиенты ожидают стандартное поведение. Для детализации ошибки используйте тело ответа с полем `error` или `code`.

Если хотите, могу детально разобрать **как обрабатывать коды на клиенте** (retry logic, fallback) или **как настроить алерты на 5xx в Prometheus**.

### 📜 OpenAPI и Swagger: В чем разница?

Исторически эти понятия связаны, но важно их различать. **OpenAPI — это стандарт, а Swagger — это инструменты.**

История такова: изначально существовал проект Swagger, который включал и спецификацию описания API, и инструменты для работы с ней. В 2015 году спецификация была передана под крыло Linux Foundation в новую OpenAPI Initiative и переименована в OpenAPI Specification (OAS). Сейчас последняя стабильная версия — **OpenAPI 3.1**.

Таким образом:
*   **OpenAPI Specification** — это открытый, вендор-нейтральный стандарт на языке YAML или JSON для описания REST API. Он описывает эндпоинты, входные и выходные данные, методы авторизации и многое другое.
*   **Swagger** — это бренд и набор инструментов от компании SmartBear, которые реализуют работу со спецификацией OpenAPI (например, Swagger UI, Swagger Editor, Swagger Codegen).

На практике термины часто используют как синонимы, но полезно помнить об этой разнице.

---

### 🏗️ Структура OpenAPI-спецификации

Спецификация — это, по сути, структурированный документ (YAML или JSON), состоящий из нескольких ключевых разделов:

*   `openapi`: Указывает версию спецификации.
*   `info`: Метаданные об API — название, версия, описание, контактная информация.
*   `servers`: Базовые URL-адреса для различных окружений (dev, staging, production).
*   `paths`: **Сердце документа**. Здесь описываются все эндпоинты (URL) и доступные для них HTTP-методы (GET, POST и т.д.).
*   `components`: Блок для переиспользуемых элементов, чтобы не дублировать код.
    *   `schemas`: Определения объектов данных (например, модель `User`).
    *   `parameters`: Переиспользуемые параметры запроса.
    *   `responses`: Шаблоны ответов.
    *   `securitySchemes`: Определения методов аутентификации.

Проще всего начать изучение структуры с "шпаргалки" (cheat sheet) по OpenAPI, где в сжатом виде представлены все основные возможности и синтаксис спецификации.

### 🛠️ Как и чем генерировать документацию?

Это самый практический вопрос. Существует два подхода, и выбор зависит от вашей философии разработки.

#### 1. Code-First (генерация из кода) 🔼

Этот подход наиболее популярен, так как он гарантирует, что документация **всегда актуальна**, ведь она генерируется автоматически из кода приложения. 

Вы просто добавляете аннотации к своему коду, а библиотека в рантайме собирает спецификацию. Затем можно использовать любой рендерер для визуализации документации.

*   **Для Java (Spring Boot):** используется **SpringDoc OpenAPI**. Добавляете зависимость `springdoc-openapi-starter-webmvc-ui` в проект, и библиотека автоматически сканирует ваши `@RestController` и строит спецификацию. Swagger UI становится доступен по адресу `/swagger-ui.html`.
*   **Для Python (FastAPI):** FastAPI имеет встроенную поддержку. Он автоматически генерирует спецификацию OpenAPI из ваших аннотаций типов и Pydantic-моделей. Документация Swagger UI доступна по умолчанию по адресу `/docs`.
*   **Для Node.js (Express):** популярна библиотека `swagger-jsdoc`. Вы описываете API с помощью комментариев в формате JSDoc, а библиотека собирает их в единую спецификацию. Swagger UI подключается через `swagger-ui-express`.

**Главный минус:** код "обрастает" специфическими аннотациями, что немного засоряет его.

#### 2. Design-First (сначала дизайн) 📝

В этом подходе сначала создается YAML/JSON-файл спецификации, и только потом на его основе генерируются заглушки для кода сервера (server stubs) и клиентские библиотеки.

Этот подход отлично подходит для крупных проектов, где API проектируется отдельной командой или архитектором, и важно, чтобы все строго следовали контракту. Для этого можно использовать визуальные редакторы вроде **Stoplight** или онлайн-редактор **Swagger Editor**.

**Главный минус:** высокий риск расхождения спецификации с реальным кодом, если не настроить автоматическую синхронизацию.

#### 3. Выбор инструмента для визуализации 📄

Самый простой и популярный способ увидеть документацию — использовать **Swagger UI**. Это тот самый интерфейс с синим фоном и выпадающими списками эндпоинтов. Его главный плюс — встроенная кнопка **"Try it out"**, позволяющая отправлять реальные запросы прямо из браузера. Минус — устаревший дизайн и возможная медленная работа при огромном количестве методов.

**Redoc** — это более современная альтернатива с красивым дизайном в три колонки. Сайты документации Stripe, Twilio и других крупных компаний выглядят именно так. Главный недостаток бесплатной версии — отсутствие интерактивной консоли для тестирования.

**Альтернативы:**
*   **Scalar:** Современный инструмент с фокусом на опыт разработчика, темной темой и удобным поиском.
*   **Stoplight:** Не просто рендерер, а полноценная платформа для дизайна, документирования и тестирования API.
*   **Mintlify:** Платформа для создания документации, похожей на сайт Vercel, с поддержкой OpenAPI, React-компонентами и аналитикой.

Также существуют коммерческие платформы полного цикла, такие как **ReadMe** и **GitBook**, которые предлагают не только красивый интерфейс, но и возможности для онбординга пользователей, генерации кода и аналитики.

### 💎 Лучшие практики документирования

Чтобы документация была полезной, а не просто формальностью, стоит следовать нескольким правилам:

1.  **Пишите осмысленные описания (descriptions).** Не ограничивайтесь просто `summary: Get bookings`. Добавьте `description`, объясняющую, что делает эндпоинт, какие параметры и как использовать. Это значительно улучшит опыт разработчика, который интегрирует ваш API.
2.  **Всегда добавляйте примеры (examples).** Для параметров, тела запроса и тела ответа. Это самый быстрый способ понять, как работает API.
3.  **Детально описывайте коды ответов.** Не только 200, но и все возможные ошибки (400, 401, 404, 422 и т.д.) с пояснениями.
4.  **Используйте семантическое версионирование (SemVer).** Указывайте версию API в поле `info.version`.

### ⚡️ Расширенные возможности OpenAPI

OpenAPI — это не только про чтение. Это мощный инструмент для автоматизации:

*   **Генерация кода:** Используя спецификацию, можно с помощью инструмента **OpenAPI Generator** автоматически сгенерировать SDK для клиента на десятках языков (Go, Java, Python, TypeScript и др.) или каркас (stub) для сервера.
*   **Автоматическое тестирование:** Спецификация может служить контрактом для тестов. Например, инструмент **Dredd** может проверять, что реальный API соответствует своему OpenAPI-описанию. Это основа для **контрактного тестирования**.
*   **Стандартизация ошибок:** OpenAPI позволяет строго описать единый формат ответа с ошибкой, что делает обработку ошибок на клиенте предсказуемой.

Документирование API — это не просто завершающий этап, а важнейшая часть процесса разработки, которая определяет успех продукта. Связка OpenAPI (стандарт) и Swagger (инструменты) стала отраслевым стандартом. Какой бы подход — Code-First или Design-First — вы ни выбрали, наличие актуальной, интерактивной и хорошо описанной спецификации сэкономит массу времени вам, вашей команде и всем, кто будет использовать ваше API.

Если появятся вопросы по внедрению или выбору конкретных инструментов для вашего стека технологий, обращайтесь, будем разбираться вместе.
