Вы перешли к ключевому навыку инженера — **проектированию API**. Это искусство создания интерфейсов, которые будут удобны для клиентов, стабильны в работе, безопасны и эволюционируют без боли для обеих сторон.

Проектирование API — это не просто «написать контроллер». Это продумывание контракта, который будут использовать другие разработчики (или даже вы сами через полгода). Плохой API может сломать клиентов при каждом обновлении; хороший — служить годами.

Я разобью ответ на принципы, подходы (REST, GraphQL, gRPC), версионирование, безопасность, документацию и пример.

## 🧠 Основные принципы проектирования API

| Принцип | Суть | Пример нарушения | Хорошее решение |
|---------|------|------------------|------------------|
| **Согласованность** | Одинаковые стили именования, форматы ответов, коды ошибок | `/getUser`, `/fetch_products`, `/deleteById` | Все глаголы в HTTP-методах: `GET /users`, `POST /products` |
| **Предсказуемость** | Клиент может догадаться о поведении, не читая документацию | `DELETE /order` удаляет заказ, но не возвращает что удалил | Возвращать 204 No Content или удалённый объект |
| **Простота** | Минимум вложенности, избыточных полей | `/api/v1/shops/123/orders/456/items/789` | `/api/v1/items/789` или использовать query параметры |
| **Безопасность** | Защита от OWASP Top 10, валидация, rate limiting | Принимает `{ "id": "1; DROP TABLE users" }` | Использовать параметризованные запросы, белый список полей |
| **Эволюционность** | Добавление новых полей не ломает старых клиентов | Убрали поле `phone` из ответа | Добавлять поля опционально, не удалять без deprecation |
| **Документированность** | API самодокументируем (OpenAPI) или есть всегда актуальная спецификация | «Документация в коде, смотри комментарии» | OpenAPI (Swagger), Postman коллекции, gRPC proto |

## 🧭 Выбор стиля API: REST, GraphQL, gRPC, Webhook

### 1. REST (Representational State Transfer) — классика

- **Ресурсная модель** — всё сущности (`/users`, `/orders`).
- **HTTP методы**:
  - `GET` — получение (идемпотентный, безопасный).
  - `POST` — создание / действие (не идемпотентный).
  - `PUT` — полная замена ресурса (идемпотентный).
  - `PATCH` — частичное обновление.
  - `DELETE` — удаление.
- **Коды ответа**:
  - `200 OK`, `201 Created`, `204 No Content`
  - `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`
  - `500 Internal Server Error`
- **Плюсы**: простота, кеширование через HTTP, понятность.
- **Минусы**: over-fetching / under-fetching, много запросов для сложных UI.

**Пример REST API:**
```
GET    /users          → список пользователей
GET    /users/123      → конкретный пользователь
POST   /users          → создать пользователя (тело в JSON)
PUT    /users/123      → заменить целиком
PATCH  /users/123      → обновить email
DELETE /users/123      → удалить
```

### 2. GraphQL — клиент выбирает поля

- **Один эндпоинт** (обычно `/graphql`).
- **Клиент описывает** какие поля ему нужны.
- **Плюсы**: нет over-fetching, одно запрос на несколько ресурсов.
- **Минусы**: сложность кеширования, возможны медленные запросы (нужен depth limit, анализатор сложности).

**Пример запроса:**
```graphql
query {
  user(id: 123) {
    name
    email
    posts(last: 5) { title }
  }
}
```

### 3. gRPC (HTTP/2 + Protobuf) — для высокопроизводительных внутренних API

- **Строгая схема** в `.proto` файлах.
- **Генерация клиентов и серверов** на многих языках.
- **Поддерживает стриминг** (однонаправленный, двунаправленный).
- **Плюсы**: скорость, типизация, бинарный протокол.
- **Минусы**: сложнее отладка (нужны инструменты типа grpcurl), браузер напрямую не поддерживает (но можно через grpc-web).

**Пример proto:**
```protobuf
service UserService {
  rpc GetUser (GetUserRequest) returns (User);
}
message GetUserRequest { int64 id = 1; }
message User { string name = 1; string email = 2; }
```

### 4. Webhook — когда сервер сам вызывает клиента

- **Обратный API** — вы даёте URL, и сервер отправляет POST на него при наступлении события.
- **Плюсы**: асинхронное уведомление, снижение polling.
- **Минусы**: клиент должен быть доступен и безопасен (проверка подписи запроса).

## 📐 Проектирование ресурсов и эндпоинтов (REST)

### Именование
- **Существительные во множественном числе**: `/users`, `/orders`.
- **Вложенность для связей**: `/users/123/orders` (заказы пользователя), но не глубже 2-3 уровней.
- **Действия, не укладывающиеся в CRUD** — выносим в подресурс с глаголом: `POST /users/123/activate`, `POST /orders/456/cancel`.

### Пагинация, фильтрация, сортировка
- Через query параметры:
  - `?limit=20&offset=40` (простая) или `?page=2&size=20`.
  - Ключ-смещение (cursor) для бесконечной прокрутки: `?cursor=abc123&limit=20`.
- Фильтры: `?status=active&role=admin`.
- Сортировка: `?sort=-created_at` (минус для убывания).

### Частичные ответы (sparse fields)
- Позволяет клиенту запросить только нужные поля: `GET /users/123?fields=id,name`.
- Полезно для уменьшения трафика.

### Версионирование
- **В URL**: `/v1/users`, `/v2/users` — просто, наглядно, но раздувает эндпоинты.
- **В заголовке**: `Accept: application/vnd.myapi.v1+json` — чище URL, но сложнее для тестирования.
- **Через пользовательский заголовок**: `API-Version: 1`.
- **Стратегия**: поддерживать несколько версий параллельно, deprecation policy (например, 6-12 месяцев).

**Важно**: Версионируйте только ломающие изменения (удаление поля, изменение типа, другое поведение). Добавление опциональных полей — не ломает.

## 🔒 Безопасность API

| Угроза | Защита |
|--------|--------|
| Неавторизованный доступ | Аутентификация (JWT, OAuth2, API-ключи) + авторизация (роли, scopes) |
| Подслушивание | Только HTTPS (TLS 1.2+) |
| Повтор атак | Nonce, timestamp, подпись запроса |
| Инъекции (SQL, NoSQL) | Параметризованные запросы, ORM, валидация входных данных |
| DoS / DDoS | Rate limiting (лимит на пользователя/IP), WAF, CDN |
| Чрезмерное использование ресурсов | Пагинация, лимиты на размер запроса/ответа |
| Утечка данных | Не возвращать чувствительные поля без необходимости, логировать обращения |

## 📄 Документирование API

### OpenAPI (Swagger) — стандарт для REST
- Описывает эндпоинты, параметры, ответы, схемы.
- Генерирует интерактивную документацию (Swagger UI).
- Позволяет генерировать клиентские SDK.

Пример спецификации (YAML):
```yaml
openapi: 3.0.0
paths:
  /users/{id}:
    get:
      summary: Get user by ID
      parameters:
        - name: id
          in: path
          required: true
          schema: { type: integer }
      responses:
        '200':
          description: OK
          content:
            application/json:
              schema:
                type: object
                properties:
                  id: { type: integer }
                  name: { type: string }
```

### Инструменты
- **Swagger UI** — визуализация.
- **ReDoc** — альтернативный рендер.
- **Postman Collections** — экспорт/импорт.
- **AsyncAPI** — для событийных/вебхук API.
- **gRPC reflection** + **protoc** генерирует документацию из `.proto`.

## 🧪 Тестирование API (в контексте проектирования)

Хорошо спроектированный API легко тестировать:

- **Контрактные тесты** (Pact) — проверяют, что сервер и клиент соблюдают спецификацию.
- **Автотесты** (Supertest, REST Assured) — проверяют каждый эндпоинт на:
  - Валидные запросы → ожидаемый ответ.
  - Невалидные → ожидаемые ошибки (400, 404, 422).
  - Граничные условия (пагинация, лимиты).
- **Нагрузочные тесты** — подскажут, нужно ли добавить rate limiting.

## 📦 Пример проектирования простого API (интернет-магазин)

### Ресурсы
- `GET    /products` — список товаров (пагинация, фильтр по категории)
- `GET    /products/{id}` — детали товара
- `POST   /cart` — добавить товар в корзину (анонимную)
- `GET    /cart` — получить содержимое корзины
- `POST   /orders` — оформить заказ (требует аутентификации)
- `GET    /orders` — заказы текущего пользователя
- `GET    /orders/{id}` — детали заказа

### Аутентификация
- Регистрация/логин: `POST /auth/register`, `POST /auth/login` → возвращает JWT.

### Версионирование
- Начинаем с `/v1/...`.
- При ломающих изменениях (например, поле `price` становится объектом `{amount, currency}`) создаём `/v2/...`.

### Rate limiting
- Для неавторизованных: 100 запросов/час.
- Для авторизованных: 1000 запросов/час.

### Документация
- OpenAPI (Swagger) по адресу `/docs`.

## 💡 Рекомендации для начинающих

1. **Начинайте с REST** — он самый понятный и распространённый. Переходите на GraphQL или gRPC, когда REST станет мешать (over-fetching, много запросов).
2. **Сначала напишите спецификацию** (OpenAPI) — даже если не генерируете код, она прояснит мысли.
3. **Используйте идемпотентность** для `PUT`, `DELETE` — повторный вызов не должен менять состояние.
4. **Возвращайте понятные сообщения об ошибках**:
   ```json
   { "error": { "code": "INVALID_EMAIL", "message": "Email must be valid", "field": "email" } }
   ```
5. **Не раскрывайте внутренние детали** (стек трейсы, имена таблиц) в ответах API.
6. **Продумайте deprecation policy** — например, помечайте устаревшие поля в документации и заголовке `Deprecation: true`, удаляйте не раньше чем через 6 месяцев.

### 🧱 Архитектура и Место в Системе

Платежный шлюз — это сервис-посредник, который безопасно передает данные о платеже между вашим сайтом, банком-эквайером и платежными системами.

В современных проектах часто применяется паттерн **API Gateway**, который становится единой точкой входа для всех клиентов. Он берет на себя маршрутизацию запросов к внутренним сервисам (включая платежный), проверку токенов, ограничение частоты запросов и централизованное логирование. Это позволяет абстрагировать внутреннюю структуру платежной системы от внешнего мира.

> **Профессиональный совет:** Для ускорения разработки и абстракции от конкретных провайдеров можно использовать готовые self-hosted решения, такие как **RusPay Core**. Этот шлюз объединяет под единым API нескольких российских провайдеров (YooKassa, Tinkoff Pay, SberPay) и может быть развернут за 10 минут через `docker compose up`.

### 🛡️ Безопасность: Три Кита Платежного Шлюза

1.  **Соответствие стандарту PCI DSS (Payment Card Industry Data Security Standard)**
    Это свод из 12 требований к безопасности, обязательный для любой компании, которая обрабатывает, хранит или передает данные держателей карт. Ваш шлюз **обязан** его соблюдать. Это включает в себя шифрование данных при передаче (TLS) и защиту серверов от атак.

2.  **Токенизация (Tokenization)**
    Это лучшая практика безопасности, которая позволяет вам никогда не хранить и даже не видеть данные карты клиента. Система шлюза заменяет реальные данные карты на уникальный цифровой идентификатор — **токен**. Вы сохраняете только этот токен, и в случае утечки вашей базы данных, данные карт остаются в безопасности у провайдера.

3.  **Протокол 3D Secure (3DS)**
    Это дополнительный уровень аутентификации картодержателя со стороны его банка (например, подтверждение по SMS или в мобильном банке). Современные шлюзы используют **3D Secure 2.0**, который применяет "пассивную" оценку рисков для создания бесшовного опыта: только транзакции с высоким уровнем риска требуют активного подтверждения, остальные проходят без лишних шагов.

### 📨 Вебхуки (Webhooks): Обработка Асинхронных Событий

Платеж — это не мгновенный процесс. Банк может обрабатывать его несколько секунд. Поэтому шлюзы используют **вебхуки** — механизм, когда шлюз сам отправляет HTTP-запрос на ваш указанный URL, когда статус транзакции меняется (успех, ошибка, возврат). Это сердце надежной интеграции.

#### Идемпотентность при обработке вебхуков
Из-за сетевых сбоев вебхук может быть доставлен дважды. Ваша система **обязана** корректно обрабатывать дубликаты, чтобы не зачислить клиенту бонус дважды или не отправить одно и то же уведомление на email.

Правильная реализация обработчика выглядит так:

```python
def handle_payment_webhook(request):
    # 1. Получаем уникальный ID события от шлюза (например, Stripe event.id)
    event_id = request.json['id']

    # 2. Атомарная операция: пытаемся вставить ID в БД с уникальным ограничением
    with db.transaction():
        if db.execute('INSERT INTO processed_events (event_id) VALUES (%s)', [event_id]):
            # 3. ID не было в БД -> обрабатываем событие
            process_payment_logic(request.json)
            db.commit()
        else:
            # 4. Дубликат, просто игнорируем
            return Response(status=200)
```

Этот подход гарантирует, что каждое уникальное событие будет обработано ровно один раз.

#### Надежность через асинхронную очередь
Чтобы избежать таймаутов, обработчик вебхука должен быть максимально быстрым. Лучшая практика — принимать вебхук, быстро возвращать статус `200 OK` и класть полученные данные в очередь (например, RabbitMQ, Kafka), а бизнес-логику выполнять фоновыми воркерами.

### 💎 Лучшие Практики и Инструменты

*   **Отказоустойчивость и повторы**: Всегда проектируйте систему с учетом возможности сбоев. Ваш обработчик вебхука должен корректно обрабатывать повторные вызовы (см. идемпотентность), а сам шлюз — иметь механизм автоматических повторов при недоступности вашего эндпоинта.
*   **Логирование**: Ведите детальное логирование всех входящих запросов от шлюза и исходящих к нему. Это незаменимо для отладки.
*   **Мониторинг**: Настройте алерты на падение обработчика вебхуков, резкое увеличение количества ошибок или появление событий в dead-letter queue (DLQ).
*   **Стартовый шаблон (FastAPI + Docker)**: Для быстрого старта интеграции вы можете использовать следующий базовый код на Python с использованием фреймворка FastAPI:

```python
from fastapi import FastAPI, HTTPException, Request, Header
from pydantic import BaseModel
import hashlib, hmac

app = FastAPI()

class PaymentWebhook(BaseModel):
    event_id: str
    event_type: str
    payment_id: str
    amount: int
    # ... другие поля

def verify_signature(payload: bytes, signature: str, secret: str) -> bool:
    # Сравнивает подпись, вычисленную с вашим секретом, с подписью из заголовка
    computed = hmac.new(secret.encode(), payload, hashlib.sha256).hexdigest()
    return hmac.compare_digest(computed, signature)

@app.post("/webhook/payment")
async def payment_webhook(
    request: Request,
    webhook_data: PaymentWebhook,
    x_signature: str = Header(...),
):
    # 1. Проверяем подпись, чтобы убедиться, что запрос пришел именно от шлюза
    raw_body = await request.body()
    if not verify_signature(raw_body, x_signature, "YOUR_WEBHOOK_SECRET"):
        raise HTTPException(status_code=401, detail="Invalid signature")
    
    # 2. Реализуем идемпотентность
    async with db.transaction():
        if await db.fetch_val("SELECT 1 FROM processed_events WHERE event_id = $1", webhook_data.event_id):
            return {"status": "ignored"}
        await db.execute("INSERT INTO processed_events (event_id) VALUES ($1)", webhook_data.event_id)
    
    # 3. Кладем задачу в очередь для фоновой обработки
    await queue.publish("payment_tasks", webhook_data.dict())
    
    return {"status": "queued"}
```

Этот пример демонстрирует ключевые принципы: проверку подписи, идемпотентность и асинхронную обработку.

**Характеристики API** — это набор свойств и метрик, по которым оценивают качество API. Они делятся на **функциональные** (что API делает) и **нефункциональные** (как оно это делает). Для бизнеса и разработчиков важны обе группы, так как они влияют на пользовательский опыт, стоимость разработки и поддержки.

Ниже — ключевые характеристики с пояснениями и примерами.

---

## 1. Функциональные характеристики (что умеет API)

| Характеристика | Описание | Пример |
|----------------|----------|--------|
| **Полнота (Completeness)** | API предоставляет все необходимые операции для работы с ресурсом | API управления заказами должно позволять создать, прочитать, обновить, отменить, вернуть заказ |
| **Правильность (Correctness)** | API следует спецификации, бизнес-правилам, не содержит ошибок | `POST /payment` должен отклонять отрицательную сумму |
| **Идемпотентность (Idempotency)** | Повторный вызов с теми же параметрами не изменяет состояние (для безопасных методов) | `PUT /users/123` с одинаковым телом дважды → результат тот же |
| **Безопасность (Security)** | API защищает данные, аутентифицирует и авторизует клиентов, устойчиво к атакам | Требует JWT, валидирует входные данные, логирует подозрительную активность |
| **Согласованность (Consistency)** | Однотипные операции выглядят одинаково (именование, форматы, коды ошибок) | Все GET-запросы используют пагинацию через `?limit`, `?offset` |

---

## 2. Нефункциональные характеристики (как работает API)

### 2.1. Производительность (Performance)

- **Latency (задержка)** — время от отправки запроса до получения ответа (обычно p50, p95, p99). Чем меньше, тем лучше.
- **Throughput (пропускная способность)** — количество запросов в секунду, которое выдерживает API.
- **Response time** — время обработки на сервере (без учёта сети).

> Хороший API: p95 latency < 200 мс для чтения, < 500 мс для записи.

### 2.2. Надёжность (Reliability)

- **Availability (доступность)** — процент времени, когда API работает. Измеряется в «девятках»: 99.9% (три девятки) — допустимо 8.76 часов простоя в год, 99.99% — 52 минуты в год.
- **Error rate** — процент запросов, завершившихся с ошибкой 5xx.
- **Resilience (отказоустойчивость)** — способность восстанавливаться после сбоев (retries, circuit breaker, graceful degradation).

### 2.3. Масштабируемость (Scalability)

- **Horizontal scaling** — возможность добавлять больше серверов для увеличения пропускной способности.
- **Vertical scaling** — увеличение мощности одного сервера (ограничено).

Хорошо спроектированное API является stateless, что позволяет легко масштабировать его горизонтально.

### 2.4. Простота использования (Usability / Developer Experience)

- **Читаемость** — понятные имена эндпоинтов, параметров, сообщений об ошибках.
- **Документация** — наличие интерактивной документации (Swagger/OpenAPI), примеров кода, Postman-коллекций.
- **Простота аутентификации** — стандартные схемы (Bearer token, API key), а не кастомные танцы с бубном.
- **Предсказуемость** — поведение API не сюрпризное (например, `DELETE` возвращает `204` или `200` с удалённым объектом — нужно единообразие).

### 2.5. Эволюционная способность (Evolvability)

- **Обратная совместимость** — добавление новых полей не ломает старых клиентов.
- **Версионирование** — поддержка нескольких версий одновременно с понятной политикой устаревания (deprecation).
- **Расширяемость** — возможность добавлять новые эндпоинты или методы без переписывания всего API.

### 2.6. Безопасность (Security — пересекается с функциональной)

- **Аутентификация и авторизация** — чёткое разделение ролей.
- **Шифрование** — TLS/HTTPS обязателен.
- **Защита от атак** — rate limiting, валидация входа, санитизация вывода.
- **Аудит** — логирование доступа к критическим операциям.

### 2.7. Наблюдаемость (Observability)

- **Логирование** — каждый запрос имеет уникальный correlation ID, логируются ошибки.
- **Метрики** — Prometheus-метрики: число запросов, ошибки, задержки по эндпоинтам.
- **Трассировка** — распределённая трассировка (Jaeger, Zipkin) для микросервисов.

## 3. Дополнительные характеристики (в зависимости от контекста)

| Характеристика | Описание |
|----------------|----------|
| **Транзакционность** | Поддержка атомарных операций (например, двухфазный коммит) — редко для REST, чаще для gRPC или GraphQL с мутациями |
| **Кешируемость** | Возможность кешировать ответы на клиенте или на CDN (заголовки `Cache-Control`, `ETag`) |
| **Сжатие** | Поддержка `gzip`/`br` для уменьшения трафика |
| **Поддержка потоков** | Стриминг (WebSocket, gRPC streams, Server-Sent Events) для реального времени |
| **Политика ограничений** | Rate limits (лимит запросов в минуту/час), квоты (месячный лимит) |

Логирование API — это не просто запись событий, а фундамент наблюдаемости, безопасности и отладки вашего сервиса. В современной разработке, где системы становятся всё более распределёнными, грамотно настроенные логи — это "чёрный ящик", без которого невозможно понять, что происходит с вашим API в production.

Ниже — полная картина того, как устроены логи API: от базовых принципов до интеграции с мощными инструментами вроде OpenTelemetry.

### 🎯 Что логировать в API

Главное правило: каждый лог должен быть самодостаточным и помогать ответить на вопросы "Что произошло?", "Когда?", "С кем?" и "Почему?".

Вот минимальный, но достаточный набор полей для каждого запроса:

| Категория | Обязательные поля | Для чего нужны |
| :--- | :--- | :--- |
| **Идентификация запроса** | Уникальный `request_id` / `trace_id` | Позволяет отследить путь одного запроса через все микросервисы. Связывает логи, метрики и трейсы воедино. |
| **Время** | Таймстемп в формате ISO8601, общее время обработки (`request_time`) | Для анализа производительности (например, "какой эндпоинт самый медленный?"). Используйте единый часовой пояс (UTC). |
| **Контекст клиента** | IP-адрес, User-Agent, ID клиента (если есть) | Помогает идентифицировать источник проблемы (например, все ошибки идут от одного типа устройств). |
| **Детали запроса** | HTTP-метод, путь (`path`), строка запроса (`query`), протокол | Понять, какое действие выполнял пользователь. |
| **Результат** | HTTP-статус ответа, размер ответа в байтах | Основной индикатор здоровья API (например, резкий рост 5xx). |
| **Ошибки** | Сообщение об ошибке, stack trace (для уровня `ERROR` и только в dev/тестовых контурах) | Главный инструмент для отладки. Без stack trace вы, скорее всего, не поймёте причину падения. |
| **Системный контекст** | Имя сервиса, версия API, окружение (`dev`, `staging`, `prod`) | Незаменимо в микросервисной архитектуре для понимания, где именно произошёл сбой. |

### 📐 Форматы и структура: почему JSON — стандарт де-факто

Писать логи в виде простых строк — это тупиковый путь. Классический формат Nginx "combined" хорош для человека, но кошмарен для автоматического парсинга. Любое изменение формата ломает все парсеры и пайплайны.

**Структурированное логирование в формате JSON** решает эту проблему. Вот как выглядит идеальная JSON-строка для одного API-вызова:

```json
{
  "timestamp": "2025-04-11T21:02:40.171Z",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "span_id": "00f067aa0ba902b7",
  "request": {
    "method": "GET",
    "path": "/api/v1/users",
    "query": "?page=2"
  },
  "response": {
    "status": 200,
    "size": 1024
  },
  "client": {
    "ip": "192.168.1.1",
    "user_agent": "Mozilla/5.0...",
    "id": "user-123"
  },
  "service": {
    "name": "user-service",
    "version": "v1.2.3",
    "env": "production"
  },
  "performance": {
    "duration_ms": 145,
    "db_query_ms": 120
  },
  "level": "INFO"
}
```

**Почему это удобно:**
*   **Машинная читаемость**: JSON могут парсить любые системы, от простых скриптов до сложных пайплайнов.
*   **Гибкость и эволюция**: В JSON можно легко добавить новое поле (например, `"db_query_ms": 120`) без поломки существующих парсеров.
*   **Мощный поиск**: Системы вроде Elasticsearch позволяют делать сложные запросы вроде "найти все логи уровня ERROR для сервиса `payment` за последний час".

### ⚙️ Уровни логирования: как не утонуть в информационном шуме

Использование уровней логирования помогает быстро фильтровать логи по степени критичности. Вот принятая в индустрии классификация:

*   **`ERROR`**: Используйте для критических проблем, которые уже привели к отказу функции. Сервис всё ещё может работать, но его функциональность нарушена. **Обязательно логируйте сюда все 5xx ошибки** для немедленного реагирования.
*   **`WARN`**: Для нештатных, но не фатальных ситуаций. Например, клиент приближается к лимитам запросов, используется устаревший эндпоинт (deprecated) или запрос выполняется неоправданно долго.
*   **`INFO`**: Для отметки важных бизнес-событий: пользователь зарегистрировался, заказ создан, платёж проведён. Это основные вехи, по которым можно понять, что система работает штатно.
*   **`DEBUG`**: Только для разработки и отладки на небоевых контурах (`dev`/`staging`). Содержит максимум деталей: внутренние переменные, полные стеки вызовов, тело запроса и т.д. Включать `DEBUG` на `production` — прямой путь к утечке данных и дикой нагрузке.
*   **`TRACE`**: Ещё более детальный уровень, чем `DEBUG`. Используется крайне редко, для низкоуровневой отладки фреймворков.

### 🔒 Безопасность и конфиденциальность

Логи — это лакомый кусок для злоумышленника. Никогда и ни при каких обстоятельствах не логируйте чувствительные данные в открытом виде. Это нарушает требования PCI DSS, GDPR и других регуляторов.

*   **Запрещённые данные**: Пароли, токены доступа (JWT, API keys), номера банковских карт (PAN), персональные данные (PII) и любая другая информация, которую можно использовать для кражи личности или доступа.
*   **Как защитить логи**: Используйте механизмы **маскирования (masking)**. Например, оставлять только последние 4 цифры карты (`**** **** **** 1234`) или полностью заменять значение поля `password` на `******`.
*   **Современные инструменты**: Существуют высокопроизводительные инструменты, например написанный на Rust `PhantomTrace`, который в реальном времени может сканировать поток логов, находить и маскировать конфиденциальную информацию по заданным шаблонам, обеспечивая соответствие стандартам безопасности.

### 🔗 Инструменты и лучшие практики: наблюдаемость (Observability)

Логи — это часть более широкой концепции **наблюдаемости**. Она включает в себя логи, метрики и трассировки.

#### Связка Логов и Трассировки (Trace-Log Correlation)

Самый эффективный способ отладки в микросервисной архитектуре — это возможность перейти от конкретного трейса (лога запроса) к детальным логам сервисов, которые его обрабатывали, и наоборот.

*   **Как это работает**: Вы генерируете уникальный `trace_id` для каждого входящего запроса. Затем этот `trace_id` передаётся через все сервисы (например, в HTTP-заголовке). Все логи внутри этих сервисов записываются с этим же `trace_id`. OpenTelemetry делает это автоматически, добавляя в JSON-логи поля `trace_id` и `span_id`.
*   **Пример**: Представьте, что запрос пользователя прошёл через API Gateway, сервис заказов и сервис платежей. Если на одном из этапов произошла ошибка, вы можете просто взять `trace_id` из ошибки в Kibana, перейти в Jaeger и увидеть весь путь запроса, включая время каждого вызова. Это незаменимо при поиске узких мест.

#### Популярные системы управления логами

Хранить и анализировать логи в файлах на сервере — непозволительная роскошь для современного продакшена. Вам нужна централизованная система для сбора, хранения и анализа логов.

*   **ELK Stack (Elasticsearch, Logstash, Kibana)**: Мощный, зрелый стек для сложного полнотекстового поиска и глубокой аналитики. Требует знаний для настройки и может быть ресурсоёмким.
*   **Grafana Loki**: Более лёгкая, простая и дешёвая альтернатива. Отлично интегрируется с экосистемой Grafana. Идеальна для проектов, где важен бюджет и простота.
*   **SigNoz**: Современная платформа, объединяющая логи, метрики и трейсы "из коробки", что упрощает корреляцию данных для быстрого поиска корневых причин проблем.

### 💎 Главные правила для хороших логов

1.  **Структурированность**: Пишите логи в формате JSON — это основа основ.
2.  **Сквозная идентификация**: Всегда передавайте уникальный ID запроса (`correlation_id` или `trace_id`) через все сервисы.
3.  **Безопасность**: Никогда не логируйте пароли, токены и персональные данные.
4.  **Информированность**: Логируйте не только ошибки, но и успешные запросы, чтобы иметь полную картину.
5.  **Централизация и хранение**: Сразу отправляйте логи в центральную систему. Определите политику ротации и хранения логов в зависимости от их ценности и требований законодательства. Например, дебажные логи можно хранить неделю, а аудиторские — несколько лет.

В отличие от логов (каждый запрос — отдельная запись), метрики сжимают информацию и позволяют строить графики, выставлять алерты и анализировать тренды. Это основа мониторинга здоровья, производительности и бизнес-активности вашего API.

## 🎯 Ключевые типы метрик (RED и USE)

Для API чаще всего используют две модели: **RED** (для сервисов) и **USE** (для ресурсов). Начнём с **RED** — она идеально подходит для микросервисов и API-шлюзов.

| Метрика | Что означает | Пример для API | Как измерять |
|---------|--------------|----------------|---------------|
| **Rate** (скорость запросов) | Количество запросов в секунду / минуту | RPS (requests per second) на эндпоинт `GET /users` | Счётчик запросов за интервал |
| **Errors** (ошибки) | Доля или количество ответов с ошибками (4xx, 5xx) | Процент запросов к `/payment`, вернувших 500 | Счётчик ошибок / общее число запросов |
| **Duration** (длительность) | Время обработки запроса (latency) | p50, p95, p99 latency для `POST /orders` | Гистограмма или суммарное время |

**Дополнительно** для ресурсов (USE):
- **Utilization** — утилизация CPU, памяти, соединений.
- **Saturation** — перегрузка (очереди, тредпулы).
- **Errors** — ошибки на уровне ресурсов (например, таймауты БД).

## 📊 Какие метрики собирать для API (подробно)

### 1. Traffic (нагрузка)
- **Total requests** — общее число запросов (разбивка по эндпоинтам, методам, статусам).
- **Requests per second (RPS)** — мгновенная нагрузка.
- **Concurrent connections** — количество активных соединений.
- **Rate per client / API key** — нагрузка от конкретного потребителя.

### 2. Errors
- **HTTP status code distribution** — сколько 200, 400, 401, 403, 404, 429, 500.
- **Error rate** — `errors / total requests` (порог алерта >1% для 5xx).
- **Application errors** — специфичные коды ошибок (например, `INSUFFICIENT_FUNDS`).
- **Panics / exceptions** — счётчик необработанных падений.

### 3. Performance (latency)
- **p50 (медиана)** — половина запросов быстрее этого времени.
- **p95** — 95% запросов быстрее (обычно SLA обещают по p95 или p99).
- **p99** — 99% запросов быстрее (для критичных API).
- **Max** — максимальное время (шумно, лучше не использовать для алертов).
- **Apdex score** — метрика удовлетворённости (0–1), например, порог T=0.5c: быстрые <0.5c, терпимые <2c, медленные >2c.

### 4. Saturation (перегрузка)
- **Queue length** — размер очереди запросов перед обработкой.
- **Thread pool utilization** — сколько занято потоков.
- **Connection pool usage** — пул соединений к БД.
- **Rate limiting hits** — сколько запросов отброшено из-за лимитов.

### 5. Business metrics (специфичные для API)
- **Number of payments** / **Total amount**.
- **Registrations per minute**.
- **Search queries per second**.
- **Cache hit ratio** (для кешируемых API).

## 📈 Как задавать метрики и алерты

Хороший алерт основан на **изменении поведения**, а не на абсолютных значениях (хотя и они важны).

| Ситуация | Метрика | Порог алерта | Сложность |
|----------|---------|--------------|-----------|
| API перестал отвечать | `up` (probe success) | 0 за 1 минуту | P0 |
| Рост ошибок 5xx | `rate(http_requests_total{status=~"5.."}[5m])` | >0.05 (5%) | P1 |
| Медленные ответы | `histogram_quantile(0.95, duration_bucket)` | >1 секунда | P1 |
| Нехватка мест в очереди | `queue_length` | >100 | P2 |
| Подозрительная активность (брутфорс) | `rate(http_requests_total{path="/login", status=401}[1m])` | >10 | P2 |

**Правила алертов (избегать шума):**
- Используйте `for: 5m` — срабатывать только если условие держится 5 минут.
- Не алертить на единичные выбросы (кроме P0).
- Настраивать уведомления по серьёзности (P0 — звонок, P1 — Telegram/Slack, P2 — email).

## 🧩 Связь метрик с логами и трейсами (Observability)

Метрики, логи и трейсы должны быть связаны общим `request_id` / `trace_id`, чтобы можно было **из метрики провалиться в логи** (например, увидев рост 500, посмотреть конкретные запросы).

**Пример связки:**
1. На графике Grafana видим всплеск p99 latency для `/api/payment`.
2. Кликаем на точку — переходим в Loki (логи) с фильтром `path="/api/payment"` за этот час.
3. Находим несколько медленных запросов, берём их `trace_id`.
4. Идём в Jaeger (трейсинг) и видим, что проблема в медленном запросе к БД.

Это называется **корреляция данных наблюдаемости**.

## 📋 Пример дашборда Grafana для API

Типичный дашборд (4 ряда):

1. **Общая статистика:**

4. **Системные метрики:**
   - CPU, память, количество подов (если Kubernetes)

## ⚠️ Частые ошибки при работе с метриками API

| Ошибка | Почему плохо | Исправление |
|--------|--------------|-------------|
| Измерять только среднюю latency (avg) | Выбросы скрываются | Использовать p95, p99 |
| Слишком много меток (labels) | Кардинальность (high cardinality) убивает Prometheus | Не добавлять `user_id` как label; использовать логи |
| Не агрегировать по эндпоинтам | Невозможно понять, какой именно путь тормозит | Добавлять `endpoint` label |
| Алерт на абсолютное число ошибок | При низком трафике 5 ошибок = 100% | Использовать `rate()` или относительную долю |
| Не удалять старые метрики | Нагрузка на хранилище | Настраивать retention (30–90 дней) |

## 🚀 Продвинутые практики

- **SLI / SLO / SLA**:
  - SLI (индикатор) — например, p95 latency < 300ms.
  - SLO (цель) — 99.9% запросов должны укладываться в SLI.
  - SLA (договор с клиентом) — штрафы при нарушении SLO.
- **Burn rate alerts** — алерты на скорость сжигания SLO (если за час мы исчерпаем месячный бюджет ошибок — срочно реагировать).
- **Anomaly detection** — автоматическое обнаружение аномалий (например, с помощью машинного обучения в Datadog или Prometheus + Prophet).

**Эндпоинт (endpoint)** — это конкретный URL, по которому клиент обращается к API, и метод HTTP, который определяет тип операции. Простыми словами: «ручка», которую можно дёрнуть, чтобы что-то получить или изменить.

Каждый эндпоинт — это точка входа в ваш сервис. Описание эндпоинта — это контракт, который объясняет, как его вызвать, что передать и что получить в ответ.

---

## 🧱 Из чего состоит описание эндпоинта

| Компонент | Описание | Пример |
|-----------|----------|--------|
| **HTTP метод** | Действие над ресурсом | `GET`, `POST`, `PUT`, `PATCH`, `DELETE` |
| **URL (путь)** | Адрес ресурса | `/users/{id}`, `/orders` |
| **Параметры пути (path params)** | Часть URL, идентифицирует ресурс | `{id}` → `/users/123` |
| **Параметры запроса (query params)** | Фильтрация, пагинация, сортировка | `?page=2&limit=20&sort=-created_at` |
| **Заголовки (headers)** | Метаинформация (формат, авторизация) | `Authorization: Bearer <token>`, `Content-Type: application/json` |
| **Тело запроса (request body)** | Данные для создания/обновления | `{ "name": "John", "email": "john@example.com" }` |
| **Ответ (response)** | Формат, структура, коды статуса | `200 OK` с телом `{ "id": 123, ... }` |
| **Ошибки (errors)** | Специфичные коды и сообщения | `400 Bad Request`, `404 Not Found`, `429 Too Many Requests` |

## 📝 Пример описания эндпоинта в документации (человекочитаемое)

### `GET /users/{id}` — получить пользователя по ID

**Метод:** `GET`  
**Путь:** `/users/{id}`  
**Аутентификация:** требуется (JWT в заголовке `Authorization`)

**Параметры пути:**
| Параметр | Тип | Обязательный | Описание |
|----------|-----|--------------|----------|
| `id` | integer | да | ID пользователя |

**Параметры запроса (опционально):**
| Параметр | Тип | Описание |
|----------|-----|----------|
| `fields` | string | Список полей через запятую (например, `fields=id,name,email`) |

**Заголовки:**
| Заголовок | Значение | Обязательный |
|-----------|----------|--------------|
| `Authorization` | `Bearer <token>` | да |
| `Accept` | `application/json` | нет (по умолчанию) |

**Успешный ответ (200 OK):**
```json
{
  "id": 123,
  "name": "Alex Smith",
  "email": "alex@example.com",
  "created_at": "2025-01-01T12:00:00Z"
}
```

**Ошибки:**
| Код | Описание |
|-----|----------|
| 401 Unauthorized | Отсутствует или недействительный токен |
| 403 Forbidden | Недостаточно прав (пользователь пытается получить чужой профиль) |
| 404 Not Found | Пользователь с таким ID не найден |
| 429 Too Many Requests | Превышен лимит запросов |

## 📄 Машинное описание (OpenAPI / Swagger)

Для автоматической генерации документации, клиентских SDK и валидации используют спецификацию OpenAPI (бывший Swagger). Вот как выглядит описание того же эндпоинта в YAML:

```yaml
paths:
  /users/{id}:
    get:
      summary: Get user by ID
      operationId: getUserById
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: integer
        - name: fields
          in: query
          required: false
          schema:
            type: string
          description: Comma-separated list of fields to return
      security:
        - bearerAuth: []
      responses:
        '200':
          description: Successful response
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/User'
        '401':
          description: Unauthorized
        '403':
          description: Forbidden
        '404':
          description: User not found
        '429':
          description: Too many requests
```

Этот файл можно загрузить в Swagger UI, ReDoc, Postman, или использовать для генерации кода.

## 🧠 Практические рекомендации по описанию эндпоинтов

### 1. Именование
- Используйте **существительные во множественном числе**: `/users`, `/orders`, `/products`.
- Глубина вложенности — не более 3: `/users/{userId}/orders/{orderId}/items` — ок, но `/users/{userId}/orders/{orderId}/items/{itemId}/details` — уже перебор.
- Для действий, не укладывающихся в CRUD, используйте глаголы: `/users/{id}/activate`, `/orders/{id}/cancel`.

### 2. Параметры
- **Path params** — для идентификации ресурса (обязательные).
- **Query params** — для фильтрации, пагинации, сортировки (опциональные).
- **Body** — для `POST`, `PUT`, `PATCH` (содержит данные).

### 3. Коды ответов (используйте стандартные HTTP)
- `200 OK` — успешный GET, PUT, PATCH, DELETE.
- `201 Created` — успешный POST (создание). В заголовке `Location` — URL нового ресурса.
- `204 No Content` — успешный DELETE или PUT/PATCH без тела.
- `400 Bad Request` — неверные входные данные (валидация).
- `401 Unauthorized` — нет аутентификации.
- `403 Forbidden` — аутентификация есть, но недостаточно прав.
- `404 Not Found` — ресурс не существует.
- `422 Unprocessable Entity` — семантическая ошибка (например, email уже занят).
- `429 Too Many Requests` — превышены лимиты.
- `500 Internal Server Error` — ошибка на сервере (не должен возвращаться в нормальном API).

### 4. Стабильность и версионирование
- Включайте версию в URL: `/v1/users`, `/v2/users`.
- При ломающих изменениях создавайте новую версию.
- Старые версии поддерживайте минимум 6–12 месяцев с пометкой `deprecated`.

### 5. Документирование ошибок
Возвращайте понятное тело ошибки, а не просто статус:
```json
{
  "error": {
    "code": "EMAIL_ALREADY_EXISTS",
    "message": "User with email alex@example.com already exists",
    "field": "email",
    "details": "https://docs.example.com/errors/EMAIL_ALREADY_EXISTS"
  }
}
```

### 6. Примеры
В документации обязательно приводите примеры:
- Пример запроса (curl, JavaScript, Python).
- Пример успешного ответа.
- Пример ответа с ошибкой.

## 💡 Итог

- **Эндпоинт** = метод + путь + параметры + тело + ожидаемые ответы.
- **Хорошее описание** включает: метод, путь, параметры, заголовки, форматы запроса и ответа, коды ошибок, примеры.

**Хардкод (hardcode)** в контексте API — это практика, когда значения, конфигурации, логика или данные зашиваются непосредственно в код обработчика эндпоинта, а не выносятся в переменные окружения, конфигурационные файлы, базы данных или внешние сервисы. Это антипаттерн, который приводит к негибкости, ошибкам и проблемам с безопасностью.

Ниже — разбор, почему хардкод вреден, как он проявляется в API и как с ним бороться.

---

## 🚫 Примеры хардкода в API (плохо)

### 1. Хардкод URL-ов и адресов сервисов
```python
# Плохо: адрес платежного шлюза зашит в коде
def process_payment(request):
    response = requests.post('http://192.168.1.100:8080/pay', json=data)
```
**Проблема:** при смене IP или переезде на другой стенд (dev → staging → prod) код нужно перекомпилировать и передеплоить.

### 2. Хардкод секретов (токенов, паролей, ключей)
```python
# Плохо: API-ключ прямо в коде
PAYMENT_API_KEY = "sk_live_4eC39HqLyjWDarjtT1zdp7dc"
```
**Проблема:** ключ попадёт в систему контроля версий, его увидят все разработчики, а при компрометации репозитория — и злоумышленники.

### 3. Хардкод бизнес-логики (например, ставки НДС)
```python
# Плохо: ставка НДС зашита
def calculate_total(price):
    return price * 1.20  # 20% НДС
```
**Проблема:** при изменении ставки (например, с 20% на 22%) нужно менять код, пересобирать и деплоить API.

### 4. Хардкод значений по умолчанию и лимитов
```python
# Плохо: лимиты и пагинация зафиксированы в коде
def list_users(request):
    limit = 100
    offset = request.GET.get('offset', 0)
```
**Проблема:** разные клиенты могут захотеть другой размер страницы; максимальный лимит должен быть настраиваемым через конфигурацию.

### 5. Хардкод сообщений об ошибках с внутренними деталями
```python
# Плохо: возвращаем детали SQL-запроса
except IntegrityError as e:
    return {"error": f"Duplicate key violates constraint {e.diag.constraint_name}"}
```
**Проблема:** раскрывается внутренняя структура БД (имена индексов, таблиц) — угроза безопасности.

---

## ✅ Как избежать хардкода: лучшие практики

| Вместо хардкода | Используйте | Пример |
|----------------|-------------|--------|
| URL, порты, адреса внешних сервисов | **Переменные окружения** | `PAYMENT_GATEWAY_URL = os.getenv("PAYMENT_URL")` |
| Секреты (ключи, пароли) | **Менеджеры секретов** (Vault, AWS Secrets Manager, Kubernetes Secrets) | `secret = vault.read('secret/payment_api_key')` |
| Бизнес-правила (налоговые ставки, лимиты) | **База данных или конфигурационный сервис** | Таблица `settings` с полями `key`, `value`, `valid_from` |
| Значения по умолчанию для параметров | **Конфигурационные файлы** (YAML, JSON) с возможностью переопределения | `default_limit = config.get("pagination.default_limit", 20)` |
| Форматы дат, кодировки | **Константы в отдельном модуле** (но через переменные окружения) | `DATE_FORMAT = os.getenv("API_DATE_FORMAT", "%Y-%m-%dT%H:%M:%SZ")` |
| Логика валидации (допустимые значения) | **Схемы из БД или внешний справочник** | `ALLOWED_CURRENCIES = settings.get("allowed_currencies")` |

## 🧠 Пример рефакторинга: хардкод → гибкий API

**До (хардкод):**
```python
@app.get('/api/orders/{id}')
def get_order(id):
    db_host = 'localhost'
    db_password = 'root'
    conn = psycopg2.connect(host=db_host, password=db_password)
    # ... бизнес-логика с фиксированной скидкой 10%
    discount = 0.1
```

**После (без хардкода):**
```python
import os
from somewhere import get_secret, get_discount_setting

@app.get('/api/orders/{id}')
def get_order(id):
    db_host = os.getenv('DB_HOST', 'localhost')  # из окружения
    db_password = get_secret('db_password')       # из Vault
    conn = psycopg2.connect(host=db_host, password=db_password)
    
    # Получаем актуальную скидку из БД настроек
    discount = get_discount_setting('order_discount')
```

## ⚠️ Когда хардкод оправдан (редкие исключения)

1. **Абсолютные константы, которые никогда не меняются:** число π, количество миллисекунд в секунде, коды HTTP-статусов (можно импортировать из библиотеки).
2. **Данные для локальной разработки** (но только в тестовых конфигах, не в коде).
3. **Минимальные демо-примеры** для обучения (но с пометкой, что не для production).

Но даже в этих случаях лучше выносить в переменные окружения или конфиги.

## 🔗 Связь с предыдущими темами

| Тема | Как связан хардкод |
|------|--------------------|
| **Проектирование API** | Хардкод противоречит принципу эволюционности и переносимости API между стендами. |
| **Характеристики API** | Ухудшает масштабируемость, безопасность, сопровождаемость. |
| **Стенды (окружения)** | Хардкод делает невозможным использование разных конфигураций для dev/staging/prod. |
| **Логи API** | Хардкод в сообщениях об ошибках может раскрыть внутренние детали. |
| **Безопасность** | Хардкод секретов — критическая уязвимость. |
| **Инфраструктура** | Инфраструктура как код (Terraform, Ansible) позволяет управлять конфигурациями без хардкода. |

## 💡 Резюме

- **Хардкод в API** — это зашитые в исходный код значения, которые должны меняться между окружениями или со временем.
- **Главные проблемы**: невозможность переключения стендов, утечка секретов, сложность изменений, риск ошибок.
- **Решения**: переменные окружения, менеджеры секретов, конфигурационные файлы, базы данных настроек.
- **Исключения**: только неизменяемые константы и демо-код.

Помните: код должен быть **конфигурируемым**, а не зашитым. Любое значение, которое может отличаться на dev, staging и prod, должно быть вынесено из кода.
