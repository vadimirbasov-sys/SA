Ниже приведены подробные ответы на ваши вопросы. Они охватывают фундаментальные понятия клиент-серверного взаимодействия, веб-протоколов, архитектур и смежных тем.

---

### 1. Что такое клиент-серверная архитектура?

**Клиент-серверная архитектура** — это модель взаимодействия, в которой участники сети делятся на две роли:
- **Клиент** — инициатор запроса (например, браузер, мобильное приложение). Он отправляет запрос и ожидает ответ.
- **Сервер** — пассивный участник, который постоянно ожидает входящие запросы, обрабатывает их и возвращает результат.

Ключевые характеристики:
- Сервер предоставляет услуги (ресурсы, данные, вычисления).
- Клиент потребляет эти услуги.
- Коммуникация обычно идёт через сеть (интернет, локальную) по протоколам (чаще всего HTTP/HTTPS).
- Примеры: вы открываете сайт — браузер (клиент) запрашивает страницу у веб-сервера; почтовый клиент (Thunderbird) — у почтового сервера (IMAP).

---

### 2. Чем фронтенд отличается от бэкенда?

| Признак | Фронтенд (Frontend) | Бэкенд (Backend) |
|---------|---------------------|------------------|
| **Где работает** | На стороне клиента (браузер, устройство пользователя) | На стороне сервера |
| **Технологии** | HTML, CSS, JavaScript, фреймворки (React, Vue, Angular) | Языки: Python, Java, Go, Node.js, PHP; базы данных, серверное ПО |
| **Задачи** | Отображение интерфейса, обработка действий пользователя, отправка запросов на бэкенд | Бизнес-логика, хранение данных, аутентификация, вычисления, интеграции |
| **Вид для пользователя** | Видимый интерфейс | Невидим (работает "под капотом") |

---

### 3. Какие бывают виды архитектур?

Основные виды архитектур программного обеспечения (в контексте распределённых систем):

1. **Монолитная** — всё приложение (фронтенд, бэкенд, БД) собрано в одной исполняемой единице. Простота, но сложность масштабирования.
2. **Клиент-серверная** — разделение на клиента и сервер (см. выше).
3. **Многоуровневая (n-tier)** — уточнение клиент-серверной: выделяются слои (презентации, бизнес-логики, данных).
4. **Микросервисная** — приложение разбито на множество мелких независимых сервисов, каждый выполняет одну функцию, общаются по сети (HTTP, gRPC, очереди). Плюсы: масштабируемость, независимое развертывание.
5. **Серверлесс (Serverless)** — разработчик не управляет серверами; код запускается как функции по запросу (AWS Lambda, Cloud Functions). Платформа сама масштабирует.
6. **Событийно-ориентированная (Event‑driven)** — компоненты реагируют на события (сообщения) через брокеры (Kafka, RabbitMQ). Асинхронная, слабосвязанная.

---

### 4. Что такое многослойная архитектура? Назвать минимум 3 слоя.

**Многослойная (layered) архитектура** — способ организации приложения, при котором функциональность разбивается на горизонтальные уровни (слои). Каждый слой решает свою задачу и взаимодействует только с соседними (обычно сверху вниз).

Минимум 3 слоя:
1. **Слой представления (Presentation)** — отвечает за интерфейс пользователя (UI) и взаимодействие с пользователем. Пример: веб-страницы, REST-контроллеры.
2. **Слой бизнес-логики (Business Logic)** — содержит основные алгоритмы, правила, вычисления, валидацию. Пример: сервисы в Spring, FastAPI.
3. **Слой доступа к данным (Data Access)** — обеспечивает чтение/запись в базы данных или внешние системы. Пример: репозитории, ORM.

Часто добавляют **слой интеграции** (для внешних API) и **слой кэширования**.

---

### 5. Что такое унифицированный интерфейс?

**Унифицированный интерфейс (Uniform Interface)** — один из принципов REST (см. вопрос 18). Он означает, что взаимодействие между клиентом и сервером должно быть стандартизировано и единообразно.

Включает 4 подтребования:
1. **Идентификация ресурсов** — каждый ресурс имеет уникальный идентификатор (URI).
2. **Представление ресурса** — клиент и сервер обмениваются представлениями ресурса (JSON, XML, HTML), а не самим ресурсом.
3. **Самодостаточность сообщений** — каждое сообщение содержит достаточно информации для его обработки (метод, заголовки, тело, ссылки на связанные ресурсы).
4. **HATEOAS** — гипермедиа как движок состояния приложения (см. вопрос 19).

Унифицированный интерфейс упрощает архитектуру и делает API предсказуемым.

---

### 6. Что такое модель OSI?

**Модель OSI (Open Systems Interconnection)** — эталонная семиуровневая модель взаимодействия открытых систем, разработанная ISO. Она описывает, как данные передаются от приложения одного компьютера к приложению другого через сеть.

7 уровней (снизу вверх):
1. **Физический** — кабели, разъёмы, сигналы.
2. **Канальный** — Ethernet, Wi-Fi, MAC-адреса.
3. **Сетевой** — IP, маршрутизация.
4. **Транспортный** — TCP, UDP, порты.
5. **Сеансовый** — управление сессией, контрольные точки.
6. **Уровень представления** — преобразование данных (шифрование, сжатие, кодировки).
7. **Прикладной** — HTTP, FTP, SMTP, API.

Модель OSI — теоретическая, на практике чаще используется модель TCP/IP (4 уровня).

---

### 7. Что есть HTTP и зачем он нужен?

**HTTP (HyperText Transfer Protocol)** — протокол прикладного уровня для передачи гипертекста (веб-страниц) и любых других данных по сети. Работает поверх TCP.

**Зачем нужен:**
- Основа веба: браузеры и серверы общаются через HTTP.
- Формат «запрос-ответ»: клиент отправляет запрос (метод, URL, заголовки, тело), сервер возвращает ответ (статус, заголовки, тело).
- Используется для веб-API (REST, GraphQL) и передачи любых данных (JSON, XML, файлы).
- Без HTTP современный интернет (сайты, API) был бы невозможен.

---

### 8. Что есть HTTPS и зачем он нужен?

**HTTPS (HTTP Secure)** — это HTTP, работающий внутри зашифрованного TLS (ранее SSL) соединения. Добавляет уровень безопасности поверх HTTP.

**Зачем нужен:**
- **Шифрование** — никто в сети (провайдер, хакер) не может прочитать передаваемые данные (пароли, токены, личная информация).
- **Аутентификация сервера** — клиент может проверить, что он общается именно с нужным сайтом (с помощью цифровых сертификатов).
- **Целостность данных** — данные не могут быть изменены в пути.
- Необходим для защиты пользователей, обязательно для любых систем, работающих с конфиденциальными данными (банки, соцсети, API). Современные браузеры помечают HTTP-сайты как «небезопасные».

---

### 9. Чем HTTP отличается от TCP, UDP?

| Протокол | Уровень (OSI) | Назначение | Особенности |
|----------|---------------|------------|--------------|
| **HTTP** | Прикладной (7) | Передача гипертекста, API | Работает поверх TCP. Запрос-ответ, текстовый. |
| **TCP** | Транспортный (4) | Надёжная доставка данных между приложениями | Установка соединения, контроль потерь, упорядочивание пакетов. Гарантия доставки. |
| **UDP** | Транспортный (4) | Быстрая, но ненадёжная доставка | Без соединения, нет гарантии доставки и порядка. Используется для видео/голоса, DNS. |

**Ключевое различие:** HTTP — это прикладной протокол с семантикой (методы, статусы, заголовки). TCP/UDP — транспортные протоколы, которые обеспечивают передачу байтов. HTTP всегда использует TCP (или иногда QUIC, но QUIC основан на UDP). Без TCP/UDP HTTP не может работать.

---

### 10. Что такое IP?

**IP (Internet Protocol)** — протокол сетевого уровня (уровень 3 OSI). Он отвечает за **адресацию** и **маршрутизацию** пакетов данных между разными сетями.

- Каждое устройство в сети имеет уникальный IP-адрес (например, `192.168.1.1` или `2001:db8::1`).
- IP разбивает данные на пакеты, каждый пакет содержит заголовок с IP-адресами отправителя и получателя.
- Маршрутизаторы используют IP-адреса для пересылки пакетов через интернет.
- Существуют две версии: IPv4 (32-бит, 4.3 млрд адресов) и IPv6 (128-бит, практически бесконечно).

IP сам по себе **ненадёжен** (пакеты могут теряться, дублироваться, приходить не по порядку). Поверх IP работают TCP или UDP, которые добавляют надёжность или скорость.

### 11. Из чего состоит HTTP запрос?

HTTP-запрос состоит из трёх частей (последняя может отсутствовать):

1. **Стартовая строка (Request Line)**:  
   `<МЕТОД> <URI> <ВЕРСИЯ_HTTP>`  
   Пример: `GET /users/123 HTTP/1.1`

2. **Заголовки (Headers)** — строки вида `Ключ: Значение`.  
   Примеры: `Host: api.example.com`, `Accept: application/json`, `Authorization: Bearer ...`  
   Заголовки заканчиваются пустой строкой.

3. **Тело (Body)** — опционально, для методов POST, PUT, PATCH. Содержит передаваемые данные (JSON, XML, form-data). Формат тела указан в `Content-Type`.

**Пример полного запроса**:
```
POST /v1/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Content-Length: 42

{"name":"Alice","email":"alice@example.com"}
```

### 12. Из чего состоит URL запроса?

**URL (Uniform Resource Locator)** — адрес ресурса в сети. Полный синтаксис:

```
[протокол]://[хост]:[порт]/[путь]?[query-параметры]#[якорь]
```

Составные части:

| Компонент | Пример | Назначение |
|-----------|--------|-------------|
| Протокол | `https` | Способ доступа (http, https, ftp) |
| Хост | `api.example.com` | Доменное имя или IP-адрес сервера |
| Порт | `:443` | Необязательный (по умолчанию 80 для http, 443 для https) |
| Путь | `/users/123` | Определяет ресурс на сервере |
| Query | `?page=2&limit=10` | Параметры запроса (фильтры, пагинация) |
| Якорь (фрагмент) | `#section` | Внутренняя ссылка на часть документа (не отправляется серверу) |

Пример: `https://example.com:8080/products?category=books#reviews`  
- протокол: `https`, хост: `example.com`, порт: `8080`, путь: `/products`, query: `category=books`, якорь: `reviews`.

### 13. Чем в HTTP метод PUT отличается от POST?

| Критерий | POST | PUT |
|----------|------|-----|
| **Основное назначение** | Создание нового ресурса (сервер генерирует ID) или выполнение действия | Полная замена существующего ресурса (или создание, если клиент знает ID) |
| **Идемпотентность** | Не идемпотентен (два одинаковых POST могут создать два ресурса) | Идемпотентен (повторная отправка не меняет состояние) |
| **URI** | На коллекцию (`POST /users`) | На конкретный ресурс (`PUT /users/123`) |
| **Ответ** | 201 Created + Location заголовок | 200 OK или 204 No Content |
| **Безопасность** | Изменяет состояние | Изменяет состояние |

**Общее правило:** POST — для создания новых объектов, PUT — для полного обновления или создания с известным ID.

---

### 14. Что есть идемпотентность и какие из методов неидемпотентны?

**Идемпотентность** — свойство операции, при котором **многократное повторение одного и того же запроса с одинаковыми данными приводит к такому же состоянию сервера, как и однократное выполнение**. При этом ответы могут отличаться (например, первый раз 201 Created, последующие 200 OK), но ресурс не изменяется после первого вызова.

**Идемпотентные методы:** GET, HEAD, PUT, DELETE, OPTIONS, TRACE.

**Неидемпотентные методы:** POST, PATCH (спецификация не гарантирует идемпотентность, хотя реализация может быть идемпотентной, если разработчик заложил это, например, через Idempotency-Key).

Почему это важно: при сбоях сети клиент может безопасно повторять идемпотентные запросы, не опасаясь дублирования (например, двойного списания денег).

---

### 15. Какой HTTP метод выбрать, чтобы выполнять абсолютно все операции и не использовать другие методы?

**POST**. Технически POST может делать всё, что угодно: создавать, читать, обновлять, удалять, выполнять сложные действия. Однако это **антипаттерн**, потому что:
- Нарушает принципы REST и стандартные семантики HTTP.
- Ухудшает читаемость API (непонятно, что делает эндпоинт).
- Ломает кеширование, идемпотентность и безопасность.
- Клиенты (браузеры, прокси) не смогут правильно обработать повторные запросы.

**Правильный подход:** использовать методы по назначению — GET для чтения, POST для создания/действий, PUT/PATCH для обновления, DELETE для удаления.

### 16. Чем 200 HTTP статус отличается от 202?

| Статус | Класс | Название | Смысл | Когда использовать |
|--------|-------|----------|-------|--------------------|
| 200 | 2xx (Успех) | OK | Запрос успешно выполнен, ответ содержит результат (тело). | Для синхронных операций: GET, PUT, PATCH, DELETE, когда сервер возвращает данные. |
| 202 | 2xx (Успех) | Accepted | Запрос принят в обработку, но результат ещё не готов (асинхронный режим). | Для длительных операций (обработка отчётов, массовые рассылки). Сервер может вернуть ссылку на проверку статуса в заголовке `Location`. |

**Ключевое различие:** 200 означает, что результат уже есть в ответе. 202 означает, что запрос поставлен в очередь, ответ будет позже.

### 17. Что имеется в виду под REST? Знать все принципы.

**REST (Representational State Transfer)** — архитектурный стиль для построения распределённых систем (обычно веб-API). Предложен Роем Филдингом в 2000 году. RESTful API должны следовать шести ограничениям (принципам):

1. **Клиент-сервер** — разделение ролей, независимая эволюция.
2. **Отсутствие состояния (Stateless)** — сервер не хранит контекст между запросами; каждый запрос содержит всю необходимую информацию.
3. **Кеширование** — ответы должны явно указывать, можно ли их кешировать, для улучшения производительности.
4. **Единообразие интерфейса (Uniform Interface)** — идентификация ресурсов через URI, представления (JSON/XML), самодостаточные сообщения, HATEOAS (см. ниже).
5. **Многоуровневая система (Layered System)** — клиент не знает, напрямую ли он общается с сервером или через прокси/балансировщик.
6. **Код по требованию (Code on Demand, опционально)** — сервер может передавать клиенту исполняемый код (например, JavaScript). Это необязательный принцип.

Практически любой современный «REST API» соблюдает первые 5 принципов (HATEOAS часто опускают, тогда его корректнее называть HTTP API). REST не привязан к HTTP, но чаще всего реализуется поверх него.

---

### 18. Что такое HATEOAS?

**HATEOAS (Hypermedia As The Engine Of Application State)** — один из принципов REST (часть Uniform Interface). Означает, что сервер должен возвращать вместе с ресурсом **гиперссылки** (URL) на возможные действия, которые клиент может выполнить дальше.

Клиент не должен «зашивать» в коде URL-шаблоны (например, `/users/{id}`). Вместо этого он получает ссылки из ответа сервера.

**Пример ответа с HATEOAS (JSON)**:
```json
{
  "id": 123,
  "name": "Alice",
  "_links": {
    "self": { "href": "/users/123" },
    "orders": { "href": "/users/123/orders" },
    "edit": { "href": "/users/123", "method": "PUT" }
  }
}
```

HATEOAS делает API более гибким, позволяет серверу менять URL без обновления клиента. На практике реализуется редко из-за сложности, но считается «чистым» REST.

---

### 19. Что есть XSD, XML?

**XML (eXtensible Markup Language)** — язык разметки, похожий на HTML, но более строгий и расширяемый. Он используется для хранения и передачи структурированных данных. XML-документ состоит из элементов, атрибутов, текста, имеет иерархическую структуру.

Пример XML:
```xml
<user id="123">
  <name>Alice</name>
  <email>alice@example.com</email>
</user>
```

**XSD (XML Schema Definition)** — язык описания структуры и правил для XML-документов. XSD определяет:
- какие элементы и атрибуты допустимы;
- их типы данных (строка, число, дата);
- количество вхождений (один, много, опционально);
- ограничения (регулярные выражения, диапазоны).

XSD используется для валидации XML: проверки, что документ соответствует ожидаемой структуре. Пример XSD:
```xml
<xs:element name="user">
  <xs:complexType>
    <xs:sequence>
      <xs:element name="name" type="xs:string"/>
      <xs:element name="email" type="xs:string"/>
    </xs:sequence>
    <xs:attribute name="id" type="xs:integer" use="required"/>
  </xs:complexType>
</xs:element>
```

В современных API XML во многом вытеснен JSON, но всё ещё используется в корпоративных системах (SOAP, файлы конфигураций).

### 20. Что такое экранирование данных?

**Экранирование данных (escaping)** — процесс преобразования специальных символов в тексте, чтобы они не интерпретировались языком разметки, SQL, регулярными выражениями и т.д. Необходимо для **безопасности и корректной обработки**.

Примеры:
- **HTML-экранирование**: символы `<`, `>`, `&` заменяются на `&lt;`, `&gt;`, `&amp;`, чтобы браузер не воспринял их как теги. Это защищает от XSS-атак.
- **SQL-экранирование**: символ `'` заменяется на `''` (или используются параметризованные запросы), чтобы предотвратить SQL-инъекции.
- **URL-экранирование** (percent-encoding): пробел заменяется на `%20`, символы типа `/` и `?` кодируются, чтобы их не спутали с разделителями URL.

Экранирование отличается от **валидации** (проверка допустимости) и **санитизации** (удаление или замена опасных конструкций). Лучшая практика — использовать готовые библиотеки для экранирования, а не писать свои функции.

### 21. Что означают термины stateless и stateful?

| Термин | Stateful (с сохранением состояния) | Stateless (без сохранения состояния) |
|--------|-----------------------------------|--------------------------------------|
| **Сервер хранит информацию о клиенте** | Да (сессии, контекст, корзина) | Нет |
| **Как идентифицируется клиент** | Обычно через cookie, сессионный ID | Каждый запрос содержит все данные (токен, параметры) |
| **Масштабируемость** | Сложная (нужно синхронизировать состояние между серверами) | Простая (любой сервер может обработать любой запрос) |
| **Примеры** | Старый веб с сессиями на сервере, FTP, базы данных | REST API (по требованию stateless), JWT, HTTP в целом |
| **Надёжность** | Может потерять состояние при падении сервера | Не зависит от конкретного сервера |

**Stateless** — принцип REST: каждый запрос должен быть самодостаточным. Это упрощает масштабирование и отказоустойчивость.

**Stateful** — когда сервер помнит состояние (например, вашу корзину в сессии). Требует липких сессий или внешнего хранилища (Redis).

### 22. Кэширование — на чьей стороне выполняется?

Кэширование может выполняться на **обеих сторонах**, а также на промежуточных узлах:

- **Сторона клиента** (браузер, мобильное приложение): кеширует ответы согласно заголовкам `Cache-Control`, `Expires`. При повторном запросе может использовать локальную копию.
- **Сторона сервера** (например, кеш в памяти, Redis, кеш БД): сервер может сохранять результаты дорогих операций (SQL-запросов, вычислений) и отдавать их без повторного выполнения.
- **Промежуточные узлы**: прокси-серверы, CDN (Content Delivery Network), обратные прокси (например, Nginx с кешированием). Они кешируют ответы для многих клиентов, снижая нагрузку на основной сервер.

HTTP-кеширование управляется заголовками: `Cache-Control` (max-age, no-store, public/private), `ETag` (условные запросы), `Last-Modified`.

Кэширование повышает производительность и снижает задержки, но может привести к отдаче устаревших данных. Для критических данных используют заголовки `no-cache` / `must-revalidate`.
